# 01 접근 기록

이 문서는 정확한 작업 일지를 복원하려는 문서가 아니라, 현재 구현을 만든 핵심 선택을 다시 읽기 좋게 정리한 기록이다.

## 핵심 설계 선택
### 1. 제어는 TCP, 입력/스냅샷은 UDP로 분리했다
- 이유: 반드시 도착해야 하는 이벤트와 최신성만 중요한 데이터를 같은 채널에 넣지 않기 위해서다.
### 2. TCP는 `VERB key=value` 라인 프로토콜로 만들었다
- 이유: 디버깅하기 쉽고, 발표 자료와 테스트 로그에서 사람이 읽을 수 있다.
### 3. 서버 authoritative 시뮬레이션을 유지했다
- 이유: 클라이언트가 상태를 확정하지 않도록 해야 충돌 판정과 결과 저장 기준이 흔들리지 않는다.
### 4. 방 단위 작업은 `boost::asio::strand`에서 직렬화했다
- 이유: 멀티스레드 `io_context` 환경에서도 같은 방 상태를 안전하게 다루려면 직렬 실행 구역이 필요하다.
### 5. 테스트를 unit / integration / load 3층으로 나눴다
- 이유: 프로토콜, 상태 로직, 실제 서버 프로세스 검증을 한 종류 테스트로 모두 해결하려 하면 디버깅이 어려워진다.

## 보류하거나 버린 선택
- 모든 통신을 TCP 하나로 처리: head-of-line blocking 때문에 실시간 입력 지연이 커질 위험이 있다.
- 모든 상태를 UDP로만 처리: 로그인, 결과 저장, reconnect 제어처럼 신뢰성이 필요한 흐름을 안전하게 다루기 어렵다.
- 전역 mutex 하나로 전체 서버 상태 보호: 방 단위 직렬화와 전역 자료구조 보호를 분리하는 편이 구조를 설명하기 쉽다.

## 구현 순서 메모
- 프로토콜/직렬화 계층을 먼저 고정했다.
- 그다음 authoritative 시뮬레이션과 SQLite persistence를 구현했다.
- 마지막에 서버 본체, bot/load test, 발표 자료까지 연결했다.
