# UDP Pinger — 연결 없이 핑을 보내다

## TCP와 다른 세계

웹 서버 과제에서는 TCP 소켓을 다뤘다. `connect()` → 데이터 전송 → `close()`, 이 흐름이 자연스러웠다. 연결이라는 개념이 있으니, 상대가 데이터를 받았는지 TCP가 알아서 보장해줬다.

UDP는 달랐다. 연결이라는 게 아예 없다. 패킷을 보내면(`sendto()`), 상대가 받았는지 받지 못했는지 알 수 없다. 네트워크 어딘가에서 패킷이 사라져도 아무런 알림이 오지 않는다.

이 과제에서 만든 것은 UDP 기반 핑 클라이언트다. 10개의 핑 메시지를 서버에 보내고, 응답이 돌아오면 왕복 시간(RTT)을 측정하고, 응답이 안 오면 "손실"로 처리하는 프로그램이다.

## 제공된 서버가 하는 일

서버는 이미 완성되어 제공된 코드였다. 이 서버의 핵심 동작은 두 가지다:

1. UDP 메시지를 받으면 그대로 대문자로 바꿔서 에코한다
2. 받은 메시지의 30%를 랜덤으로 **무시한다** — 패킷 손실 시뮬레이션

이 30% 드롭이 과제의 핵심이었다. 실제 네트워크에서는 라우터 큐 오버플로, 무선 간섭, 경로 변경 등으로 패킷이 사라진다. 서버가 이걸 인위적으로 만들어주니, 클라이언트가 "손실을 어떻게 감지하고 처리할 것인가"를 고민해야 했다.

## 소켓 타임아웃이 곧 손실 감지

UDP 소켓에서 `recvfrom()`을 호출하면 데이터가 올 때까지 무한히 기다린다. 패킷이 영원히 안 오면 프로그램이 영원히 멈춘다. 그래서 타임아웃을 걸었다:

```python
client_socket = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
client_socket.settimeout(1)  # 1초
```

이 1초가 과제에서 정해준 "손실 판정 기준"이다. 핑을 보내고 1초 안에 응답이 안 오면 "이 패킷은 잃어버린 것"으로 간주한다.

이 접근은 완벽하지 않다. 서버가 살아있지만 응답이 1.5초 걸린다면? 그건 손실이 아니라 지연인데, 타임아웃 기반 판정은 둘을 구분하지 못한다. 하지만 실제 핑 유틸리티도 같은 방식을 쓴다. 타임아웃 시간을 적절히 설정하는 것이 사실상 유일한 조정 수단이다.

## RTT 측정: 시간을 메시지에 심어 보내기

왕복 시간을 측정하려면 "보낸 시각"을 알아야 한다. 방법은 간단했다. 핑 메시지를 보내기 직전에 `time.time()`으로 현재 시각을 기록하고, 메시지 자체에도 이 타임스탬프를 포함시켰다:

```python
send_time = time.time()
message = f"Ping {seq} {send_time}"
```

응답이 돌아오면 `time.time()`을 다시 찍어서 차이를 구한다:

```python
recv_time = time.time()
rtt_ms = (recv_time - send_time) * 1000
```

시간을 메시지 안에도 넣은 이유는, 원래 과제 명세가 "메시지에 시퀀스 번호와 타임스탬프를 포함하라"고 요구했기 때문이다. 실제로 RTT 계산 자체에는 로컬 변수 `send_time`만 있으면 충분하다.

## 통계 집계

10개의 핑이 끝나면 통계를 출력한다:
- 보낸 패킷 수, 받은 패킷 수, 손실률
- 최소/평균/최대 RTT

이건 리스트에 RTT 값을 모아두고 `min()`, `max()`, `sum()/len()`으로 계산하면 된다. 만약 응답이 하나도 안 왔다면(`rtt_list`가 비어있다면) 별도 메시지를 출력한다.

처음에는 통계 없이 그냥 각 핑 결과만 출력했다. 하지만 10줄의 결과를 눈으로 훑어보는 것과, "30% 손실, 평균 RTT 0.4ms"라는 한 줄 요약을 보는 것은 정보 밀도가 완전히 다르다. 실제 `ping` 명령어가 Ctrl+C를 누르면 통계를 출력해주는 이유를 이해하게 됐다.

## try/except socket.timeout

코드에서 가장 중요한 패턴은 이 구조였다:

```python
try:
    client_socket.sendto(message.encode(), server_address)
    data, addr = client_socket.recvfrom(1024)
    # → RTT 계산, 결과 출력
except socket.timeout:
    print(f"Ping {seq}: Request timed out")
```

`socket.timeout`은 `settimeout()`으로 설정한 시간이 지났을 때 발생한다. 이 예외를 잡아서 "타임아웃"으로 처리하면, 나머지 핑은 계속 진행된다.

처음에는 예외 대신 `select.select()`로 읽기 가능 여부를 먼저 확인하는 방법도 고려했다. 하지만 이 과제에서는 핑 하나당 하나의 응답만 기다리므로, `settimeout` + `try/except`가 더 간결했다.

서버를 먼저 띄우고, 클라이언트를 실행하는 2단계 프로세스다:

```bash
# 터미널 1: 서버 시작
python3 problem/code/udp_pinger_server.py 12000

# 터미널 2: 클라이언트 실행

```

서버 터미널에서는 `[ECHO]`와 `[DROP]` 로그가 섞여서 나온다. 어떤 패킷이 드롭됐는지 서버 쪽에서 확인할 수 있다.

## TCP 핑 vs UDP 핑 vs ICMP 핑

이 과제를 마친 뒤 나중에 ICMP 핑 과제도 진행했다. 세 가지 방식의 차이를 정리하면:

- **UDP 핑**: 애플리케이션 레벨에서 에코 서버를 만들어 사용. 표준 유틸리티는 아님
- **ICMP 핑**: 네트워크 레이어에서 직접 동작. `ping` 명령어가 이것. Raw 소켓 필요
- **TCP 핑**: 연결 수립 자체로 도달 여부를 확인. `telnet host port`가 사실상 TCP 핑

UDP 핑은 교육 목적으로 가장 적합하다. 연결 설정이 없어서 코드가 짧고, 패킷 손실을 명시적으로 다뤄야 하니 비신뢰 전송의 본질을 직접 겪게 된다.

## 가져간 것

TCP에서는 "보내면 도착한다"가 기본이었다. UDP에서는 "보내도 안 갈 수 있다"가 기본이다. 이 차이가 코드에 어떤 영향을 주는지 — 타임아웃 설정, 재전송 로직, 통계 집계 — 를 직접 경험한 과제였다.

나중에 다룰 RDT 프로토콜이나 Go-Back-N에서는 이 "비신뢰 채널 위의 신뢰 전송"을 정식으로 구현하게 된다. UDP 핑은 그 전초전이었다.
