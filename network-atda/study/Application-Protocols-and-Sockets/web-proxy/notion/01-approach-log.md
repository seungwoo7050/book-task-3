# 01 접근 기록

이 문서는 정확한 작업 일지를 복원하려는 문서가 아니라, 현재 구현을 만든 핵심 선택을 다시 읽기 좋게 정리한 기록이다.

## 핵심 설계 선택
### 1. URL 파싱을 별도 함수로 분리했다
- 이유: 호스트/포트/경로/쿼리 조합의 경우의 수가 많아 네트워크 코드와 섞어 두면 읽기 어렵다.
### 2. 원본 서버에는 `Connection: close`로 요청한다
- 이유: 응답 종료 시점을 단순하게 만들고 캐시 저장을 쉽게 하기 위해서다.
### 3. 캐시 파일명은 URL의 MD5 해시를 쓴다
- 이유: 슬래시와 쿼리 문자열이 들어 있는 URL을 그대로 파일명으로 쓰면 운영체제별 제약을 바로 만난다.
### 4. 클라이언트마다 스레드를 만든다
- 이유: 프록시가 동시에 여러 연결을 다루는 기본 구조를 가장 직관적으로 보여준다.

## 보류하거나 버린 선택
- URL 원문을 그대로 캐시 파일명으로 사용: 파일 시스템 안전성과 경로 길이 문제가 바로 생긴다.
- 고급 캐시 정책을 먼저 넣기: 이 프로젝트의 핵심은 중개 동작과 재현 가능한 캐시 히트 증명이다.

## 구현 순서 메모
- 우선 URL 파싱과 원본 서버 포워딩을 완성했다.
- 그다음 캐시 저장/조회, 에러 코드 분기를 붙였다.
- 마지막에 멀티스레딩과 테스트 자동화를 정리했다.
