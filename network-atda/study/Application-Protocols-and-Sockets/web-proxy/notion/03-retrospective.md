# 03 회고

## 이번 프로젝트에서 확실히 얻은 것
- 프록시는 단순 중계가 아니라 URL 재해석, 헤더 재구성, 캐시 정책을 동시에 다루는 소프트웨어라는 점을 배웠다.
- 원본 서버를 일부러 끈 뒤 캐시 히트를 확인하는 방식이 기능 검증에 매우 설득력 있다는 점을 경험했다.

## 아직 약한 부분
- 캐시 만료, 조건부 GET, HTTPS 터널링이 없어 실제 브라우징 프록시로 쓰기에는 부족하다.
- 멀티스레드 구조지만 자원 사용량을 계측하거나 제한하지는 않았다.

## 포트폴리오로 키우려면
- LRU 캐시와 만료 정책을 추가해 캐시 설계 문서까지 함께 공개한다.
- 요청/응답 로그, cache hit/miss 지표를 넣어 시연 자료로 확장한다.
