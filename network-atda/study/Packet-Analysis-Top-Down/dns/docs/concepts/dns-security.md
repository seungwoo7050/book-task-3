# DNS 보안: DNSSEC, DoH, DoT

## 개요

DNS는 원래 보안 기능 없이 설계되었다. 모든 쿼리와 응답이 평문 UDP로 전송되므로, 도청·위조·캐시 포이즈닝 등의 공격에 취약하다. 이 문서는 DNS 보안 확장 기술들을 개괄한다.

## DNS 보안 위협

| 위협 | 설명 | 영향 |
| :--- | :--- | :--- |
| DNS 스푸핑 | 위조된 DNS 응답을 주입 | 사용자를 악성 서버로 유도 |
| 캐시 포이즈닝 | DNS 캐시에 잘못된 레코드 삽입 | 대규모 사용자에게 영향 |
| DNS 도청 | 쿼리/응답 모니터링 | 브라우징 기록 노출 |
| DNS 증폭 공격 | 작은 쿼리로 큰 응답을 유발 (DDoS) | 피해자 서버 과부하 |

## DNSSEC (DNS Security Extensions)

### 동작 원리

DNSSEC은 DNS 응답에 **전자 서명**을 추가하여 데이터 무결성과 출처 인증을 제공한다.

```
1. 도메인 소유자가 RRSIG 레코드로 DNS 레코드에 서명
2. 해석기가 DNSKEY 레코드로 서명 검증
3. DS 레코드로 상위 도메인과 신뢰 체인 구성 (Trust Chain)
```

### 추가 레코드 타입

| 레코드 | 설명 |
| :--- | :--- |
| RRSIG | 리소스 레코드 집합의 전자 서명 |
| DNSKEY | 도메인의 공개 키 |
| DS | 자식 도메인의 DNSKEY 해시 (신뢰 체인 연결) |
| NSEC/NSEC3 | 존재하지 않는 도메인에 대한 인증된 거부 |

### 한계

- 데이터를 **암호화하지 않음** → 쿼리/응답 내용은 여전히 노출
- 응답 크기 증가 → UDP 패킷이 512바이트를 초과하여 TCP 폴백 또는 EDNS0 필요
- 배포 복잡성 → 모든 레벨에서 서명 관리 필요

## DNS over HTTPS (DoH)

### 동작 원리

DNS 쿼리를 **HTTPS** 연결 안에 캡슐화하여 전송.

```
기존 DNS: Client → UDP:53 → DNS Server (평문)
DoH:      Client → HTTPS(TCP:443) → DoH Server (암호화)
```

### 장점

- 쿼리/응답 암호화 → ISP, 공격자의 도청 방지
- 기존 HTTPS 인프라 활용 → 방화벽 통과 용이
- HTTP/2 멀티플렉싱으로 성능 향상

### 단점

- DNS 트래픽이 HTTPS에 숨겨져 네트워크 관리자의 모니터링 어려움
- 몇몇 대형 DoH 제공자(Google, Cloudflare)에 DNS 트래픽 집중
- 기존 DNS 캐싱 인프라와의 호환성 문제

## DNS over TLS (DoT)

```
기존 DNS: Client → UDP:53 → DNS Server (평문)
DoT:      Client → TLS(TCP:853) → DNS Server (암호화)
```

DoH와 달리 전용 포트(853)를 사용하므로 네트워크 관리자가 DNS 트래픽을 식별할 수 있다.

## Wireshark 실습과의 연관

실습에서 관찰한 DNS 패킷은 모두 **평문 UDP:53** 트래픽이다:
- 쿼리의 도메인 이름이 그대로 노출
- 응답의 IP 주소가 그대로 노출
- 패킷을 캡처하는 누구나 사용자의 방문 사이트를 파악 가능

DoH/DoT를 사용하면 이 정보가 암호화되어 Wireshark로도 내용을 볼 수 없다.
