# 02 디버그 로그

이 파일은 과거의 날짜 순서를 복원하려는 문서가 아니다. 백업 노트와 현재 테스트를 함께 보고, 지금도 재현하거나 설명할 수 있는 문제만 남긴다.

## 사례 1. out-of-order 패킷이 즉시 전달되어 메시지 순서가 뒤섞임
- 증상: 도착한 순서대로 상위에 넘기면 원문 메시지 순서가 바뀌었다.
- 원인: SR receiver는 패킷을 받을 수 있는 순서와 애플리케이션에 전달할 수 있는 순서를 분리해야 한다.
- 수정: `recv_base` 기준으로 앞쪽이 연속으로 채워질 때만 버퍼를 비우도록 바꿨다.
- 확인: 기본 fixture 전송과 순서 보장 확인으로 검증했다.

## 사례 2. 패킷 하나가 늦었을 뿐인데 윈도 전체가 다시 전송됨
- 증상: SR을 구현했는데도 GBN처럼 과도한 재전송이 발생했다.
- 원인: 전역 timeout이나 윈도 단위 재전송 로직이 남아 있으면 SR의 장점이 사라진다.
- 수정: 패킷별 타이머를 유지하고 만료된 시퀀스만 다시 보내도록 수정했다.
- 확인: 로그를 보며 특정 시퀀스만 재전송되는지 확인하고 테스트를 다시 돌렸다.

## 사례 3. 이미 수신한 중복 패킷에 ACK를 다시 보내지 않아 sender가 오래 대기함
- 증상: receiver는 패킷을 이미 갖고 있는데 sender가 timeout까지 기다렸다.
- 원인: 중복 패킷이라도 ACK 손실 가능성을 고려해 재응답해야 한다.
- 수정: `seq < recv_base`인 경우에도 적절한 ACK를 재전송하도록 정리했다.
- 확인: 손실 환경 반복 실행과 `make test`로 sender 대기 구간이 줄어드는지 확인했다.
