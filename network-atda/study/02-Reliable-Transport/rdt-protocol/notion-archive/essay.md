# 비신뢰 채널 위의 신뢰 전송 — RDT 3.0과 Go-Back-N

## 왜 신뢰 전송이 필요한가

IP 계층은 최선형(best-effort) 서비스를 제공할 뿐이다. 패킷은 유실될 수 있고, 비트가 뒤집혀 도착할 수도 있다. TCP가 이 혼돈 위에서 "모든 데이터가 순서대로 정확히 도착한다"는 보장을 만들어내는 원리를 직접 체험하기 위해, 교과서가 제시하는 두 프로토콜을 하나의 채널 모델 위에서 구현했다.

## 채널과 패킷 — 제공된 인프라

시작점은 두 개의 제공 모듈이다.

`channel.py`의 `UnreliableChannel`은 send/receive 인터페이스를 가진 큐인데, send 시점에 확률적으로 패킷을 잃어버리거나(`loss_rate`), 임의의 비트를 뒤집어 손상시킨다(`corrupt_rate`). 내부적으로 `random.random()`으로 결정하고, 손상의 경우 `bytearray`에서 무작위 바이트의 무작위 비트를 XOR한다. 실제 네트워크의 비결정적 행동을 단일 프로세스 안에서 재현하는 것이다.

`packet.py`는 패킷 포맷을 정의한다: `[checksum 4B][seq_num 4B][payload]`. 체크섬은 seq_num + payload에 대한 MD5 해시의 앞 4바이트다. `is_corrupt()`는 수신 패킷의 체크섬을 재계산해서 불일치를 감지하고, `make_ack(seq)`는 payload가 `b"ACK"`인 패킷을 만든다. 이 두 모듈은 수정하지 않는다 — 프로토콜 구현만이 내 영역이다.

## RDT 3.0: Stop-and-Wait

### 핵심 아이디어

한 번에 하나의 패킷만 보내고, ACK가 올 때까지 기다린다. 시퀀스 번호는 0과 1을 번갈아 사용한다 — 이것만으로 "이전 패킷의 중복 ACK"와 "현재 패킷의 ACK"를 구분할 수 있다. 타임아웃이 발생하면 같은 패킷을 재전송한다.

### 단일 스레드 이벤트 루프

실제 네트워크처럼 sender와 receiver를 별도 프로세스로 두는 대신, 하나의 while 루프 안에서 양쪽 역할을 교대로 수행한다. 루프의 한 반복에서:

1. **receiver side** — 데이터 채널에 패킷이 있으면 꺼내서 검사한다. 손상되지 않았고 기대하는 시퀀스 번호라면 수신 목록에 추가하고 ACK를 보낸다. 그렇지 않으면(중복이거나 손상) 이전 ACK를 다시 보낸다.
2. **sender side** — ACK 채널에 패킷이 있으면 꺼내서 `is_ack(pkt, send_seq)`로 검증한다. 유효하면 시퀀스 번호를 뒤집고 다음 메시지를 전송한다.
3. **timeout check** — `time.time() - timer_start > 0.5`이면 현재 패킷을 재전송하고 타이머를 리셋한다.

`time.sleep(0.01)`로 busy-waiting의 CPU 점유를 줄인다.

### alternating bit의 의미

시퀀스 번호가 0 → 1 → 0 → 1로 번갈아가는 이유는 단순하다: ACK이 유실되면 sender가 같은 패킷을 재전송하는데, receiver는 시퀀스 번호를 보고 "이건 이미 받은 것"임을 알 수 있다. 2비트면 충분한 이유는, Stop-and-Wait에서는 미확인 패킷이 최대 1개이므로 "현재"와 "이전"만 구분하면 되기 때문이다.

## Go-Back-N: 파이프라인의 시작

### Stop-and-Wait의 한계

채널의 RTT가 50ms이고 패킷 전송 시간이 1ms라면, 전체 시간의 98%는 ACK를 기다리며 놀고 있다. 이 utilization 문제를 해결하는 것이 파이프라인이다.

### 윈도우와 누적 ACK

GBN은 최대 N개의 패킷을 ACK 없이 연속 전송한다. `base`(가장 오래된 미확인 패킷)와 `next_seq`(다음 전송할 패킷) 사이가 윈도우다. 기본 윈도우 크기는 4로 설정했다.

receiver는 극단적으로 단순하다: 기대하는 시퀀스 번호의 패킷만 수신하고, 나머지는 전부 버린다. ACK는 항상 "여기까지 순서대로 받았다"는 누적(cumulative) ACK이다.

### 타임아웃 시 전체 재전송

이것이 GBN의 핵심이자 약점이다. 타이머가 만료되면 `base`부터 `next_seq - 1`까지 모든 미확인 패킷을 재전송한다. 하나의 패킷이 유실되면 그 뒤에 이미 성공적으로 전달된 패킷들까지 다시 보내야 한다. receiver가 버퍼링을 하지 않기 때문이다.

### 구현의 차이점

RDT 3.0과의 코드 차이를 보면:

- 시퀀스 번호가 0/1이 아닌 0부터 증가하는 정수
- 패킷을 미리 전부 빌드해둠 (`packets = [make_packet(i, ...) for i in range(total)]`)
- sender의 상태가 `base`, `next_seq` 두 포인터
- ACK 수신 시 `base = ack_seq + 1`로 윈도우 슬라이드
- 타임아웃 시 for 루프로 범위 재전송

## 두 프로토콜의 비교

같은 테스트 메시지 파일, 같은 손실률(0.2)과 손상률(0.1)에서 실행하면 차이가 뚜렷하다. RDT 3.0은 메시지 하나당 최소 하나의 RTT를 소모하므로 메시지가 늘어나면 시간이 선형으로 증가한다. GBN은 윈도우 크기만큼 병렬 전송하므로 손실이 없으면 N배 빠르다. 하지만 손실이 잦으면 불필요한 재전송이 폭발적으로 늘어난다.

두 프로토콜을 `--loss`와 `--window` 인자를 바꿔가며 실행하면, 교과서의 utilization 공식이 체감으로 다가온다.

## 채널 모델의 가치

실제 네트워크 없이 `UnreliableChannel` 하나로 프로토콜을 검증할 수 있다는 점이 이 과제의 핵심 설계다. loss_rate=0, corrupt_rate=0으로 설정하면 이상적인 채널에서의 정상 동작을 확인하고, 값을 높여가며 극한 상황을 시험한다. 비결정적이므로 같은 설정에서도 매 실행 결과가 다르지만, 프로토콜이 정확하다면 **항상** 모든 메시지가 순서대로 도착해야 한다.

## 테스트 전략

테스트는 두 계층으로 나뉜다:

1. **패킷 유틸리티 단위 테스트** — `make_packet → parse_packet` 왕복, 손상 감지, ACK 생성/검증, 서로 다른 시퀀스 번호의 체크섬 차이. 이것은 프로토콜 구현의 기반이 정확한지 확인한다.
2. **통합 테스트** — `test_rdt.sh`가 RDT 3.0과 GBN을 각각 실행하고 "SUCCESS" 문자열이 출력되는지 확인한다. 비결정적 채널 위에서 돌아가므로, 성공은 프로토콜의 정확성을 확률적으로 보증한다.

## 이 과제로부터 배운 것

Checksum, sequence number, timer, ACK — 전송 계층의 네 가지 무기를 직접 조합하면서, TCP의 설계 결정들이 왜 그렇게 이루어졌는지를 체감했다. 특히 GBN의 "불필요한 재전송" 문제를 눈으로 보고 나면, Selective Repeat가 왜 필요한지가 자연스럽게 이해된다. 다음 프로젝트인 SR 구현은 이 경험 위에서 시작한다.
