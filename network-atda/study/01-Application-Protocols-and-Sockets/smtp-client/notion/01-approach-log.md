# 01 접근 기록

이 문서는 정확한 작업 일지를 복원하려는 문서가 아니라, 현재 구현을 만든 핵심 선택을 다시 읽기 좋게 정리한 기록이다.

## 핵심 설계 선택
### 1. 송신과 응답 검사를 helper 함수로 분리했다
- 이유: 명령-응답-검증 흐름을 반복해서 써야 하므로 단계별 책임을 나누는 편이 읽기 쉽다.
### 2. 응답 코드는 매 단계에서 즉시 검사한다
- 이유: 한 단계가 어긋난 상태로 다음 명령을 보내면 어디서 실패했는지 추적하기 어렵다.
### 3. 메일 헤더와 SMTP envelope를 따로 생각했다
- 이유: 사용자에게 보이는 `To:` 헤더와 실제 `RCPT TO`는 다른 계층이라는 점을 분명히 남기고 싶었다.
### 4. mock 서버를 기본 검증 축으로 삼았다
- 이유: 실제 외부 서버 상태에 문서와 테스트가 흔들리지 않게 하려면 로컬 재현이 우선이다.

## 보류하거나 버린 선택
- 실제 Gmail/회사 SMTP 서버를 기준으로 개발: 인증과 정책 차이가 커서 과제의 핵심인 기본 SMTP 흐름보다 운영 이슈가 앞서기 때문이다.
- 헤더 생성을 라이브러리에 맡기기: 직접 줄을 조립해 봐야 `DATA`와 RFC 5322 헤더 형식을 체감할 수 있다.

## 구현 순서 메모
- greeting 수신과 `HELO` 교환부터 통과시켰다.
- 그다음 envelope 명령(`MAIL FROM`, `RCPT TO`)과 `DATA` 본문 전송을 붙였다.
- 마지막에 실패 처리와 CLI/테스트를 정리했다.
