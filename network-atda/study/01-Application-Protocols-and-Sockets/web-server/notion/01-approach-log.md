# 01 접근 기록

이 문서는 정확한 작업 일지를 복원하려는 문서가 아니라, 현재 구현을 만든 핵심 선택을 다시 읽기 좋게 정리한 기록이다.

## 핵심 설계 선택
### 1. `/` 요청은 `hello.html`로 매핑했다
- 이유: 브라우저와 `curl`로 가장 먼저 확인할 경로를 고정해야 테스트와 설명이 단순해진다.
### 2. 헤더는 최소한만 읽고 응답 뒤에 연결을 닫는다
- 이유: 이 과제의 초점은 파일 서빙과 상태 코드이지 keep-alive 구현이 아니다.
### 3. 확장자 기반 `Content-Type` 표를 코드에 명시했다
- 이유: HTML과 텍스트가 섞이는 과제에서 브라우저 동작 차이를 바로 설명할 수 있다.
### 4. 클라이언트마다 새 스레드를 만든다
- 이유: 소켓 accept 루프와 요청 처리를 분리해 동시 접속 개념을 바로 드러내기 쉽다.

## 보류하거나 버린 선택
- `asyncio` 기반 비동기 서버: 학습 순서상 먼저 소켓과 스레드의 기본 흐름을 체감하는 편이 낫다고 판단했다.
- 범용 웹 프레임워크 사용: HTTP를 직접 파싱하는 경험이 사라져 과제의 학습 포인트가 흐려진다.

## 구현 순서 메모
- 소켓 생성과 `listen`/`accept` 흐름을 먼저 고정했다.
- 요청 라인 파싱과 파일 읽기, `200`/`404` 응답을 붙였다.
- 그 뒤 `Content-Type`, 멀티스레딩, 테스트 정리를 얹었다.
