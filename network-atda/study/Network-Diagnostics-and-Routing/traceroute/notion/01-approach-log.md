# 01 접근 기록

이 문서는 정확한 작업 일지를 복원하려는 문서가 아니라, 현재 구현을 만든 핵심 선택을 다시 읽기 좋게 정리한 기록이다.

## 핵심 설계 선택
### 1. UDP 송신과 raw ICMP 수신을 별도 소켓으로 분리했다
- 이유: 운영체제 traceroute가 하는 일과 동일한 계층 구조를 가장 분명하게 드러낼 수 있다.
### 2. probe 포트는 `33434 + offset` 규칙으로 만들었다
- 이유: TTL과 probe index를 포트에 인코딩하면 돌아온 ICMP 메시지와 원래 probe를 매칭하기 쉽다.
### 3. 응답 파싱은 nested header를 단계별로 푼다
- 이유: 겉 ICMP 헤더만 보면 어느 probe에 대한 응답인지 알 수 없다.
### 4. hop 출력은 성공 주소와 `*`를 같은 줄 형식으로 정리했다
- 이유: 실제 traceroute 출력과 비교하기 쉬운 학습용 UI를 의도했다.

## 보류하거나 버린 선택
- ICMP Echo 기반 traceroute로 단순화: 이번 구현은 UDP probe + ICMP error 구조를 이해하는 데 목적이 있다.
- 응답 매칭을 송신 시각만으로 추정: 동시에 여러 probe를 날리면 충돌 가능성이 커진다.

## 구현 순서 메모
- 먼저 probe 포트 생성 규칙과 출력 포맷을 고정했다.
- 그다음 nested ICMP 응답 파서를 만들었다.
- 마지막에 전체 trace 루프와 종료 조건을 붙였다.
