# Project-Set Audit

현재 프로젝트 집합은 네트워크 학습을 **작게 구현하고, 근거를 관찰하고, 진단 도구로 확장하고, 마지막에 통합 설계로 묶는 흐름**으로 설계되어 있습니다. 아래 평가는 `2026-03-10` 기준 현재 저장소에 존재하는 프로젝트만 대상으로 합니다.

## 현재 구성이 잘하는 점

- 구현 과제와 패킷 분석 과제가 분리되어 있어 "만들기"와 "읽기"를 번갈아 훈련할 수 있습니다.
- 거의 모든 프로젝트가 `make test` 기반의 canonical 검증을 갖고 있어, 저장소를 받은 사람이 재현하기 쉽습니다.
- README가 문제 요약, 학습 포인트, 현재 한계, 포트폴리오 확장 방향까지 함께 보여 주도록 재정비되었습니다.
- capstone이 별도 트랙으로 분리되어 있어, 작은 과제들의 의미가 최종 산출물과 연결됩니다.

## 의도적으로 남겨 둔 공백

- 운영 환경 수준의 인증, 보안, 배포, 관측성은 모든 프로젝트에 넣지 않았습니다. 학습 아카이브가 과도하게 비대해지는 것을 막기 위한 선택입니다.
- 패킷 분석 랩은 새 trace를 만드는 실습까지 강제하지 않습니다. 대신 현재 trace를 정확히 읽는 습관을 먼저 잡도록 했습니다.
- 구현 과제 중 일부 live 실행은 네트워크 환경이나 권한에 따라 흔들리므로, 완료 판정은 deterministic 검증을 우선합니다.

## 현재 세트가 학습용으로 충분한 이유

- 응용 계층, 전송 계층, 네트워크 계층, 링크 계층, 보안 계층을 모두 한 번 이상 직접 다룹니다.
- 구현 프로젝트와 분석 프로젝트가 서로를 보완합니다. 예를 들어 `web-server`와 `http`, `icmp-pinger`와 `ip-icmp`, `rdt-protocol`과 `tcp-udp`가 좋은 짝이 됩니다.
- `traceroute`, `selective-repeat`, `tactical-arena-server` 같은 프로젝트는 기존 교재 흐름에서 비어 있기 쉬운 연결 지점을 메워 줍니다.

## 앞으로 프로젝트를 더 추가할 때의 기준

- 이미 있는 프로젝트와 학습 포인트가 크게 겹친다면 새 프로젝트를 늘리기보다 기존 README와 docs를 더 강하게 만드는 쪽이 낫습니다.
- 새 프로젝트는 최소 하나 이상의 뚜렷한 공백을 메워야 합니다. 예를 들어 `HTTP/2`, `QUIC`, `BGP`, `NAT traversal`처럼 현재 세트에 없는 주제가 그렇습니다.
- 포트폴리오용 확장은 기존 프로젝트를 깊게 키우는 방식도 좋습니다. 예를 들어 `web-proxy`에 `CONNECT`와 만료 정책을 붙이거나, `tactical-arena-server`에 replay viewer를 추가하는 식입니다.
