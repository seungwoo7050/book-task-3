# 문제 프레이밍

> 프로젝트: 에디터
> 아래 내용은 `notion-archive/00-problem-framing.md`에 남아 있던 기존 학습 기록을 현재 공개 노트 기준으로 다듬은 버전이다.

## 문제를 처음 읽었을 때

BOJ 1406은 텍스트 에디터의 커서 이동, 삽입, 삭제를 시뮬레이션하는 문제다.
초기 문자열이 주어지고, 커서는 문자열 끝에서 시작한다.
L(왼쪽 이동), D(오른쪽 이동), B(왼쪽 삭제), P x(삽입) 네 가지 명령을 처리해야 한다.

처음에는 "리스트에 커서 인덱스를 두고 insert/pop 하면 되겠다"고 생각했다.
그런데 $M$이 최대 50만이고 문자열 길이가 10만이다.
리스트의 `insert()`와 `pop(i)`는 O(n)이므로, 최악의 경우 $5 \times 10^{10}$ 연산이 된다.
**리스트로는 시간 안에 못 푼다.**

이 지점에서 "커서 위치에서 O(1) 삽입/삭제가 되는 자료 구조"가 필요하다는 걸 깨달았다.

## 문제의 핵심 구조

- **입력**: 초기 문자열 (길이 $\le 100\,000$), 명령 수 $M$ ($\le 500\,000$)
- **출력**: 모든 명령 처리 후의 최종 문자열
- **핵심**: 커서 위치에서의 삽입/삭제가 $O(1)$이어야 한다

## 왜 이 문제를 골랐는가

Core-01-Array-List에서 "리스트(배열)의 한계"를 체감하게 만드는 문제다.
배열의 insert/delete가 O(n)이라는 사실을 알고 있어도, 실제로 시간 초과를 경험하지 않으면 체감이 안 된다.
이 문제는 그 체감을 강제한다.

## 지금 이 프로젝트에서 먼저 고정할 것

- `에디터`에서 실제로 확인하려는 학습 목표는 `순차 자료구조를 선택하고 편집 연산의 비용 모델을 설명하는 연습`이다.
- 문제를 읽을 때는 "정답을 맞힌다"보다 어떤 상태와 규칙을 끝까지 유지해야 하는지를 먼저 적어 두는 편이 좋다.
- 비슷한 유형을 다시 만났을 때 바로 적용할 수 있는 작은 판단 기준을 남겨 두는 것이 핵심이었다.
- 학습자 입장에서는 `05-development-timeline.md`를 같이 열어 두면 실제 재현 순서와 문제 해석이 어떻게 맞물리는지 더 잘 보인다.
