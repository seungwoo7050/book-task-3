# 회고 — 자료 구조 선택이 곧 성능이다

## 이 문제가 남긴 것

### "맞는 코드"와 "통과하는 코드"는 다르다

리스트에 insert/pop(i)를 쓰면 논리적으로 맞는 결과가 나온다.
하지만 시간 초과로 제출이 안 된다.
"정확성은 필요 조건이지 충분 조건이 아니다"라는 걸 이 문제에서 체감했다.

### Two-Stack 패턴의 발견

이 패턴은 에디터만이 아니라, 커서 기반 탐색이 필요한 곳이면 어디든 적용된다.
나중에 BOJ 5397(키로거)에서 거의 같은 구조를 재활용했다.

### Python list의 append/pop은 O(1)이다 (amortized)

Python의 list는 동적 배열이므로 끝에서의 append/pop은 amortized O(1).
하지만 중간 삽입(insert)은 O(n)이다.
이 차이를 "알고 있다"와 "경험했다"는 완전히 다르다.

## 다음에 적용할 것

- 삽입/삭제 위치가 고정적이면 스택 기반 모델 검토
- 연결 리스트가 필요할 것 같은 문제에서도, Python에서는 스택 2개가 더 현실적
