# 접근 과정 — 슬라이싱이냐 투 포인터냐

## 가장 먼저 떠오른 방법

문자열을 뒤집어서 원본과 비교하면 끝이다.
Python에서는 `word[::-1]`이라는 슬라이싱 한 줄이면 되니까, 생각하자마자 코드가 나왔다.

```
word == word[::-1]
```

이렇게 쓰면 읽기도 편하고, 틀릴 여지도 거의 없다.

## 그런데 투 포인터가 머릿속에 남아 있었다

docs에 있는 approach.md를 보면 "양끝 포인터(two pointers)로 좌우 문자를 동시에 비교"라는 전략이 적혀 있다.
실제로 회문 판별의 정석적인 방법은 양쪽 끝에서 중앙으로 좁혀 오면서 비교하는 것이다.

이 방식의 장점은:
- 불일치가 발견되는 즉시 `False`를 반환할 수 있어서 **평균적으로 더 빠르게 종료**한다
- 추가 메모리를 쓰지 않는다 (슬라이싱은 새 문자열을 만든다)

하지만 이 문제에서 문자열 최대 길이가 100이다.
100글자짜리 문자열을 하나 더 만드는 게 성능에 영향을 줄 리가 없다.

## 최종 선택: 슬라이싱

결국 **가독성을 우선**해서 슬라이싱 방식을 택했다.
코드가 10줄도 안 되는 문제에서 투 포인터를 쓰면 오히려 과한 느낌이다.

다만, 이 선택 과정 자체가 나중에 더 큰 문자열 문제를 만났을 때 참고가 된다.
"언제 슬라이싱을 쓰고 언제 투 포인터를 쓰는가"에 대한 감각을 여기서 처음 잡았다.

## 복잡도 정리

| 항목 | 슬라이싱 | 투 포인터 |
|------|---------|----------|
| 시간 | O(n) | O(n) |
| 공간 | O(n) | O(1) |
| 가독성 | 높음 | 보통 |

n=100이라는 제약에서는 두 방법 모두 동일하게 안전하다.
