# 디버그 로그

> 프로젝트: 배열 돌리기 1
> 아래 내용은 `notion-archive/02-debug-log.md`에 남아 있던 기존 학습 기록을 현재 공개 노트 기준으로 다듬은 버전이다.

## 가장 많이 실수한 부분: 코너 중복

처음 구현했을 때 레이어에서 원소를 추출하면서 코너를 두 번 세는 실수를 했다.
예를 들어 윗변의 마지막 원소(오른쪽 상단 코너)를 추출하고, 오른변의 첫 원소로 다시 추출하면 같은 원소가 두 번 들어간다.

해결: 각 변의 범위를 정확히 설정해서 코너가 한 번만 포함되도록 했다.
- 윗변: `[k, M-1-k]` (양쪽 코너 포함)
- 오른변: `[k+1, N-1-k]` (윗 코너 제외, 아래 코너 포함)
- 아랫변: `[M-2-k, k]` (오른 코너 제외, 왼 코너 포함)  
- 왼변: `[N-2-k, k+1]` (아래 코너 제외)

## R을 모듈러 안 한 실수

처음에 `ring = ring[R:] + ring[:R]`로 바로 썼다가, R이 ring 길이보다 큰 경우에
`ring[R:]`이 빈 리스트가 되는 문제가 생겼다.
`r = R % L`을 먼저 계산해야 한다.

## 출력 형식

각 행을 공백으로 구분해서 출력해야 한다.
`' '.join(map(str, row))`를 사용했다. print의 기본 구분자와 혼동하지 않도록 주의.

## 검증 결과

- fixture 테스트: `Results: 2/2 passed, 0 failed`
- 수동 케이스: 3×4 직사각형에서 1회 회전 시 레이어 인덱싱 정확성 확인

## 왜 이 디버그 메모가 중요한가

- `배열 돌리기 1`는 `작은 입력을 안정적으로 읽고, 조건 분기를 코드와 문서로 함께 정리하는 감각`를 연습하는 프로젝트라서, 작은 경계 조건 하나가 전체 상태 전이를 망가뜨릴 수 있다.
- 그래서 이 문서는 "무엇이 틀렸는가"보다 "어떤 징후를 보면 같은 실수를 다시 알아차릴 수 있는가"를 남기는 데 의미가 있다.
- 비슷한 유형을 다시 만났을 때 바로 적용할 수 있는 작은 판단 기준을 남겨 두는 것이 핵심이었다.
- 재현 경로가 필요할 때는 `05-development-timeline.md`의 단계 순서와 이 디버그 로그를 같이 보면, 어느 시점에 어떤 실패를 확인해야 하는지 더 선명해진다.
