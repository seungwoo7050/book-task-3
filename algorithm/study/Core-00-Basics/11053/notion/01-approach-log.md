# 접근 로그

> 프로젝트: 가장 긴 증가하는 부분 수열
> 아래 내용은 `notion-archive/01-approach-log.md`에 남아 있던 기존 학습 기록을 현재 공개 노트 기준으로 다듬은 버전이다.

## 첫 번째 생각: "다 해보면 되지 않나?"

모든 부분 수열을 만들어서 증가하는 것만 골라내고 길이를 재면 된다.
하지만 원소가 $N$개면 부분 수열은 $2^N$개다. $N=1\,000$이면... 말도 안 된다.

이 지점에서 "전부 보지 않고도 정답을 알아내는 방법"이 필요하다는 걸 깨달았다.

## 두 번째 생각: DP로 "각 위치에서의 최선"을 기록하자

핵심 아이디어는 이렇다:
- `dp[i]` = 인덱스 `i`에서 **끝나는** LIS의 길이
- 초기값: 모든 `dp[i] = 1` (자기 자신만으로도 길이 1)
- 전이: `j < i`이고 `a[j] < a[i]`인 모든 `j`에 대해 `dp[i] = max(dp[i], dp[j] + 1)`
- 답: `max(dp)`

이걸 처음 이해했을 때 든 감정은 "아, 이전 결과를 재활용하는 게 이런 거구나"였다.
모든 가능한 이전 위치를 보면서, 현재 원소를 뒤에 붙일 수 있는 가장 긴 수열을 찾는다.

## 왜 O(N²)인데 괜찮은가

$N \le 1\,000$이므로 $N^2 = 10^6$.
Python에서도 $10^6$ 정도의 단순 연산은 문제없이 통과한다.

실제로 더 빠른 $O(N \log N)$ 방법(Patience sorting + 이분탐색)이 있다.
하지만 학습 초기에 이분탐색 방식을 쓰면, "왜 그게 되는지"를 설명하기가 훨씬 어렵다.
DP 방식은 각 단계의 의미가 명확하고, 디버깅할 때 `dp` 배열을 찍어 보면 즉시 상태를 확인할 수 있다.

## 실수하기 쉬운 지점

1. **`<` vs `<=`**: 문제가 "엄격하게 증가"를 요구하므로 `<`를 써야 한다. `<=`를 쓰면 같은 값도 포함되어 답이 커진다.
2. **dp 초기값**: `0`이 아니라 `1`이다. 원소 하나만으로도 길이 1의 LIS다.
3. **답 집계**: 마지막 `dp[n-1]`이 아니라 `max(dp)` 전체를 봐야 한다. LIS가 반드시 마지막 원소에서 끝나지 않기 때문이다.

## Python과 C++ 모두 같은 전략

이 문제에서는 Python과 C++ 구현을 모두 유지했다.
두 언어 모두 동일한 `dp[i]` 정의와 이중 루프 구조를 사용한다.
수동 케이스(`[10, 20, 10, 30, 20, 50, 40, 60]`)에서 두 구현의 출력이 `5`로 일치하는 것을 확인했다.

## 복잡도 정리

| 항목 | 값 |
|------|-----|
| 시간 | $O(N^2)$ |
| 공간 | $O(N)$ |
| N 상한 | 1,000 → $10^6$ 연산, 안전 |
