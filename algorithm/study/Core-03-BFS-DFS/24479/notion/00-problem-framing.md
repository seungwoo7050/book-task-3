# 문제 프레이밍

> 프로젝트: 알고리즘 수업 - 깊이 우선 탐색 1
> 아래 내용은 `notion-archive/00-problem-framing.md`에 남아 있던 기존 학습 기록을 현재 공개 노트 기준으로 다듬은 버전이다.

## 첫인상

1260번에서 DFS 결과를 "방문 순서대로 출력"했다면, 이 문제는 한 단계 더 나아간다. 각 정점이 **몇 번째로** 방문되었는지를 기록해야 한다. 단순한 차이처럼 보이지만, 구현에서는 "카운터를 어떻게 관리할 것인가"라는 새로운 설계 결정이 필요하다.

## 문제 구조 파악

- $N$개 정점, $M$개 간선의 무방향 그래프
- 시작 정점 $R$에서 DFS 수행
- 이웃 중 **번호가 작은 정점** 먼저 방문
- 각 정점의 방문 순서 출력 (방문하지 않았으면 0)

핵심은 "방문 순서"를 단순 리스트가 아닌 **배열 인덱싱**으로 기록하는 것이다.

## 왜 이 문제를 선택했는가

1260의 DFS를 경험한 후, 방문 순서를 '기록'하는 변형을 훈련하기 위해 선택했다. 백준의 "알고리즘 수업" 시리즈는 교과서 의사코드와 실제 구현의 갭을 메우는 데 적합하다.

## 난이도 평가

Bronze 등급이지만, $N$이 최대 100,000까지 가능해서 재귀 한도 관리가 필요하다. 알고리즘 자체보다 Python 런타임 제약(재귀 한도, I/O 속도)을 다루는 실전 경험이 핵심이다.

## 지금 이 프로젝트에서 먼저 고정할 것

- `알고리즘 수업 - 깊이 우선 탐색 1`에서 실제로 확인하려는 학습 목표는 `그래프 표현을 고르고 방문 순서를 안정적으로 제어하는 연습`이다.
- 문제를 읽을 때는 "정답을 맞힌다"보다 어떤 상태와 규칙을 끝까지 유지해야 하는지를 먼저 적어 두는 편이 좋다.
- 비슷한 유형을 다시 만났을 때 바로 적용할 수 있는 작은 판단 기준을 남겨 두는 것이 핵심이었다.
- 학습자 입장에서는 `05-development-timeline.md`를 같이 열어 두면 실제 재현 순서와 문제 해석이 어떻게 맞물리는지 더 잘 보인다.

## 시작 전 성공 기준

- `problem/README.md`의 입력 계약을 내 말로 다시 쓸 수 있는가?
- `python/src/solution.py`를 읽기 전에 어떤 자료구조나 상태 정의가 필요할지 예측했는가?
- `make -C problem test`를 돌렸을 때 어떤 fixture가 왜 필요한지 설명할 수 있는가?
