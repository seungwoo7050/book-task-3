# 회고 — "진짜로 하지 않아도 되는 연산"을 구별하는 눈

## 이 문제가 남긴 것

### 논리적 회피의 발상
R을 만날 때마다 실제로 뒤집는 것은 O(n)이지만, 플래그를 토글하면 O(1)이다.
이건 단순한 최적화가 아니라, "연산의 관찰 가능한 효과만 유지하면 된다"는 사고의 전환이다.
이 사고방식은 이후 lazy propagation, 세그먼트 트리 등에서도 반복된다.

### 입력 파싱의 중요성
이 문제에서 실제로 가장 많이 실수하는 부분은 알고리즘이 아니라 JSON-like 배열 파싱이다.
빈 배열, 원소 하나, 대괄호/콤마 처리 등 "파싱 로직이 알고리즘 못지않게 중요하다"는 걸 체감했다.

## 다음에 적용할 것
- 실제 연산 대신 논리적 플래그로 처리 가능한지 먼저 검토
- 문자열 파싱은 edge case를 항상 따로 확인 (빈 입력, 원소 1개 등)
