### Session 2
simulator가 안정되니까 이제 transpose를 건드렸다. `32x32`부터 시작.

첫 시도는 naive double loop. miss가 `400`대로 나왔다. 목표는 `<300`. 8x8 block으로 나누니까 `300` 근처까지 내려왔는데 거기서 막혔다. diagonal block에서 conflict miss가 유독 많이 나오는 걸 보고 처음엔 "block size를 바꿔야 하나" 생각했다.

그런데 원인이 block size가 아니었다. direct-mapped cache에서 `A[i][j]`와 `B[j][i]`가 같은 cache line에 map되는 경우가 있다. diagonal block(i == j인 8x8 구역)에서 이게 특히 심했다. 해결은 생각보다 간단했다. diagonal element를 바로 쓰지 않고, 임시 변수에 보관했다가 같은 row의 나머지를 다 처리한 뒤에 쓴다.

```c
    for (ii = 0; ii < 32; ii += 8) {
        for (jj = 0; jj < 32; jj += 8) {
            for (i = ii; i < ii + 8; ++i) {
                int diag_value = 0, diag_index = -1;
                int j;
                for (j = jj; j < jj + 8; ++j) {
                    int value = load_a(32, A, i, j, cache);
                    if (ii == jj && i == j) {
                        diag_value = value;
                        diag_index = i;
                    } else {
                        store_b(32, B, j, i, value, cache);
                    }
                }
                if (diag_index >= 0)
                    store_b(32, B, diag_index, diag_index, diag_value, cache);
            }
        }
    }
```

이 짧은 조건 하나로 `32x32` miss가 `284`로 내려갔다. 목표 `<300` 달성. 처음엔 대단한 알고리즘이 필요하다고 생각했는데, 결국은 "어떤 순서로 쓰느냐"가 cache 동작을 결정한다는 아주 기본적인 사실이었다.
