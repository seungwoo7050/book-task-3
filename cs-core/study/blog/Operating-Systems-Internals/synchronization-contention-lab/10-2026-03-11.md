# Synchronization Contention Lab 개발 타임라인 (2026-03-11)

## 2026-03-11
### Session 1
세 가지 contention scenario를 구현했다: mutex counter, semaphore gate, bounded buffer. 처음엔 세 개를 독립적으로 보고 시작했는데, 나중엔 "공통 패턴이 있다"는 게 보였다.

공통 패턴: (1) 공유 상태를 가진 중앙 객체, (2) 여러 스레드가 이 객체를 놓고 경쟁, (3) 대기 이벤트를 count해서 contention을 측정.

counter 시나리오가 가장 단순했다. mutex 하나로 `counter`를 보호한다. 스레드 수를 늘리면 wait_events가 올라간다.

```c
static void *counter_worker(void *raw_arg) {
    counter_arg_t *arg = (counter_arg_t *)raw_arg;
    counter_state_t *state = arg->state;
    int i;
    for (i = 0; i < state->iterations; ++i) {
        if (pthread_mutex_trylock(&state->lock) != 0) {
            state->wait_events++;
            pthread_mutex_lock(&state->lock);
        }
        state->counter++;
        pthread_mutex_unlock(&state->lock);
    }
    return NULL;
}
```

`pthread_mutex_trylock`으로 먼저 non-blocking 시도를 하고, 실패하면 `wait_events`를 올린 뒤 blocking lock을 한다. 이렇게 하면 "경쟁이 얼마나 심한가"를 측정할 수 있다.

처음엔 `wait_events++`가 mutex 바깥에서 일어나서 race라는 걸 놓쳤다. 여러 스레드가 동시에 trylock에서 실패하면 `wait_events`를 동시에 증가하니까 count가 부정확해진다. 하지만 이 프로젝트에서는 정확한 count보다 "경쟁의 상대적 크기"를 보는 게 목적이므로 수용했다.

### Session 2
bounded buffer의 producer-consumer가 예상보다 복잡했다. condition variable 두 개가 필요하다: `not_empty`(소비자가 기다림)와 `not_full`(생산자가 기다림).

```c
typedef struct {
    pthread_mutex_t lock;
    pthread_cond_t not_empty;
    pthread_cond_t not_full;
    int buffer_count;
    int capacity;
    ...
} buffer_state_t;
```

처음엔 condition variable 하나로 해보려고 했다. `pthread_cond_broadcast`로 모두를 깨우면 되지 않나? 이게 동작은 하지만 불필요한 wake가 많아진다. `not_empty`와 `not_full`을 분리하면 생산자는 `not_full`을, 소비자는 `not_empty`를 wait하고, signal도 정확한 대상에게만 보낼 수 있다.

bounded buffer에서 `overflow`와 `underflow` 카운터도 추가했다. 이건 동기화 버그를 잡기 위한 방어적 측정이다. 올바른 구현에서는 항상 0이어야 한다.

### Session 3
CLI:

```bash
$ cd c
$ make clean && make
$ ./contention_lab counter 4 1000
$ ./contention_lab gate 6 3 500
$ ./contention_lab buffer 3 2 200
```

세 scenario를 모두 돌렸다. counter에서 스레드 수를 늘릴수록 wait_events가 선형 이상으로 늘어났다. gate의 `max_concurrency`를 1로 줄이면 mutex와 동일한 패턴이 됐다.

이 프로젝트에서 핵심으로 체감한 건, "synchronized"와 "correct"는 다른 문제라는 것이다. mutex로 보호한다고 해서 프로그램이 올바른 게 아니다. 어떤 invariant를 언제 보호해야 하는지를 먼저 정하고, 그다음 synchronization 메커니즘을 고른다.
