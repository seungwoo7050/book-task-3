# Virtual Memory Lab 개발 타임라인 (2026-03-11)

## 2026-03-11
### Session 1
네 가지 page replacement policy(FIFO, LRU, Clock, OPT)를 하나의 `simulate_policy` 함수로 처리하는 구조를 잡는 게 처음부터 도전이었다.

가장 먼저 한 건 각 policy의 "eviction 결정 로직"을 분리하는 것이었다. FIFO는 가장 오래 전에 load된 frame을, LRU는 가장 오래 전에 사용된 frame을, Clock은 referenced bit이 0인 frame을, OPT는 앞으로 가장 늦게 쓰일 frame을 evict한다.

```python
@dataclass
class Frame:
    page: int
    dirty: bool
    loaded_at: int
    last_used: int
    referenced: bool = True
```

`Frame` dataclass에 `loaded_at`, `last_used`, `referenced`를 모두 넣어 두니까 각 policy가 필요한 필드만 골라 쓸 수 있었다. 처음엔 policy별로 다른 frame 구조가 필요할 줄 알았는데, 필드를 넉넉하게 넣으면 하나로 통합할 수 있었다.

### Session 2
Clock algorithm에서 멈췄다. "clock hand가 한 바퀴 돌면서 referenced bit이 0인 첫 번째 frame을 evict하고, 1이면 0으로 바꾸고 넘어간다"는 설명은 알고 있었는데, 구현에서 clock_hand를 전역으로 관리하는 게 맞는지, 아니면 state로 넘기는 게 맞는지 헷갈렸다.

결국 `simulate_policy` 함수 안에서 `clock_hand` 변수를 local로 유지했다. eviction이 일어날 때마다 hand를 advance하고, 아닐 때는 그대로 둔다.

처음 구현했을 때 clock algorithm의 fault count가 FIFO보다 높게 나왔다. 뭔가 잘못됐다고 생각해서 trace를 직접 그렸더니, referenced bit 초기화를 page access 때마다 하지 않고 eviction 시에만 했다. hit이 일어날 때도 `referenced = True`로 갱신해야 했다.

### Session 3
OPT는 미래를 봐야 하니까 온라인으로 구현하면 안 된다. trace를 완전히 받아 놓고 retrospective하게 eviction을 결정한다.

```python
future_uses = {
    page: [i for i, acc in enumerate(trace) if acc.page == page]
    for page in {acc.page for acc in trace}
}
```

미래 사용 시점 dict를 만들어 두고, eviction이 필요할 때 frame 중 가장 나중에 사용되는(또는 사용되지 않는) 것을 고른다. OPT는 이론적 lower bound를 제공하니까 다른 policy의 성능을 평가하는 기준이 된다.
