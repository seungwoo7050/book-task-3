# Scheduling Simulator 개발 타임라인 (2026-03-11)

## 2026-03-11
### Session 1
처음에 FCFS → SJF → RR → MLFQ 순서로 구현했는데, 각 policy마다 metric 계산 방식을 다르게 짜기 시작했다가 멈췄다. "policy별로 metric 계산식이 다르면 비교 자체가 의미 없어진다"는 걸 깨달은 시점이다.

metric 계산을 `ProcessState`로 통합했다. 어떤 policy든 `start`, `completion`, `arrival`이 기록되면 `waiting_time`, `response_time`, `turnaround_time`을 항상 같은 공식으로 계산한다.

```python
@dataclass(frozen=True)
class ProcessMetric:
    pid: str
    waiting_time: int
    response_time: int
    turnaround_time: int
```

이 dataclass가 생기기 전에는 policy별로 dict를 반환했는데, key를 오타 내면 runtime에서야 실패했다. frozen dataclass로 바꾸니까 at least IDE에서 attribute 이름이 잡히게 됐다.

### Session 2
MLFQ에서 boost 타이밍이 문제였다. idle fill, arrival enqueue, boost boundary, level demotion의 순서를 어떻게 잡느냐에 따라 replay 문자열이 달라진다.

처음엔 "tick 중간에 boost를 한다"는 아이디어로 짰는데, 같은 fixture에서 돌릴 때마다 결과가 약간씩 달라서 golden test가 계속 flaky했다. 원인을 추적하니 boost가 process arrival과 겹치는 tick에서 처리 순서가 불명확했다.

결론: boost를 dispatch boundary에서만 처리한다. tick을 advance하기 전에 boosted process를 꺼내고, 그다음 arrival을 처리한다. 이 순서를 고정하니까 replay가 결정론적으로 됐다.

CLI:

```bash
$ cd python
$ pip install -e .
$ pytest tests/ -v
$ python -m os_scheduling convoy.json --policy mlfq
```

FCFS, SJF, RR, MLFQ 네 policy가 모두 결정론적 replay를 출력한다.

- SJF: 짧은 job을 먼저 실행해서 convoy effect를 완화한다. waiting time이 FCFS보다 낮다.
- RR: response time이 제일 좋다. quantum이 1일 때 모든 process가 1 tick 안에 첫 응답을 받는다.
- MLFQ: interactive process(짧은 burst)가 background process보다 높은 queue에서 오래 산다. 실제 OS의 priority 조정을 단순화한 모델이다.

### Session 3
`load_fixture` 경계 검증이 필요했다. `burst <= 0`인 fixture를 넣으면 simulation이 무한 루프에 빠진다. 처음엔 아무것도 검사하지 않았다가, invalid fixture 테스트를 짜면서 추가했다.

```python
def load_fixture(path: str | Path) -> list[ProcessSpec]:
    data = json.loads(Path(path).read_text())
    specs = [ProcessSpec(**item) for item in data]
    for spec in specs:
        if spec.arrival < 0 or spec.burst <= 0:
            raise ValueError("arrival must be >= 0 and burst must be > 0")
    return specs
```

이 검증이 없으면 CLI에서 오류 메시지 없이 멈춰 버린다. 사용자 입력 경계에서 검증하는 게 얼마나 중요한지를 직접 체험했다.
