# Parser-Interpreter 개발 타임라인 (2026-03-11)

## 2026-03-11
### Session 1
Lexer를 먼저 짰다. 처음엔 regex로 한 번에 모든 토큰을 잡으려고 했는데, 같은 문자가 맥락에 따라 다른 토큰이 되는 경우(예: `-`이 단항 연산자인지 이항 연산자인지)는 regex 단계에서 구분이 안 된다. Lexer는 문자열만 보고 토큰 종류를 결정하고, 역할 구분은 parser에서 한다.

처음에 keyword를 identifier와 구분하는 방법을 못 정했다. "let"이 keyword인데, lexer가 identifier "let_value"도 le트로 시작하니까 헷갈릴 것 같았다. 결론은 간단했다: maximal munch로 가장 긴 identifier를 읽고, reserved words 목록에 있으면 keyword 토큰으로 바꾼다.

### Session 2
Parser에서 `let expr = ... in expr` 구조가 예상보다 까다로웠다. `in` 이후의 expression은 `=` 오른쪽 expression보다 우선순위가 낮아야 한다. 처음에 단순히 `parse_expression`을 재귀 호출했더니 `in` 이후 expression이 탐욕적으로 너무 많이 먹었다.

예: `let x = 1 in x + 2`에서 `x + 2`가 `in`의 body여야 하는데, 우선순위를 잘못 잡으면 `x + (2 in something_else)` 형태로 파싱됐다.

결국 `let`과 `fun` 표현식을 binary operator 우선순위 아래로 두고, `parse_expression`이 이 둘을 먼저 확인하도록 했다. Pratt parser 기법은 아니고 간단한 recursive descent지만, 이 프로젝트 크기에서는 충분했다.

### Session 3
Evaluator에서 closure를 구현할 때 environment 캡처가 핵심이었다.

```python
@dataclass(frozen=True)
class Closure:
    params: tuple[Param, ...]
    return_annotation: object | None
    body: Expr
    environment: Environment
```

`Closure`가 environment를 직접 들고 있다. 이렇게 하면 함수가 정의된 시점의 environment가 나중에 호출될 때도 유효하다.

처음엔 closure가 왜 필요한지 잘 느껴지지 않았다. `let f = fun (x) = x + y in f(1)`에서 `y`가 `f` 정의 시점의 scope에 있다면, `f`가 호출될 때 `y`를 어떻게 찾는가? `f`가 `environment`를 capture하고 있으니까, 호출 시 그 environment에서 `y`를 찾는다. 이게 closure semantics다.
