`floatFloat2Int` C++ 버전을 옮기는데 signed shift 해석이 미묘하게 달라 보여서 멈추었다. C에서는 `int` 타입의 signed right shift가 implementation-defined이긴 하지만 대부분의 컴파일러에서 arithmetic shift다. C++도 마찬가지이긴 한데, `unsigned`를 `int`로 캐스팅하는 시점에서 behavior가 다를 수 있다는 의심이 들었다.

결론부터 말하면 — 실제로는 차이가 없었다. 같은 컴파일러(clang)를 쓰니까 당연히 같은 결과가 나왔다. 하지만 이 과정에서 float 함수들의 분기 구조를 일부러 C와 동일하게 유지한 것이, 나중에 "왜 이 두 구현이 같은 edge case를 통과하는가"를 설명하기 훨씬 편하게 만들었다.

이 구간에서 판단을 고정한 코드는 아래였다.

```c
int floatFloat2Int(unsigned uf) {
    int sign = uf >> 31;
    int exp = ((uf >> 23) & 0xFF) + (~127 + 1);
    int frac = (uf & 0x7FFFFF) | 0x800000;

    if (((uf >> 23) & 0xFF) == 0xFF) {
        return 0x80000000u;
    }
    if (exp < 0) {
        return 0;
    }
    if (exp > 30) {
        return 0x80000000u;
    }
    if (exp > 23) {
        frac = frac << (exp + (~23 + 1));
    } else {
        frac = frac >> (23 + (~exp + 1));
    }
    return sign ? (~frac + 1) : frac;
```

`floatFloat2Int`는 NaN, 범위 밖 exponent, 음수/양수 반환을 한 함수 안에서 정리한다. 공식 verifier를 통과하려면 산술 트릭보다 먼저 “어떤 입력이 sentinel로 떨어지는가”를 분기별로 명확히 유지해야 한다는 점이 이 코드에 그대로 남아 있다.

마지막으로 `floatScale2`에서 하나 남은 edge case를 잡았다. denorm 입력(`exp == 0`)에서 mantissa를 1-bit 왼쪽으로 shift하면 자동으로 norm으로 넘어간다는 게 처음엔 명확하지 않았다. denorm의 implicit leading bit이 0이고, shift 후 exponent field에 carry가 생기면 `0x00800000`이 되어 exp=1 normalized로 해석된다. 이를 C 코드로 표현하면 단순한 조건 분기인데, 수식 뒤의 이유를 설명할 수 있어야 의미가 있다.
