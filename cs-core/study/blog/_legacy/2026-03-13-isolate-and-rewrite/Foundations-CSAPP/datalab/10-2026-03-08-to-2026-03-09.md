## 2026-03-08
### Session 1
처음엔 `bits.c` 계약서를 열어 놓고 쉬워 보이는 것부터 건드렸다. `bitXor`는 드모르간 법칙 한 줄이니까 금방 끝날 거라 생각했는데, 실제로는 그랬다. 문제는 그 다음이었다.

`isTmax`에서 처음 막혔다. `INT_MAX`를 판별해야 하는데 조건문도 비교 연산자도 쓸 수 없다. 처음 시도는 `x + 1`이 `INT_MIN`인지 확인하는 방식이었는데, `-1`도 같은 조건을 만족해 버린다는 걸 테스트에서 맞고 나서야 알았다.

```c
int isTmax(int x) {
    int y = x + 1;
    return !(~(x + y)) & !!y;
}
```

`!!y`를 붙인 건 `-1`을 걸러내기 위해서다. 처음엔 왜 `-1`이 문제인지도 바로 떠오르지 않았다. `-1 + 1 = 0`이니까 `~(x + y) = ~(-1 + 0) = ~(-1) = 0`이 되어 이것도 통과해 버리는 거였다. 이걸 깨닫기까지 `btest`를 세 번은 돌렸다.

- 판단: 정수 퍼즐은 "범용 공식을 찾자"보다 "어떤 edge case가 공식을 깨뜨리는가"부터 보는 편이 더 안전하다.

`isLessOrEqual`에서도 비슷하게 당했다. 처음에는 단순히 `y - x >= 0`이면 된다고 봤는데, 부호가 다른 두 값의 뺄셈은 overflow로 바로 터진다. `INT_MIN`과 `INT_MAX`를 넣었을 때 뺄셈 결과 자체가 wrap around 되니까, 부호 비트를 먼저 분리한 뒤에야 비교가 성립했다.

```c
int isLessOrEqual(int x, int y) {
    int sx = x >> 31;
    int sy = y >> 31;
    int diff_sign = sx ^ sy;
    int diff = y + (~x + 1);
    int diff_nonneg = !(diff >> 31);
    return (diff_sign & !!sx) | ((!diff_sign) & diff_nonneg);
}
```

이 시점에서 정수 퍼즐의 패턴이 보이기 시작했다. 부호 비트를 먼저 분리하고, all-ones mask를 만들어 조건 분기를 흉내 내는 게 거의 모든 함수의 뼈대였다.

### Session 2
`logicalNeg`을 잡다가 한참 헤맸다. `0`이면 `1`, 아니면 `0`을 반환해야 하는데, 비교 연산자 없이. 처음 가설은 "모든 비트를 OR로 접어서 최하위 비트를 보면 되지 않나"였는데, 정수 연산만으로 32비트 전부를 접는 방법이 바로 떠오르지 않았다.

결국 핵심 아이디어는 이거였다. `0`만이 자기 자신과 negate를 OR했을 때 sign bit가 `0`이다. 음수는 sign bit가 1이고, 양수는 negate하면 음수가 되니까 OR 결과의 sign bit가 1이다. `INT_MIN`은 negate해도 자기 자신이라서 역시 sign은 1.

```c
int logicalNeg(int x) {
    return ((x | (~x + 1)) >> 31) + 1;
}
```

이걸 떠올리기까지의 과정: (1) 비트를 fold하려고 `x |= x >> 16; x |= x >> 8; ...` 같은 걸 시도 → operator 수 초과. (2) 음수면 sign이 1이니까 `x >> 31`로 끝나나 → 양수가 안 됨. (3) `x`와 `-x`를 둘 다 보면? → 여기서 풀렸다.

## 2026-03-09
### Session 1
어제 정수 퍼즐을 마무리하고, 오늘은 `howManyBits`부터 시작했다. 이 함수 하나가 다른 정수 퍼즐 전부보다 더 오래 걸렸다.

처음 접근은 무식했다. 비트를 하나씩 세는 방법을 생각했는데, 루프를 쓸 수 없으니까 32번의 shift를 일일이 작성하려고 했다. 당연히 operator 제한에 걸린다.

두 번째 시도. 이진 탐색처럼 접근할 수 있다는 힌트를 어디선가 봤다. "상위 16비트에 뭔가 있으면 최소 16비트는 필요하다" — 이 아이디어로 범위를 절반씩 좁히는 구조가 나왔다.

그런데 음수 처리에서 다시 막혔다. `-1`은 `howManyBits(-1) = 1`이어야 하는데, 음수를 그대로 두면 상위 비트가 전부 1이라 항상 32가 나온다. 여기서 `x ^ sign`으로 음수를 뒤집는 발상이 나왔다. 양수로 바꾼 뒤 "최상위 1-bit 위치 + 부호 비트 1"을 구하면 된다.

```c
int howManyBits(int x) {
    int sign = x >> 31;
    int b16, b8, b4, b2, b1, b0;
    x = x ^ sign;
    b16 = !!(x >> 16) << 4;  x >>= b16;
    b8  = !!(x >> 8)  << 3;  x >>= b8;
    b4  = !!(x >> 4)  << 2;  x >>= b4;
    b2  = !!(x >> 2)  << 1;  x >>= b2;
    b1  = !!(x >> 1);        x >>= b1;
    b0  = x >> 0;
    return b16 + b8 + b4 + b2 + b1 + b0 + 1;
}
```

처음엔 `x = x ^ sign`이 왜 음수에서 동작하는지도 확신이 없어서 `-1`, `-128`, `INT_MIN`을 손으로 돌려 봤다. `-1 ^ (-1) = 0` → `howManyBits = 1`. `-128 ^ (-1) = 127 = 0b1111111` → 7 + 1 = 8. 맞았다.

### Session 2
float 세 함수로 넘어갔다. 정수 퍼즐과 완전히 다른 세계였다.

`floatFloat2Int`에서 처음 빠진 함정은 exponent bias를 빼는 방식이었다. `exp = ((uf >> 23) & 0xFF) - 127`이라고 쓰면 끝날 줄 알았는데, 뺄셈 연산자를 쓸 수 없다. `~127 + 1`로 negate해야 했다. 이건 정수 퍼즐에서 배운 `negate` 패턴을 그대로 가져온 거였다.

진짜 문제는 분기 정리였다. NaN/inf, 너무 작은 exponent, 너무 큰 exponent, 정상 범위 — 이 네 경우를 깔끔하게 나누는 게 쉽지 않았다. 처음에는 if 문 순서를 잘못 잡아서 denormalized number가 `0x80000000u` sentinel로 떨어지는 버그를 냈다.

```c
int floatFloat2Int(unsigned uf) {
    int sign = uf >> 31;
    int exp = ((uf >> 23) & 0xFF) + (~127 + 1);
    int frac = (uf & 0x7FFFFF) | 0x800000;
    if (((uf >> 23) & 0xFF) == 0xFF) return 0x80000000u;
    if (exp < 0)  return 0;
    if (exp > 30) return 0x80000000u;
    if (exp > 23) frac = frac << (exp + (~23 + 1));
    else          frac = frac >> (23 + (~exp + 1));
    return sign ? (~frac + 1) : frac;
}
```

나중에 보니 분기 순서가 이 함수의 전부였다. NaN/inf를 먼저 걸러내지 않으면 `exp` 계산 자체가 의미 없어진다. `exp < 0`을 그 다음에 두는 이유도, denormalized는 어차피 정수로 바꾸면 0이니까. 이 순서를 처음부터 알았으면 훨씬 빨랐을 텐데, 실제로는 세 번 정도 순서를 바꿔 보면서 edge case 테스트를 돌린 뒤에야 확정됐다.
