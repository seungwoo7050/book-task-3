# Attack Lab 개발 타임라인 (2026-03-08 ~ 2026-03-09)

## 2026-03-08
### Session 1
공개 경계 문제가 bomblab보다 더 심각했다. 여기는 실제 exploit payload를 다루니까, raw byte를 그대로 올리면 "공격 도구 배포"가 되어 버린다. companion validator라는 개념을 bomb에서 가져왔는데, attack쪽은 조금 달랐다. 정답 자체가 아니라 "이 payload가 만족해야 하는 바이트 구조"를 조건식으로 검증하는 방식이다.

payload file을 hex text로 읽어 byte 배열로 변환하는 경로를 먼저 세웠다. phase 1은 단순한 return address overwrite — buffer를 넘겨서 `touch1`의 주소로 뛰는 것만 하면 된다.

### Session 2
phase 2와 3에서 code injection을 다뤘다. 처음엔 "buffer에 shellcode를 넣고 거기로 jump하면 된다"는 간단한 그림을 그렸는데, 실제로는 더 섬세했다.

phase 2는 `rdi`에 cookie 값을 넣고 `touch2`로 뛰어야 한다. buffer 시작 지점에 `mov` + `push` + `ret` sequence를 심고, return address를 buffer 시작으로 덮어쓴다. phase 3은 비슷하지만 cookie를 문자열로 넘겨야 해서, string을 payload 끝에 배치하고 그 주소를 `rdi`에 넣어야 한다.

```c
int attacklab_phase_2(const unsigned char *bytes, size_t len)
{
    static const unsigned char prefix[] = {
        0x48, 0xc7, 0xc7, 0x4d, 0x3c, 0x2b, 0x1a,
        0x68, 0xec, 0x17, 0x40, 0x00, 0xc3
    };
    return len == 48 &&
           memcmp(bytes, prefix, sizeof(prefix)) == 0 &&
           matches_u64_le(bytes + BUFFER_SIZE, BUFFER_START);
}
```

처음엔 바이트 하나하나를 맞추려고 했는데, 나중에 보니 핵심은 세 가지뿐이었다. (1) injection code prefix가 맞는가, (2) return address가 buffer 시작을 가리키는가, (3) phase 3에서는 cookie string이 올바른 위치에 있는가. 이 세 조건을 companion validator에 박아 두니까, payload가 "왜" 동작하는지가 조건식 자체에서 읽히게 됐다.

## 2026-03-09
### Session 1
phase 4~5 ROP chain으로 넘어갔다. NX bit가 켜져 있어서 더 이상 buffer에 코드를 심을 수 없다. 대신 기존 코드의 gadget을 체인으로 연결한다.

phase 4가 의외로 간결했다. `POP_RAX` → `MOV_RAX_RDI` → `TOUCH2`. 세 gadget만 연결하면 된다. 하지만 phase 5는 stack-relative string addressing이 필요해서 gadget 수가 확 늘었다. `MOV_RSP_RAX`로 stack pointer를 잡고, offset `48`을 더해서 string 위치를 계산하고, 그걸 `rdi`에 넣는다.

```c
int attacklab_phase_5(const unsigned char *bytes, size_t len)
{
    return len == 105 &&
           matches_u64_le(bytes + 40, GADGET_MOV_RSP_RAX) &&
           matches_u64_le(bytes + 48, GADGET_MOV_RAX_RDI) &&
           matches_u64_le(bytes + 56, GADGET_POP_RSI) &&
           matches_u64_le(bytes + 64, 48ULL) &&
           matches_u64_le(bytes + 72, GADGET_LEA_RDI_RSI_RAX) &&
           matches_u64_le(bytes + 80, GADGET_MOV_RAX_RDI) &&
           matches_u64_le(bytes + 88, TOUCH3) &&
           memcmp(bytes + 96, COOKIE_ASCII, sizeof(COOKIE_ASCII)) == 0;
}
```

phase 5의 `48ULL` offset이 처음엔 마법 숫자처럼 보였는데, payload layout을 그려 보니 stack에서 gadget chain이 끝난 뒤 cookie string이 놓이는 위치까지의 거리였다. payload의 물리적 layout을 한 번 그려 보고 나서야 offset의 의미가 잡혔다.
