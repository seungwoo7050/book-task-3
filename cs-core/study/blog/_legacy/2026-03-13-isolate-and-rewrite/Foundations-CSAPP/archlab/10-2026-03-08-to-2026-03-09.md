### Session 2
Part B에서 `iaddq` instruction을 SEQ에 추가하는 게 생각보다 까다로웠다. HCL에서 signal을 수정하는 건 한 줄인데, 그 한 줄이 pipeline의 어느 stage에 영향을 미치는지 머리로 그리기가 어렵다. 처음 시도에서는 `aluA`, `aluB` 선택만 바꾸면 된다고 생각했는데, `dstE` 설정을 빠뜨려서 레지스터에 결과가 안 쓰였다.

이 실수를 잡은 뒤에야 "SEQ에서 instruction을 추가한다는 건 fetch/decode/execute/memory/writeback 다섯 단계 전부에 해당 opcode의 흔적을 남긴다"는 당연한 사실이 체감이 됐다.

## 2026-03-09
### Session 1
Part C로 넘어왔다. `ncopy.ys`의 baseline은 element를 하나씩 옮기면서 양수인지 확인한다. 여기서 CPE를 낮추려면 loop unrolling이 핵심인 건 알았는데, 문제는 "얼마나 unroll할 것인가"와 "remainder를 어떻게 처리할 것인가"였다.

처음에는 8-way로 해봤다가 코드가 너무 길어져서 benchmark의 instruction memory 제한에 걸렸다. 4-way로 줄이니까 제한은 통과하면서도 CPE가 눈에 띄게 내려갔다.

```c
NcopyReport arch_ncopy_optimized(const int64_t *src, int64_t *dst, int64_t len)
{
    NcopyReport report;
    int64_t count = 0;
    int64_t index = 0;

    while (index + 3 < len) {
        int64_t v0 = src[index];
        int64_t v1 = src[index + 1];
        int64_t v2 = src[index + 2];
        int64_t v3 = src[index + 3];
        dst[index] = v0;  dst[index + 1] = v1;
        dst[index + 2] = v2;  dst[index + 3] = v3;
        count += v0 > 0;  count += v1 > 0;
        count += v2 > 0;  count += v3 > 0;
        index += 4;
    }
    while (index < len) {
        dst[index] = src[index];
        if (src[index] > 0) count += 1;
        index += 1;
    }
    report.count = count;
    report.cycles = optimized_cycles(len);
    report.cpe = len > 0 ? (double)report.cycles / (double)len : 0.0;
    return report;
}
```

처음엔 이 함수가 단순한 "빠른 복사"라고만 생각했는데, 나중에 보니 핵심은 "count와 copy를 같은 루프에서 동시에 처리하면서도 data hazard를 줄이는 것"이었다. 4개를 먼저 전부 읽고 나서 전부 쓰는 순서가 pipeline stall을 줄여 주는 포인트다.
