### Session 1
처음 구현 대상을 정할 때 세 가지를 동시에 생각했다: HTTP request 파싱, 서버 forwarding, connection 처리. 이걸 한꺼번에 짜면 어디서 뭐가 틀린지 알 수 없다. 그래서 순서를 정했다: 파싱 → 단일 스레드 sequential proxy → 멀티스레드 → 캐시.

`parse_uri`가 처음엔 간단해 보였다. URI에서 host, port, path를 분리하면 된다. 그런데 port가 없는 경우(기본 80), path가 없는 경우(루트 `/`), HTTPS가 아닌 HTTP만 처리하는 경우를 각각 예외 처리해야 했다.

```c
static int parse_uri(const char *uri, char *host, char *port, char *path)
{
    const char *host_start;
    const char *colon_pos;
    const char *slash_pos;
    ...
    if (strncmp(uri, "http://", 7) != 0) return -1;
    host_start = uri + 7;
    slash_pos = strchr(host_start, '/');
    colon_pos = (slash_pos != NULL)
        ? memchr(host_start, ':', (size_t)(slash_pos - host_start))
        : strchr(host_start, ':');
    ...
}
```

`slash_pos`를 기준으로 port 코론을 찾는 범위를 제한한 건, `path`에 들어 있는 `:`를 port 구분자로 잘못 인식하지 않기 위해서다. 이 케이스를 처음에는 생각 못 했다가 테스트에서 틀렸다.

### Session 2
single-thread sequential proxy를 돌렸다.

CLI:

```bash
$ cd c
$ make clean && make
$ ./proxy 8080 &
$ curl -x localhost:8080 http://example.com/
```

기본 GET 요청이 통했다. 그런데 `User-Agent` header를 교체하는 로직에서 버그가 있었다. 원래 클라이언트의 User-Agent를 그대로 forwarding하지 않고 우리 값으로 교체해야 하는데, 처음 `build_request`에서 원본 header를 그냥 통과시켰다.

수정: header 이름을 소문자 비교(`strcasecmp`)로 확인하고, `User-Agent`, `Host`, `Connection`, `Proxy-Connection`을 직접 설정하는 header 목록에서 제외한다.

## 2026-03-09
### Session 1
thread pool로 전환했다. 처음엔 요청마다 새 스레드를 spawn하려고 했는데, 연속된 요청에서 스레드 생성 비용이 눈에 띄었다. producer-consumer 구조로 바꿨다: main thread가 accept한 소켓 fd를 queue에 넣고, worker pool이 꺼내서 처리한다.

```c
static void *thread_main(void *arg)
{
    pthread_detach(pthread_self());
    while (1) {
        int connfd = *(int *)arg;
        free(arg);
        handle_client(connfd);
    }
    return NULL;
}
```

실제로는 thread pool을 완전히 구현하기보다 thread per connection + detach 방식을 유지했다. `pthread_detach`로 자원 누수를 막고, 각 스레드가 독립적으로 처리한다. 여기서 놓친 게 하나 있었는데 — `arg`로 넘기는 `connfd` 포인터를 heap에 할당해야 한다. stack 변수의 주소를 넘기면 다음 연결에서 ANSI C UB가 생긴다.

LRU 캐시도 이 세션에 붙였다. `cache_lock` mutex + doubly linked list. cache hit이면 entry를 head로 올리고, miss이면 추가한다. 용량을 초과하면 tail에서 evict한다.
