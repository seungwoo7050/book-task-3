# Shell Lab 개발 타임라인 (2026-03-08 ~ 2026-03-09)

## 2026-03-08
### Session 1
shell 구현에서 가장 먼저 헷갈린 건 SIGCHLD handler의 역할이었다. 처음에는 "child가 끝나면 waitpid로 거두면 된다"고만 알았는데, 실제로는 child가 stopped 상태가 되어도 SIGCHLD가 온다. waitpid 호출 시 flags를 어떻게 쓰느냐에 따라 job table 갱신 방식이 달라진다.

```c
static void sigchld_handler(int sig)
{
    int saved_errno = errno;
    pid_t pid;
    int status;
    while ((pid = waitpid(-1, &status, WNOHANG | WUNTRACED)) > 0) {
        if (WIFSTOPPED(status)) {
            struct job_t *job = getjobpid(jobs, pid);
            if (job != NULL) job->state = ST;
        } else {
            deletejob(jobs, pid);
        }
    }
    errno = saved_errno;
}
```

`WNOHANG | WUNTRACED`를 처음 봤을 때 두 flag가 각각 뭘 하는지 헷갈렸다. `WNOHANG`은 "대기 중인 child가 없으면 즉각 반환", `WUNTRACED`는 "stopped child도 반환". 두 개를 같이 써야 handler가 blocking하지 않고 stopped child도 처리한다.

`errno`를 저장했다가 복구하는 이유도 처음엔 불명확했다. handler가 `waitpid` 중에 errno를 변경할 수 있어서 main loop의 errno가 덮어씌워지면 안 된다. async signal safety를 위한 관례다.

### Session 2
`eval`에서 foreground job을 기다리는 `waitfg` 구현이 까다로웠다. 처음엔 단순히 busy loop로 했다가 CPU를 100% 먹는 걸 보고 `sigsuspend`로 바꿨다.

```c
static void waitfg(pid_t pid)
{
    sigset_t mask;
    sigemptyset(&mask);
    while (fgpid(jobs) == pid) {
        sigsuspend(&mask);
    }
}
```

`sigsuspend`가 mask와 함께 호출되면 해당 signal을 unblock한 채로 signal을 기다린다. signal이 오면 handler가 먼저 실행되고, `sigsuspend`는 반환한다. 그럼 while 조건 재평가 후 loop를 계속하거나 빠져나온다.

이 사이클을 이해하기까지 SIGCHLD handler와 `waitfg`의 관계를 꽤 오래 들여다봤다. "signal handler가 job table을 바꾸고, waitfg는 job table을 본다"는 연결이 핵심이었다.
