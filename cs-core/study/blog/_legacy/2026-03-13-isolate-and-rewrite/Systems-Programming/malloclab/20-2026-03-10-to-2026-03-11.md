# Malloc Lab 개발 타임라인 (2026-03-10 ~ 2026-03-11)

## 2026-03-10
### Session 1
official mdriver에서 trace별 결과를 보면서 어떤 trace가 내 allocator와 잘 맞는지 봤다. 작은 size 할당이 많은 trace는 utilization이 좋았지만, 큰 size realloc이 반복되는 trace에서 fragmentation이 심했다.

explicit free list로 넘어갈까 생각했다. explicit free list는 free block을 따로 doubly linked list로 관리하니까 first-fit search가 O(free block 수)가 되는 대신 전체 heap을 스캔하지 않아도 된다. 하지만 header/footer에 더해 prev/next pointer를 block 안에 저장해야 하고, `MIN_BLOCK_SIZE`가 더 커진다.

이미 내 구현에서 `next_free_slot(block)`, `prev_free_slot(block)`으로 pointer를 관리하고 있었으니까, 이미 어느 정도 explicit list에 가까웠다.

```c
static void **next_free_slot(void *block)
{
    return (void **)block;
}

static void **prev_free_slot(void *block)
{
    return (void **)((char *)block + WORD_SIZE);
}
```

이 함수들이 처음엔 왜 필요한지 불명확했다. 나중에 보니 free block 내부의 payload 공간을 next/prev pointer 저장소로 재활용하는 구조였다. "할당 해제된 메모리 공간을 리스트 관리에 쓴다"는 발상이 처음엔 약간 이상하게 느껴졌다.

## 2026-03-11
### Session 1
문서 정리. 이 프로젝트의 두 가지 핵심 학습을 정리했다.

첫 번째: block layout invariant. header, payload, footer, padding 순서와 각 위치 계산식. 이걸 한 번 직접 그리고 나면 후에 `next_block`, `previous_block` 공식이 자연스럽게 나온다.

두 번째: coalesce와 free list의 상호작용. 단순히 "인접한 free block을 합친다"가 아니라, "합치기 전에 인접 block을 free list에서 먼저 빼야 한다"는 순서. 이 순서를 틀리면 free list에 zombie block이 생긴다.
