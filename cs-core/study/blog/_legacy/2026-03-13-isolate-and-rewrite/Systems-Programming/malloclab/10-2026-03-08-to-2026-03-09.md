# Malloc Lab 개발 타임라인 (2026-03-08 ~ 2026-03-09)

## 2026-03-08
### Session 1
처음 시작은 naive한 판단에서 비롯됐다. "block을 나열하고, free하면 free list에 넣고, malloc하면 first-fit으로 찾으면 된다." 이렇게 생각했는데, 실제로 코드를 짜기 시작하자마자 header/footer 접근 순서가 헷갈렸다.

`header_of(block)`은 `block - WORD_SIZE`인데, `footer_of(block)`은 `block + block_size(header) - 2*WORD_SIZE`다. 이 식을 처음에 손으로 쓸 때 `-DOUBLE_WORD`인지 `-WORD_SIZE`인지 확신이 없어서 직접 그림을 그렸다. block의 layout을 한 번 그리고 나서야 footer가 "다음 block 시작 - WORD_SIZE" 위치에 있다는 게 잡혔다.

```c
static void *footer_of(void *block)
{
    return (char *)block + block_size(header_of(block)) - DOUBLE_WORD;
}
```

`coalesce`를 처음 짰을 때는 네 가지 경우를 모두 다른 방식으로 처리했는데, 결국 공통 패턴으로 정리됐다. free하려는 block의 header와 footer를 먼저 갱신하고, 합칠 게 있으면 기존 free list에서 이웃을 remove한 뒤 합쳐진 block을 다시 add한다.

```c
static void *coalesce(void *block)
{
    int prev_alloc = is_allocated(footer_of(previous_block(block)));
    int next_alloc = is_allocated(header_of(next_block(block)));
    size_t size = block_size(header_of(block));
    ...
    add_to_free_list(block);
    return block;
}
```

### Session 2
`mm_realloc` 때문에 상당한 시간을 썼다. 처음 구현은 단순했다: 새 영역 `malloc` → 내용 복사 → 기존 영역 `free`. 동작은 했는데, 기존 block와 다음 block이 합쳐진 공간이 충분할 때도 무조건 새 영역을 잡으니까 fragmentation이 늘었다.

이슈: 확장 최적화를 넣지 않으면 trace에서 utilization이 계속 낮았다.

결국 세 경우를 따로 처리했다:
1. 새 크기가 현재 block에 맞으면 그냥 반환
2. 현재 block + 다음 free block이 충분하면 제자리에서 확장
3. 그 외: malloc + memcpy + free
