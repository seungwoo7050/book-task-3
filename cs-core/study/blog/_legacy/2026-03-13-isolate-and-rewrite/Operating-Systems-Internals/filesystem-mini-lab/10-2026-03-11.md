# Filesystem Mini-Lab 개발 타임라인 (2026-03-11)

## 2026-03-11
### Session 1
이 프로젝트의 핵심은 "crash injection을 어떻게 구현하는가"였다. 실제 OS에서 crash를 시뮬레이션하려면 프로세스를 죽여야 하지만, Python 테스트에서는 그게 불가능하다. 대신 "crash_stage" 파라미터를 넘겨서 특정 시점에 `SimulatedCrash`를 발생시키는 방식을 택했다.

```python
def create(self, name: str, crash_stage: str | None = None) -> None:
    image = self._load()
    if name in image["root"]:
        raise FileExistsError(name)
    txid = self._begin_transaction(image, "create", {"name": name})
    self._maybe_crash(crash_stage, "after_prepare")
    self._commit_transaction(image, txid)
    self._maybe_crash(crash_stage, "after_commit")
    self._apply_and_finalize(image, txid)
```

세 단계: prepare → commit → apply. crash_stage가 각 단계에서 `SimulatedCrash`를 던지면 image가 해당 시점의 상태로 고정된다. 이렇게 하면 "prepare 후 crash", "commit 후 crash", "apply 후 crash" 세 시나리오를 테스트로 재현할 수 있다.

처음에는 이 구조를 "write-ahead logging"과 연결짓지 못했다. 나중에 보니 `journal`에 변경 내역을 먼저 기록(prepare) → journal이 완전히 쓰였음을 마킹(commit) → 실제 데이터에 반영(apply)하는 흐름이 WAL의 기본 패턴이었다.

### Session 2
crash 후 recovery를 구현했다.

```python
def recover(self) -> int:
    image = self._load()
    recovered = 0
    for entry in image["journal"]:
        if entry.get("status") == "committed" and not entry.get("applied"):
            self._replay(image, entry)
            entry["applied"] = True
            recovered += 1
    self._save(image)
    return recovered
```

"committed but not applied" 상태의 journal entry를 찾아서 replay한다. "committed이면 반드시 적용해야 한다"는 게 WAL의 핵심 불변 조건이다. prepared이지만 committed되지 않은 건 버린다.

이 recovery 로직을 처음 짰을 때 "apply 후 crash" 시나리오에서 이미 적용된 entry를 다시 replay해서 중복 적용이 됐다. `applied` 플래그를 추가해서 해결했다.

### Session 3
CLI를 통한 전체 흐름 테스트.

```bash
$ cd python
$ pip install -e .
$ pytest tests/ -v
$ python -m os_mini_fs demo.img mkfs 8 16
$ python -m os_mini_fs demo.img create hello.txt
$ python -m os_mini_fs demo.img write hello.txt "world"
$ python -m os_mini_fs demo.img describe
```

create → write → describe → crash simulation → recover 전체 사이클이 동작했다.

이 프로젝트에서 체감한 건, "journal에 먼저 쓰고 나중에 apply하면 crash-safe"가 개념으로는 단순해 보이지만, 실제로 "committed vs applied" 두 상태를 정확히 구분해서 관리하는 게 생각보다 섬세하다는 것이었다.
