# Attack Lab — 스택을 공격하고, 방어를 이해하다

## "입력만으로 프로그램의 실행 흐름을 바꿀 수 있다"

Attack Lab은 CS:APP 과제 중 가장 위험하게 느껴지는 랩이다. 과제의 목표가 명확하다 — **취약한 프로그램에 악의적인 입력을 만들어서 원하는 함수를 호출하라.**

공식 랩에서는 `ctarget`(코드 인젝션용)과 `rtarget`(ROP용) 두 개의 바이너리가 주어진다. 둘 다 `gets()` 함수를 사용해서 버퍼 크기를 넘는 입력을 받아들이며, 이 취약점을 이용해 리턴 주소를 덮어쓰는 것이 모든 페이즈의 출발점이다.

## 다섯 개 페이즈가 가르치는 것

페이즈 1~3은 **코드 인젝션(code injection)**을 다룬다. 스택이 실행 가능하고 주소가 고정된 환경(`ctarget`)에서:

1. **Phase 1**: 리턴 주소를 `touch1`로 덮어쓴다. 가장 단순한 형태의 스택 오버플로우.
2. **Phase 2**: 스택에 주입한 기계어 코드가 실행되도록 만든다. `cookie` 값을 `%rdi`에 세팅하고 `touch2`를 호출하는 셸코드를 작성한다.
3. **Phase 3**: 셸코드가 쿠키 문자열의 주소를 인자로 넘기도록 만든다. 문자열이 스택 어디에 놓이는지 직접 계산해야 한다.

페이즈 4~5는 **ROP(Return-Oriented Programming)**를 다룬다. 스택이 실행 불가능하고 주소가 매번 바뀌는 환경(`rtarget`)에서:

4. **Phase 4**: 기존 코드 조각(gadget)을 체이닝해서 `touch2(cookie)`를 호출한다.
5. **Phase 5**: 런타임에 변하는 스택 주소를 기준으로 상대 오프셋을 계산해서 쿠키 문자열의 주소를 만든다.

이 진행은 의도적이다. 1→3에서 "스택이 실행 가능하면 이렇게 공격한다"를 체험하고, 4→5에서 "그래서 W^X와 ASLR이 왜 필요한지"를 체감하게 만든다.

## 선행 지식

- **x86-64 스택 레이아웃**: 함수 호출 시 리턴 주소가 스택에 push되는 구조
- **호출 규약**: 첫 번째 인자 → `%rdi`, 리턴 값 → `%rax`
- **리틀 엔디안 바이트 순서**: 주소 `0x4017c0`은 메모리에 `c0 17 40 00 00 00 00 00`으로 저장
- **gdb 사용**: Bomb Lab에서 확립한 디스어셈블리/레지스터 검사 워크플로우
- **기계어 인코딩 기초**: `mov`, `push`, `ret` 명령어의 바이트 코드

## 왜 companion 검증기를 만들었는가

공식 `ctarget`/`rtarget` 바이너리는 CMU 교육 자산이다. 레포에 포함할 수 없고, 특정 인스턴스의 익스플로잇을 공개하는 것도 적절하지 않다.

그래서 각 페이즈의 **페이로드 구조를 검증하는 companion 검증기**를 C와 C++로 구현했다. 이 검증기는 "hex 페이로드가 올바른 레이아웃을 가지고 있는가"를 확인한다 — 리턴 주소 위치, 셸코드 프리픽스, 가젯 체인 순서, 쿠키 문자열 배치 등.

실제로 코드를 실행하지는 않으므로, 공식 바이너리 없이도 페이로드 설계 능력을 검증할 수 있다.

## 작업 환경

- **호스트**: macOS (Apple Silicon)
- **검증 환경**: Docker (linux/amd64) — `ctarget`/`rtarget`은 Linux ELF
- **hex2raw**: 공식 핸드아웃에 포함된 hex-to-binary 변환 도구
- **공식 복원 및 검증**: `make restore-official && make verify-official`
