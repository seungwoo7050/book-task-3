# Attack Lab — 디버깅 이야기: 바이트 하나의 실수

## 리틀 엔디안 바이트 순서 실수

Phase 1에서 가장 먼저 마주친 실수는 리틀 엔디안이었다. `touch1`의 주소 `0x4017c0`을 페이로드에 쓸 때, 처음에는 `00 00 00 00 00 40 17 c0`(빅 엔디안)으로 적었다. 당연히 실패.

x86-64는 리틀 엔디안이므로 `c0 17 40 00 00 00 00 00`으로 써야 한다. 이건 실수한 직후에 바로 잡았지만, 이후 모든 페이즈에서 주소를 쓸 때마다 "리틀 엔디안인가?"를 확인하는 습관이 생겼다.

## Phase 2: 셸코드 바이트 인코딩 오류

Phase 2의 셸코드를 손으로 인코딩할 때, `mov $0x1a2b3c4d, %rdi`의 기계어를 처음에 `48 c7 c7 1a 2b 3c 4d`로 적었다. 하지만 x86-64에서 즉시값도 리틀 엔디안 순서로 인코딩되므로, 올바른 바이트는 `48 c7 c7 4d 3c 2b 1a`이다.

이 실수를 디버깅할 때 gdb에서 셸코드 바이트를 메모리에서 디스어셈블해보는 방법이 유용했다:

```
(gdb) x/3i $rsp
```

이렇게 하면 주입한 바이트가 의도한 명령어로 해석되는지 바로 확인할 수 있다.

## Phase 3: 문자열이 덮어쓰여지는 문제

Phase 3에서 처음에는 쿠키 문자열을 셸코드 바로 뒤(오프셋 13)에 배치했다. 로컬 테스트에서는 동작했지만, 의심이 가서 `touch3` 진입 후의 스택 상태를 gdb로 확인했다.

확인 결과, `touch3`이 호출하는 `hexmatch` 함수가 스택을 적극적으로 사용해서, 버퍼 영역의 바이트를 덮어쓸 수 있는 구조였다. 문자열을 리턴 주소 슬롯 뒤(오프셋 48)로 옮기자 문제가 해결되었다.

이 경험에서 배운 건, 페이로드 내 문자열의 배치는 "현재 함수"뿐 아니라 "호출될 함수"의 스택 사용량까지 고려해야 한다는 것이다.

## Phase 4: gadget farm에서 원하는 바이트 찾기

Phase 4의 ROP 체인을 만들려면 `rtarget`의 gadget farm에서 특정 명령어 시퀀스를 찾아야 한다. 문제는 gadget이 "함수 경계에서 시작하는 정렬된 코드"가 아니라, **기존 명령어 바이트 스트림 안에서 우연히 형성되는 시퀀스**라는 것이다.

예를 들어 `pop %rax; ret`은 바이트로 `58 c3`인데, 이 2바이트 시퀀스가 다른 명령어의 중간이나 뒤에 나타날 수 있다. 디스어셈블리에서 이걸 찾으려면 기계어 바이트를 직접 스캔해야 한다:

```bash
# farm 영역 추출
cd problem
make disas-rt
make farm
# farm_gadgets.txt에서 원하는 바이트 시퀀스 검색
grep -n "58 c3\|48 89 c7 c3" farm_gadgets.txt
```

## Phase 5: %rsp 캡처 시점의 정확한 추적

Phase 5에서 가장 어려웠던 건 "mov %rsp, %rax를 실행하는 시점에 %rsp가 정확히 어디를 가리키는가"를 추적하는 것이었다. ROP 체인에서 각 `ret`이 실행될 때마다 `%rsp`가 8바이트씩 증가하므로, 체인의 위치에 따라 캡처되는 주소가 달라진다.

종이에 스택 레이아웃을 그리면서 각 gadget 실행 후의 `%rsp` 값을 추적했다. 최종적으로 문자열까지의 오프셋이 48바이트라는 것을 확정하고, 이 값을 `pop %rsi`로 가져오는 체인을 완성했다.

## Docker 환경에서의 공식 검증

공식 타겟의 검증 흐름:

```bash
cd problem
make restore-official   # target1 다운로드
make docker-image       # 이미 있으면 생략
make verify-official    # 5개 페이즈 순차 검증
```

`verify-official`은 각 페이즈를 개별적으로 Docker 안에서 실행한다:

```bash
hex2raw < data/phase1.txt | ctarget -q
hex2raw < data/phase2.txt | ctarget -q
hex2raw < data/phase3.txt | ctarget -q
hex2raw < data/phase4.txt | rtarget -q
hex2raw < data/phase5.txt | rtarget -q
```

`-q` 플래그는 서버 접속을 비활성화해서 오프라인 검증을 가능하게 한다.
