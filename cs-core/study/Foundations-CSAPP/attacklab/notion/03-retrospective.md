# Attack Lab — 회고: 공격을 통해 방어를 이해하다

## 이 과제가 바꿔놓은 관점

Attack Lab을 시작하기 전에는 버퍼 오버플로우가 "옛날 얘기"처럼 느껴졌다. 현대 시스템에는 스택 카나리, W^X, ASLR이 있으니까. 하지만 직접 공격 페이로드를 만들어보면, 이 방어 기술들이 **왜 필요한지**를 비로소 체감하게 된다.

Phase 1~3에서 코드 인젝션이 얼마나 쉬운지 경험하고 나면, Phase 4~5에서 "같은 공격이 왜 안 되는지"를 자연스럽게 이해한다. W^X가 있으면 주입한 코드가 실행되지 않고, ASLR이 있으면 절대 주소를 하드코딩할 수 없다. 방어 기술의 가치를 공격자의 관점에서 느끼는 것이 이 과제의 핵심 학습이다.

## 예상과 달랐던 것들

### Phase 5가 Phase 1보다 질적으로 다르다

Phase 1~3은 "정확한 바이트를 정확한 위치에 놓으면 된다"는 느낌이 강하다. 하지만 Phase 5는 "런타임에 결정되는 값을 기반으로 계산을 해야 한다"는 전혀 다른 차원의 문제다. 정적 분석만으로는 부족하고, 실행 흐름 중간의 레지스터 상태를 추적해야 한다.

### companion 검증기의 한계와 가치

companion 검증기는 페이로드의 구조적 정확성만 확인한다. 실제로 코드를 실행하지는 않으므로, "이 셸코드가 의도한 대로 동작하는가"를 완전히 보장하지는 못한다. 하지만 설계 의도를 코드로 문서화하는 가치가 있었고, 페이즈별 불변량(invariant)을 명시적으로 정의하는 과정에서 이해가 정리되었다.

### Bomb Lab의 인프라가 다시 한번 회수되었다

Docker 이미지, 복원 스크립트, Makefile 패턴 — Data Lab과 Bomb Lab에서 만들어둔 인프라를 Attack Lab에서도 거의 그대로 재사용했다. 과제마다 환경을 새로 구축할 필요가 없다는 것은 큰 효율 향상이었다.

## 다음에 비슷한 과제를 한다면

- **스택 레이아웃을 종이에 먼저 그린다.** 모든 페이즈에서 "지금 스택이 어떤 상태인가"를 시각화하는 것이 가장 효과적이었다.
- **리틀 엔디안 변환을 도구로 자동화한다.** 수동 변환은 실수의 원천이다.
- **gadget farm 분석을 스크립트로 한다.** 바이트 패턴 매칭을 grep으로 했는데, 전용 스크립트를 만드는 게 더 빠르다.

## 이 과제가 커리큘럼에서 갖는 위치

Attack Lab은 Bomb Lab(어셈블리 독해) 이후에 배치되어, 어셈블리를 "읽는" 것에서 "쓰는" 것으로 전환시킨다. 여기서 확립한 스택 레이아웃과 보안 개념의 이해는 이후 Architecture Lab(프로세서 설계)과 Shell Lab(프로세스 제어)에서 시스템 레벨 사고의 기반이 된다.
