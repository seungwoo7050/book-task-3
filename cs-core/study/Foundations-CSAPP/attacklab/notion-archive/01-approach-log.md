# Attack Lab — 풀이 여정: 스택 레이아웃을 이해하기까지

## 출발점: "리턴 주소를 덮어쓴다"의 의미

컴퓨터 보안 수업에서 버퍼 오버플로우라는 말은 들어봤지만, 실제로 해본 건 처음이었다. Phase 1부터 시작하면 생각보다 단순하다 — `getbuf` 함수의 스택 프레임에서 버퍼(40바이트)를 넘겨 리턴 주소를 원하는 값으로 덮으면 된다.

하지만 "40바이트를 채우고 그 다음 8바이트에 주소를 넣는다"는 문장을 실제 hex 바이트로 변환하는 과정에서, 리틀 엔디안의 체감이 달라진다. `touch1`의 주소 `0x4017c0`을 페이로드에 넣으려면 `c0 17 40 00 00 00 00 00`으로 써야 한다. 바이트 순서가 뒤집힌다는 걸 책에서 읽은 것과, 직접 hex 파일에 써넣는 것은 다른 경험이다.

## Phase 2: 셸코드를 직접 작성하다

Phase 2부터가 진짜 Attack Lab이다. 스택에 주입한 바이트가 기계어 코드로 **실제로 실행**되어야 한다. 이 코드가 해야 하는 일은:

1. `cookie` 값(0x1a2b3c4d)을 `%rdi` 레지스터에 넣는다
2. `touch2`의 주소를 스택에 push한다
3. `ret`으로 `touch2`에 진입한다

이걸 어셈블리로 쓰면:

```asm
mov $0x1a2b3c4d, %rdi
pushq $0x4017ec
ret
```

문제는 이 어셈블리를 기계어 바이트(`48 c7 c7 4d 3c 2b 1a 68 ec 17 40 00 c3`)로 변환하고, 스택 버퍼의 시작 주소(0x5561dc78)를 리턴 주소에 넣어야 한다는 것이다. "코드가 어디에 있는지"와 "실행이 어디로 점프하는지"를 동시에 관리해야 한다.

`gdb`에서 `getbuf`의 스택 프레임을 확인해서 버퍼 시작 주소를 알아내고, 그 주소를 리턴 주소 슬롯에 넣었다. 셸코드가 버퍼 앞쪽에, 리턴 주소가 버퍼 바로 뒤에 놓이는 구조가 완성된다.

## Phase 3: 문자열 배치 위치 계산

Phase 3는 Phase 2의 확장인데, `touch3`에 넘길 인자가 정수가 아니라 **문자열의 주소**라는 점이 다르다. 쿠키 문자열 `"1a2b3c4d"`를 스택 어딘가에 배치하고, 그 주소를 `%rdi`에 넣어야 한다.

여기서 처음 고민한 건 "문자열을 어디에 놓을 것인가"였다. 함수 호출이 진행되면 스택 포인터가 이동하면서 기존 데이터를 덮어쓸 위험이 있다. 그래서 문자열을 버퍼 안쪽이 아니라 **리턴 주소 슬롯 뒤쪽**에 배치했다. 40바이트 패딩 + 8바이트 리턴 주소 뒤인 오프셋 48에 문자열을 놓으면, `touch3` 함수 진입 후에도 문자열이 덮어쓰이지 않는다.

이 판단은 gdb에서 `touch3` 내부의 스택 사용량을 확인한 뒤에 확정했다.

## Phase 4로의 전환: 방어 기술의 벽

Phase 4부터는 `rtarget`을 공격한다. `ctarget`과 달리:
- 스택이 실행 불가능(W^X)
- 스택 주소가 실행마다 변동(ASLR 시뮬레이션)

셸코드를 주입해도 실행되지 않으므로, 기존 코드 조각(gadget)을 리턴 주소 체인으로 엮어야 한다. 이것이 **Return-Oriented Programming (ROP)**.

Phase 4의 ROP 체인 구조:

```
[40 bytes padding] | [pop %rax gadget addr] | [cookie value] | [mov %rax,%rdi gadget addr] | [touch2 addr]
```

핵심은 `pop %rax`가 스택에서 다음 8바이트를 `%rax`로 가져오고, `mov %rax, %rdi`가 이를 첫 번째 인자 레지스터로 옮긴다는 것이다. 각 gadget이 `ret`으로 끝나므로, 스택의 다음 주소로 자동으로 넘어간다.

이 체인을 조립하려면 `rtarget`의 "gadget farm" — 의도적으로 심어둔 유용한 바이트 시퀀스 — 에서 원하는 gadget의 주소를 찾아야 한다.

## Phase 5: 런타임 상대 주소 계산

Phase 5는 Attack Lab의 최종 보스다. `touch3`을 ROP로 호출해야 하는데, 문자열 주소를 하드코딩할 수 없다(ASLR 때문). 그래서 **현재 스택 포인터를 기준으로 오프셋을 계산**해야 한다.

ROP 체인의 흐름:
1. `mov %rsp, %rax` — 현재 스택 포인터를 캡처
2. `mov %rax, %rdi` — 캡처한 주소를 보존
3. `pop %rsi` — 오프셋 값을 가져옴
4. `lea (%rdi,%rsi), %rdi` — base + offset = 문자열 주소 계산  
5. `mov %rax, %rdi` — 최종 주소를 인자에 세팅
6. `ret` → `touch3`

이 체인을 조립하려면 각 gadget 실행 시점의 `%rsp` 값을 추적하면서, 문자열이 놓인 위치까지의 오프셋을 정확히 계산해야 한다.

Phase 5를 풀면서 "주소 공간 레이아웃 무작위화(ASLR)가 왜 효과적인 방어인지"를 체감했다. 절대 주소를 쓸 수 없으니, 공격자는 상대 주소 계산이라는 훨씬 복잡한 경로를 거쳐야 한다.

## companion 검증기 구현

공식 바이너리 없이도 페이로드 구조를 검증할 수 있도록, C/C++로 companion 검증기를 만들었다. 각 페이즈의 검증 함수는 hex 페이로드를 받아서:

- 정확한 길이인지 확인
- 리턴 주소 슬롯에 올바른 주소가 있는지 확인
- 셸코드 프리픽스가 맞는지 확인
- ROP 가젯 체인의 순서가 올바른지 확인
- 쿠키 문자열이 올바른 위치에 있는지 확인

이 검증기는 공식 바이너리를 대체하는 것이 아니라, 페이로드 설계의 구조적 정확성을 확인하는 도구다.
