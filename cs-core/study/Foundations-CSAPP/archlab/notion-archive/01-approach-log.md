# Architecture Lab — 접근 기록

## Part A: ISA 수준에서 C 로직 재현하기

### 반복 합산 — sum_list

Part A의 첫 번째 과제는 연결 리스트를 따라가며 값을 합산하는 프로그램을 Y86-64 어셈블리로 작성하는 것이다.

로직 자체는 단순하다. `while (head != NULL)`을 `andq %rdi, %rdi; je done`으로 변환하고, `head->value`를 `mrmovq (%rdi), %r10`으로 로드하고, `head->next`를 `mrmovq 8(%rdi), %rdi`로 따라가면 된다. 결과를 `%rax`에 누적하고 마지막에 `ret`하면 끝이다.

중요한 발견은 Y86-64에는 `cmpq`가 없다는 점이다. NULL 체크를 `andq %rdi, %rdi`로 처리해야 한다. x86-64에서 당연하게 쓰던 `test` 같은 명령어가 없으니, 조건 코드를 설정하는 방법부터 다시 생각해야 했다.

### 재귀 합산 — rsum_list

같은 로직을 재귀로 구현하는데, 여기서 핵심은 caller-save와 callee-save의 구분이다. 재귀 호출 전에 현재 노드의 값을 `%rbx`(callee-saved)에 미리 복사해 두거나, 스택에 push해 두어야 한번 돌아온 뒤에도 값이 살아 있다. 

`pushq %rbx; call rsum_list; popq %rbx` 패턴으로 해결했다. base case는 `irmovq $0, %rax; ret`로 깔끔하게 처리된다.

### 블록 복사 — copy_block

세 번째 과제는 소스 배열을 목적지로 복사하면서 XOR 체크섬을 계산하는 것이다. 여기서 Y86-64의 제약이 다시 드러난다: 즉시값 연산이 제한적이므로, 포인터 증분(+8)과 카운터 감소(-1)를 각각 별도 레지스터에 미리 로드해 놓아야 한다.

`irmovq $8, %r8; irmovq $1, %r9`로 상수를 준비하고, 루프 안에서 `addq %r8, %rdi`와 `subq %r9, %rdx`로 갱신하는 방식이다. XOR 체크섬은 `xorq %r10, %rax`로 한 줄이면 된다.

세 프로그램 모두 공식 시뮬레이터에서 `%rax`에 `0xcba`가 남는 것을 확인했다.

## Part B: iaddq 명령어 추가

### 제어 신호 추론

`iaddq V, rB`를 SEQ 프로세서에 추가하라는 과제다. 처음에는 "한 줄 추가하면 되지 않을까"라고 생각했는데, 실제로는 HCL 파일에서 **8군데**를 수정해야 했다:

1. `instr_valid` — iaddq가 유효한 명령어임을 선언
2. `need_regids` — 레지스터 바이트가 필요함을 선언
3. `need_valC` — 즉시값이 필요함을 선언
4. `srcB` — decode 단계에서 rB를 읽어야 함
5. `dstE` — write-back 단계에서 rB에 써야 함
6. `aluA` — execute 단계에서 valC를 ALU 입력으로 써야 함
7. `aluB` — execute 단계에서 valB를 ALU 입력으로 써야 함
8. `set_cc` — 조건 코드를 갱신해야 함

이 과정을 거치면 "명령어 하나가 프로세서의 모든 스테이지를 관통한다"는 사실을 체감하게 된다. 패치 스크립트(`apply_hcl_patches.py`)로 SEQ HCL과 PIPE HCL 두 파일 모두에 동일한 논리를 적용하되, PIPE 버전은 파이프라인 레지스터 접두사(`f_`, `D_`, `E_`)가 붙는 차이만 있다.

공식 검증은 `ptest`로 회귀 테스트를 돌려서 모든 명령어 조합이 여전히 정상 동작하는지 확인한다. iaddq 하나를 추가했을 뿐인데 기존 명령어가 깨질 수 있다는 점이 이 과제의 교훈이다.

### 컴패니언 모델

공식 HCL은 git-tracked하지 않으므로, 같은 추론을 C 코드로 표현했다. `arch_seq_iaddq()` 함수가 `next_pc = pc + 10`, `valE = valB + valC`, 그리고 ZF/SF/OF 계산을 모두 수행한다. 특히 오버플로 판정에서 `(a ^ result) & (b ^ result) < 0` 패턴이 핵심이다.

## Part C: ncopy 파이프라인 최적화

### 기본 구조 이해

`ncopy`는 소스 배열을 목적지로 복사하면서 양수 원소의 개수를 세는 함수다. 기본 구현은 한 번에 하나의 원소를 처리한다 — 로드, 저장, 양수 판정, 카운트 갱신, 포인터 증가, 루프. 이것이 원소당 약 9사이클이다.

### 8-way 언롤링과 cmovg

최적화의 핵심 전략은 루프 언롤링이다. 8개 원소를 한 번에 로드하고, 한 번에 저장하고, 양수 판정을 `cmovg`로 브랜치 없이 처리한다.

```
mrmovq 0(%rdi), %r8
mrmovq 8(%rdi), %r9
...
rmmovq %r8, 0(%rsi)
rmmovq %r9, 8(%rsi)
...
xorq %rcx, %rcx
andq %r8, %r8
cmovg %rbp, %rcx
addq %rcx, %rax
```

이 패턴의 장점은 두 가지다. 첫째, 루프 오버헤드(포인터 갱신, 조건 점프)가 8개 원소에 한 번만 발생한다. 둘째, `cmovg`는 조건 이동이라 분기 예측 실패 페널티가 없다. 파이프라인 프로세서에서 분기 예측 실패는 2~3사이클의 버블을 만드므로, 이것을 제거하는 효과가 크다.

### 꼬리 처리 — 폴스루 디스패치

8개씩 처리한 뒤 남는 원소(0~7개)는 별도의 테일 디스패치로 처리한다. `iaddq $-1, %rdx; je Tail1`을 반복하는 방식으로 남은 개수를 판정하고, Tail7에서 Tail1까지 폴스루로 연결해서 정확한 수만큼만 처리한다.

이 폴스루 패턴은 Duff's device와 유사한 아이디어다. 점프 테이블 없이, 그리고 추가 루프 없이, 남은 원소를 정확하게 소진한다.

### 결과

공식 벤치마크에서 Average CPE 9.16, Score 26.8/60.0을 기록했다. 만점에는 한참 못 미치지만, 기본 구현 대비 확실한 개선이고, "같은 결과를 내면서도 파이프라인 사이클 수를 줄인다"는 Part C의 핵심 교훈은 달성했다.

컴패니언 모델에서는 4-way 언롤링 기준으로 pseudo-CPE를 계산하고, `optimized_cpe < baseline_cpe` 부등식이 성립하는 것을 테스트로 검증한다.

## 프로젝트 구조 결정

세 파트가 워낙 이질적이라서 프로젝트를 세 개로 분리할지 고민했다. 결국 하나로 유지하되, `y86/`에 공식 핸드인 파일을, `c/`와 `cpp/`에 컴패니언 모델을 두는 구조로 결정했다. 이유는 간단하다: 세 파트 모두 "하나의 프로세서 아키텍처"라는 동일한 주제 아래 있고, Part B의 `iaddq` 추가가 Part C의 최적화에 직접 연결되기 때문이다.
