# Architecture Lab — 디버그 기록

## Y86-64에 cmpq가 없다

### 증상

Part A 어셈블리를 처음 작성할 때, x86-64 습관대로 `cmpq $0, %rdi`를 써서 NULL 체크를 하려 했다. `yas`가 즉시 에러를 뱉었다.

### 원인

Y86-64 ISA에는 `cmpq`가 존재하지 않는다. 비교 연산은 `subq`나 `andq`로 조건 코드를 설정하는 방식으로 우회해야 한다. NULL 포인터 체크처럼 0과 비교할 때는 `andq %rdi, %rdi`가 가장 자연스럽다. 값을 변경하지 않으면서 ZF만 설정하기 때문이다.

### 수정

모든 NULL 체크를 `andq %reg, %reg; je target` 패턴으로 통일했다. 이것은 x86-64의 `test %reg, %reg`와 동일한 효과다.

### 교훈

축소된 ISA에서 프로그래밍한다는 것은 "내가 당연하게 쓰던 명령어가 없을 수 있다"를 전제하고 시작하는 것이다. Y86-64의 존재 이유가 바로 이 체감에 있다.

## Part B: iaddq 패치 누락으로 ptest 실패

### 증상

HCL에 `iaddq`를 추가하고 `ptest`를 돌렸더니, 일부 테스트가 실패했다. 특히 `iaddq` 뒤에 조건 점프가 오는 패턴에서 결과가 틀렸다.

### 원인

`set_cc` 항목을 수정하지 않았다. `iaddq`는 산술 연산이므로 `OPq`와 마찬가지로 조건 코드를 갱신해야 하는데, `set_cc = icode in { IOPQ }`에 `IIADDQ`를 추가하지 않은 것이다. 결과적으로 `iaddq` 실행 후 조건 코드가 갱신되지 않아, 바로 뒤의 `je`, `jne` 등이 이전 명령어의 조건 코드를 기반으로 분기했다.

### 수정

```
set_cc = icode in { IOPQ, IIADDQ };
```

SEQ 버전에서 이것을 놓쳤고, PIPE 버전에서는 `==` 연산자를 `in` 집합 표기로 바꿔야 하는 추가 차이가 있었다:
```
# SEQ: bool set_cc = icode in { IOPQ, IIADDQ };
# PIPE: bool set_cc = E_icode in { IOPQ, IIADDQ } && ...
```

### 교훈

"명령어 하나를 추가할 때 건드려야 하는 곳"의 목록을 체크리스트로 만들어두지 않으면 하나쯤은 반드시 빠진다. 실제로 총 8군데를 수정해야 했고, 7개만 수정하면 1개 누락 때문에 전체가 실패한다.

## PIPE HCL과 SEQ HCL의 미묘한 차이

### 증상

SEQ 버전의 패치를 그대로 복사해서 PIPE HCL에 적용했더니 `make` 단계에서 컴파일 에러가 발생했다.

### 원인

PIPE HCL은 파이프라인 레지스터 접두사를 사용한다. SEQ에서 `icode`인 것이 PIPE에서는 `f_icode`(fetch), `D_icode`(decode), `E_icode`(execute) 등으로 스테이지별로 구분된다. `srcB`는 `d_srcB`가 되고, `dstE`는 `d_dstE`가 된다. 단순 텍스트 치환으로는 안 된다.

### 수정

`apply_hcl_patches.py`에서 `patch_seq()`와 `patch_pipe()`를 완전히 별도 함수로 분리했다. 각각의 치환 쌍이 해당 HCL 파일의 정확한 문맥에 맞도록 작성했다.

### 교훈

파이프라인 프로세서에서는 같은 논리적 신호라도 시간적 위치(어느 스테이지에 있느냐)에 따라 다른 이름을 갖는다. 이것은 코드 레벨의 문제가 아니라 아키텍처적 사고의 문제다.

## 시뮬레이터 빌드: -fcommon 플래그

### 증상

Docker 컨테이너 안에서 공식 시뮬레이터를 빌드했더니, GCC 10+에서 multiple definition 에러가 발생했다.

### 원인

공식 핸드아웃의 C 코드가 여러 파일에서 전역 변수를 tentative definition으로 선언한다. GCC 9 이하에서는 `-fcommon`이 기본이라 문제가 없었지만, GCC 10부터는 `-fno-common`이 기본으로 바뀌면서 링크 에러가 발생한다.

### 수정

Makefile의 `verify-*` 타겟에서 시뮬레이터 빌드 시 `CFLAGS="-Wall -O1 -g -fcommon" LCFLAGS="-O1 -fcommon"`을 명시적으로 전달했다.

### 교훈

오래된 교육용 코드를 최신 툴체인에서 빌드할 때는 컴파일러 기본값 변경을 항상 확인해야 한다. `-fcommon`은 그 전형적인 예시다.

## ncopy 최적화: cmovg vs 조건 분기

### 증상

초기 ncopy 최적화에서 양수 판정을 `andq %r8, %r8; jle skip; iaddq $1, %rax; skip:` 패턴으로 작성했다. 동작은 정확했지만 CPE가 크게 줄지 않았다.

### 원인

조건 분기(`jle`)는 파이프라인에서 분기 예측 실패 시 2~3사이클의 버블을 만든다. 데이터에 양수와 비양수가 섞여 있으면 예측률이 떨어진다.

### 수정

```
xorq %rcx, %rcx
andq %r8, %r8
cmovg %rbp, %rcx    # %rbp에는 상수 1이 들어있다
addq %rcx, %rax
```

`cmovg`는 조건 이동이라 파이프라인을 깨지 않는다. 조건이 거짓이면 이동이 일어나지 않을 뿐이고, `addq %rcx, %rax`에서 0을 더하게 되므로 결과는 동일하다. 분기 예측 실패 페널티를 완전히 제거한다.

### 교훈

파이프라인 최적화의 핵심은 "정확한 결과를 내면서 분기를 제거하는 것"이다. `cmovg`는 그 도구이고, 상수 레지스터 + 0-초기화 레지스터의 조합이 그 패턴이다.
