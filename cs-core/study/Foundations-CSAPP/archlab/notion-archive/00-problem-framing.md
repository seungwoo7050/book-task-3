# Architecture Lab — 문제 프레이밍

## 이 랩은 무엇을 묻는가

Architecture Lab은 CS:APP 랩 시리즈 중에서 가장 이질적인 과제다. 하나의 랩 안에 세 가지 완전히 다른 종류의 작업이 들어 있다.

- **Part A**는 Y86-64 어셈블리로 세 개의 프로그램을 작성하라고 요구한다. 연결 리스트 반복 합산, 재귀 합산, 블록 복사-XOR. C로 이미 로직을 알고 있는 것을 ISA 수준에서 다시 구현하는 일이다.
- **Part B**는 SEQ 프로세서의 HCL 제어 로직을 수정해서 `iaddq` 명령어를 추가하라고 요구한다. "하나의 명령어를 추가한다"는 말이 실제로는 fetch, decode, execute, memory, write-back 모든 스테이지의 제어 신호를 건드려야 한다는 것을 보여주는 과제다.
- **Part C**는 PIPE 프로세서에서 `ncopy` 함수의 파이프라인 성능을 최적화하라고 요구한다. 정확성은 기본이고, 그 위에서 CPE(Cycles Per Element)를 줄이는 것이 목표다.

세 파트가 각각 다른 사고력을 요구하기 때문에, 이 랩 하나만으로도 프로세서 아키텍처의 핵심 계층 — ISA, 제어 로직, 파이프라인 — 을 체감할 수 있다.

## 왜 이 랩이 까다로운가

첫 번째 난관은 환경이다. Y86-64 시뮬레이터와 HCL 컴파일러는 CMU의 공식 핸드아웃에만 들어 있고, 이것이 macOS Apple Silicon에서는 직접 빌드되지 않는다. Docker로 linux/amd64 에뮬레이션을 설정하고, 핸드아웃 복원 스크립트를 돌리고, 시뮬레이터를 빌드해야 비로소 공식 검증 파이프라인이 작동한다.

두 번째 난관은 Part B와 Part C 사이의 사고 전환이다. Part B에서는 "이 명령어가 어느 제어 신호를 건드리는가"를 추론하는 논리적 완전성이 중요하고, Part C에서는 "이 코드가 파이프라인에서 몇 사이클에 실행되는가"를 추론하는 성능 감각이 중요하다. 둘 다 정확성이 전제되지만, 정확성 위에서 바라보는 관점이 완전히 다르다.

세 번째 난관은 검증 경로의 복잡성이다. Part A는 `yas`로 어셈블하고 `yis`로 시뮬레이션하면 되지만, Part B는 HCL을 컴파일해서 `ssim`을 빌드하고 `ptest`로 회귀 테스트를 돌려야 하며, Part C는 `psim`을 빌드하고 `correctness.pl`과 `benchmark.pl`을 모두 통과해야 한다. 각 파트의 검증 명령이 전부 다르다.

## 선행 지식

- Y86-64 ISA 기본: 레지스터 파일, 명령어 인코딩, 조건 코드(ZF, SF, OF)
- 프로세서 파이프라인 기본: fetch/decode/execute/memory/write-back 5단계
- HCL 문법: CS:APP 교재 4장의 제어 신호 표기법
- C 프로그래밍: 연결 리스트, 배열 복사, XOR 체크섬
- Docker 기본: Apple Silicon 환경에서 linux/amd64 이미지 실행

## 이 프로젝트의 전략

공식 HCL/시뮬레이터 파일은 핸드아웃 복원으로 로컬에서만 존재하고 git-tracked되지 않는다. 대신 세 가지 레이어로 분리했다:

1. `problem/` — 공식 핸드아웃을 로컬에 복원하고 공식 검증을 실행하는 인프라
2. `y86/` — 직접 작성한 Y86-64 어셈블리(Part A)와 `ncopy.ys`(Part C), HCL 패치 스크립트(Part B)
3. `c/`와 `cpp/` — 공식 시뮬레이터 없이도 같은 추론 부하를 유지하는 컴패니언 모델

이 구조 덕분에 공식 검증도 가능하고, 핵심 로직은 추적 가능한 C/C++ 코드로 남아 있다.
