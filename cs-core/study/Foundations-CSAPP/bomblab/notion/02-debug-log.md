# Bomb Lab — 디버깅 이야기: 폭탄이 터지기 전에 멈추는 법

## explode_bomb 브레이크포인트를 잊은 첫 실행

공식 봄을 Docker 안에서 처음 실행했을 때, 아무 생각 없이 테스트 문자열을 입력했다가 바로 "BOOM!!!"이 출력되었다. 공식 봄에는 폭발 횟수를 기록하는 기능이 있으므로, 무분별한 시도는 감점 리스크가 있다.

그 뒤로 정한 원칙: **모든 gdb 세션의 첫 명령은 `break explode_bomb`**이다. 이렇게 하면 잘못된 입력을 넣어도 폭발 함수 진입 직전에 멈추기 때문에, 레지스터와 스택 상태를 확인한 뒤 `quit`으로 빠져나올 수 있다.

```bash
gdb ./bomb
(gdb) break explode_bomb
(gdb) run answers.txt
```

이미 풀린 페이즈들의 정답을 파일에 저장해두고, 새 페이즈는 gdb 안에서 직접 입력하는 패턴이 확립되었다.

## Phase 4 재귀 추적: 종이가 필수

Phase 4의 `func4`를 gdb에서 step-by-step으로 따라가는 것은 비효율적이었다. 재귀가 깊어지면 어디가 현재 스택 프레임인지 혼란이 오고, 반환 값이 어떻게 조합되는지 추적하기 어렵다.

결국 디스어셈블리를 의사코드로 번역한 뒤, 종이에 전개 트리를 그렸다. `func4(target, 0, 14)`에서 중간값은 7이고, 타겟이 7보다 작으면 왼쪽(`func4(target, 0, 6)`), 크면 오른쪽(`func4(target, 8, 14)`)으로 내려간다. 각 경로의 반환 값이 0, 1, 2, 3... 순서로 인코딩되므로, 결과가 6이 되는 경로를 역산하면 타겟이 결정된다.

이 경험에서 배운 건, 재귀 기반 어셈블리는 디버거 단계 실행보다 의사코드 + 전개 트리가 훨씬 효과적이라는 것이다.

## Phase 5 니블 마스킹: 올바른 ASCII 문자 선택

Phase 5에서 테이블 인덱스는 역산할 수 있었지만, "그 인덱스를 하위 니블로 가지는 ASCII 문자"를 고르는 과정에서 오류가 있었다. 예를 들어 인덱스 0이 필요하면 `0x30`('0'), `0x40`('@'), `0x50`('P') 등이 모두 가능한데, 입력 문자열의 길이가 정확히 6이어야 한다는 제약을 놓쳐서 처음에 5글자만 넣었다.

미니 봄에서는 `strlen(input) != 6` 검사가 명시적이지만, 공식 봄에서는 `call string_length` 후에 `cmp $0x6, %eax` 형태로 나타난다. 어셈블리의 길이 검사를 놓치면 정답을 맞춰도 실패한다.

## Phase 6 연결 리스트: `7 - x` 변환을 놓치다

Phase 6에서 가장 헷갈렸던 부분은 `7 - x` 변환이다. 입력 값을 그대로 노드 인덱스로 쓰는 게 아니라, `7 - 입력값`으로 변환한 뒤에 노드를 선택한다. 이 변환을 놓치면 노드 값과 입력 순서의 관계가 뒤집혀서, 아무리 올바른 정렬 순서를 넣어도 실패한다.

미니 봄에서 노드 값이 `{817, 233, 901, 477, 692, 105}`이고 내림차순 정렬이 필요하므로, 노드 인덱스 순서는 `3, 1, 5, 4, 2, 6`이 되고, `7 - x` 역변환을 적용하면 최종 입력은 `4 6 2 3 5 1`이다.

## Docker 환경에서의 공식 봄 검증

공식 봄 바이너리는 Linux/amd64 ELF이므로 macOS에서 직접 실행할 수 없다. Docker를 통해 검증하는 패턴은 Data Lab에서 확립한 것과 동일하다:

```bash
cd problem
make restore-official    # CMU에서 공식 셀프스터디 봄 다운로드
make docker-image        # 이미 빌드되어 있으면 생략
make verify-official     # Docker 안에서 정답 파일로 봄 실행
```

공식 봄은 자체 검증 결과가 터미널에 출력되므로, btest 같은 별도 하네스가 필요하지 않다. 6개 페이즈 모두 "Phase N defused"가 출력되면 성공이다.
