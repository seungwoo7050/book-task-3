# Bomb Lab — 접근 과정: 어셈블리에서 의도를 읽어내기까지

## 첫 번째 벽: "디스어셈블리를 보는 것"과 "의미를 읽는 것"

CS:APP 3장을 읽고 나면, `mov`, `cmp`, `jne` 같은 명령어가 뭘 하는지는 알게 된다. 하지만 실제 바이너리의 디스어셈블리를 열면 전혀 다른 느낌이다. 수백 줄의 명령어가 쏟아져 나오고, 어디서부터 읽어야 할지 감이 안 온다.

그래서 정한 규칙이 있다: **페이즈 함수의 입구부터 콜 그래프를 따라가되, 한 번에 하나의 클레임(claim)만 검증한다.** "이 함수는 문자열 두 개를 비교한다"를 먼저 확인하고, 그 다음에 "비교 대상 문자열은 이것이다"를 확인하는 식이다. 한꺼번에 전체 그림을 그리려고 하면 오히려 길을 잃었다.

## 페이즈 1: 문자열 비교 — 가장 단순한 유형

Phase 1은 디스어셈블리에서 `strings_not_equal`이나 `string_length` 같은 함수 호출을 찾는 것으로 시작한다. 이 함수가 참조하는 메모리 주소에 어떤 문자열이 있는지 `gdb`에서 `x/s 주소`로 확인하면 정답이 나온다.

미니 봄에서는 `strcmp(input, "Assembly reveals intent.")`로 구현했다. 실제 공식 봄에서도 구조는 동일 — 고정 문자열과의 일치 검사.

이 페이즈는 "바이너리에서 데이터를 추출하는 기본 워크플로우"를 확립하는 역할이다. 이후 페이즈에서는 데이터 추출보다 제어 흐름 분석에 무게가 실린다.

## 페이즈 2: 루프 기반 수열 — 반복 관계 역추론

Phase 2에서는 `read_six_numbers` 같은 파싱 함수가 호출되고, 그 뒤에 루프가 나타난다. 어셈블리에서 루프를 인식하는 신호는 명확하다: 카운터 레지스터 초기화 → 조건 비교 → 점프 → 카운터 증가.

핵심은 루프 본체에서 "이전 값과 현재 값의 관계"를 읽어내는 것이다. 미니 봄의 경우 `values[i] == values[i-1] * 2`, 즉 첫 값이 1이고 각 값이 이전의 두 배인 수열 `1 2 4 8 16 32`이다.

실제 공식 봄에서도 이 패턴은 동일하다 — 읽어야 할 것은 어셈블리에 인코딩된 "반복 관계식"이다.

## 페이즈 3: 스위치/점프 테이블 — 다중 정답의 존재

Phase 3에서는 디스어셈블리에 점프 테이블(jump table)이 나타난다. `jmpq *주소(,%rax,8)` 같은 간접 점프가 그 신호다. 이건 C의 `switch` 문이 컴파일된 결과물이다.

이 페이즈의 특징은 **정답이 여러 개**라는 것이다. 각 `case`마다 다른 값이 매핑되어 있으므로, 유효한 `(인덱스, 값)` 쌍이 여럿 존재한다. 미니 봄에서는 8개의 케이스가 있고, `(1, 311)`, `(6, 128)` 등이 모두 유효한 정답이다.

이 시점에서 gdb의 `stepi`와 `info registers`가 핵심 도구가 된다. 점프 테이블의 각 분기가 어떤 값을 세팅하는지 하나씩 따라가야 한다.

## 페이즈 4: 재귀 함수의 반환 경로 — func4 패턴

Phase 4는 이 랩에서 가장 추상적인 사고를 요구한다. 어셈블리에서 재귀 호출이 보이고, 반환 값이 인코딩된 경로 정보를 담고 있다.

미니 봄의 `func4(target, 0, 14)`는 이진 탐색의 변형이다. 중간값을 계산하고, 타겟이 중간값보다 작으면 왼쪽, 크면 오른쪽으로 재귀한다. 반환 값은 `2 * func4(...)` 또는 `2 * func4(...) + 1` 형태로, 탐색 경로를 이진수로 인코딩한다.

페이즈의 조건은 `func4(target, 0, 14) == 6`이므로, 이 경로 인코딩을 만족하는 타겟 값을 역산해야 한다. 재귀를 종이에 펼쳐서 트리로 그리면 훨씬 빠르게 답이 보인다.

## 페이즈 5: 니블 마스킹과 룩업 테이블

Phase 5는 문자열 입력의 각 글자를 `& 0x0f`로 마스킹해서 룩업 테이블의 인덱스로 쓴다. 이 테이블에서 뽑아낸 글자들을 이어붙인 결과가 특정 문자열과 일치해야 한다.

미니 봄에서는 테이블이 `"tracebulkmdifnso"`이고, 결과가 `"traces"`여야 한다. 각 위치에서 목표 글자의 테이블 인덱스를 역산하고, 그 인덱스를 하위 니블로 가지는 ASCII 문자를 찾으면 된다.

어셈블리에서 이 패턴을 인식하는 핵심 신호는 `and $0xf, %eax` 다. 이 마스킹이 보이면 "입력을 인덱스로 변환하고 있다"고 판단할 수 있다.

## 페이즈 6: 연결 리스트 재정렬 — 가장 복잡한 구조

Phase 6는 연결 리스트 노드를 입력 순서에 따라 재배열한 뒤 내림차순인지 확인한다. 구현이 복잡한 이유는 중간에 `7 - x` 변환 단계가 껴 있기 때문이다.

파이프라인으로 보면:
1. 6개의 정수를 읽는다 (1~6, 중복 불가)
2. 각 값을 `7 - x`로 변환한다
3. 변환된 인덱스로 노드 배열에서 포인터를 뽑는다
4. 뽑은 포인터들을 순서대로 재연결한다
5. 재연결된 리스트가 내림차순인지 확인한다

어셈블리에서 이걸 읽으려면 "포인터 체이닝"을 인식해야 한다. `mov (%rax), %rax` 같은 패턴이 연결 리스트 순회의 신호이고, 데이터 섹션에서 노드의 값을 직접 읽어야 정답을 역산할 수 있다.

## 시크릿 페이즈: BST 경로 인코딩

시크릿 페이즈는 이진 탐색 트리 위에서 `fun7`이라는 재귀 함수가 반환하는 경로 값이 특정 수와 일치하는지 확인한다. 구조는 Phase 4의 `func4`와 비슷하지만, 탐색 대상이 배열이 아니라 트리의 노드 값이다.

미니 봄에서는 `fun7(&root, value) == 6`을 만족하는 값을 찾아야 한다. BST의 구조를 `gdb`에서 메모리 덤프로 재구성하면, 경로 인코딩을 역산해서 목표 노드를 찾을 수 있다.

## 미니 봄 구현: 동일한 패턴, 다른 값

공식 공개가 어려운 리버스 엔지니어링 결과 대신, 같은 페이즈 패밀리를 C/C++로 재구현한 companion 미니 봄을 만들었다. 각 페이즈의 핵심 구조는 공식 봄과 동일하되, 구체적인 값과 상수는 독자적으로 설계했다.

이 접근의 장점은 두 가지다:
1. 레포에 실행 가능하고 테스트 가능한 코드가 있다
2. 각 퍼즈 패밀리의 구조를 이해했다는 것을 코드로 증명할 수 있다
