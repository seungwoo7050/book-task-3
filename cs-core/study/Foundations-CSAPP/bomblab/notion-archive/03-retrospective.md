# Bomb Lab — 회고: 리버스 엔지니어링이라는 사고방식

## 어셈블리를 "읽을 수 있게" 된 시점

Bomb Lab을 시작할 때는 디스어셈블리가 암호처럼 보였다. 끝날 때는 `mov`, `lea`, `cmp`, `je` 같은 패턴이 눈에 들어오면서 "이건 루프다", "이건 포인터 역참조다", "이건 스위치 테이블이다"가 자동으로 분류되었다.

이 전환은 특정 순간에 갑자기 일어나지 않았다. Phase 1~3을 풀면서 "개별 명령어를 하나씩 번역하는" 단계를 거치고, Phase 4~6에서 "전체 구조를 한눈에 파악하는" 단계로 넘어갔다. 충분한 수의 구체적 예시를 보면 추상화가 자연스럽게 형성되는 것이 Data Lab에서의 경험과 동일했다.

## 미니 봄을 만든 것의 가치

공식 봄을 리버스 엔지니어링하는 것과, 같은 구조의 봄을 직접 구현하는 것은 서로 보완적인 학습이었다.

리버스 엔지니어링은 "주어진 코드에서 의도를 읽는" 연습이다. 구현은 "의도를 코드로 표현하는" 연습이다. 두 방향을 모두 경험하면 어셈블리와 C 사이의 번역이 양방향으로 자유로워진다.

특히 Phase 6의 연결 리스트 재정렬을 직접 C로 구현하면서, `7 - x` 변환이 왜 필요한지를 설계자 관점에서 이해하게 되었다. 리버스 엔지니어링할 때는 "왜 이런 변환을 넣었지?" 수준에서 멈췄던 질문이, 구현하면서 "입력-인덱스 매핑을 뒤집어야 정답이 유일해진다"라는 답을 얻었다.

## 예상과 달랐던 것들

### gdb보다 종이가 효과적인 경우가 있다

재귀 기반 페이즈(Phase 4, Secret)에서는 gdb로 step-by-step 추적하는 것보다, 의사코드를 종이에 옮기고 전개 트리를 그리는 것이 훨씬 빨랐다. gdb는 현재 상태의 스냅샷을 보여주지만, 재귀의 전체 경로를 한눈에 보여주지는 않는다.

### 공개 정책이 예상보다 까다로웠다

Bomb Lab은 교육용 과제이므로, 공식 바이너리나 특정 인스턴스의 정답을 공개 레포에 포함하면 안 된다. 이 제약 때문에 미니 봄이라는 우회 전략이 필요했고, 문서에서도 "패턴의 설명"과 "구체적 정답의 공개"를 구분하는 데 주의를 기울여야 했다.

### 검증 환경은 Data Lab에서 이미 구축되어 있었다

Docker 기반 크로스 플랫폼 검증 파이프라인은 Data Lab에서 이미 만들어두었으므로, Bomb Lab에서는 동일한 인프라를 재사용할 수 있었다. `make docker-image`로 이미 빌드된 이미지를 쓰고, `make verify-official`로 봄을 실행하면 끝이었다. 이전 과제의 인프라 투자가 이후 과제에서 그대로 회수되는 경험이었다.

## 다음에 비슷한 과제를 한다면

- **explode_bomb 브레이크포인트를 잊지 않는다.** 이건 습관이 되어야 한다.
- **재귀 페이즈는 gdb 전에 의사코드부터 쓴다.** 디버거는 확인용이지 탐색용이 아니다.
- **미니 봄의 테스트를 먼저 작성한다.** 엣지케이스 테스트가 구현의 빈틈을 잡아준다.

## 이 과제가 커리큘럼에서 갖는 위치

Bomb Lab은 Data Lab(비트 수준 사고) 다음에 오는 과제로, 여기서 확립한 어셈블리 독해 능력이 Attack Lab(버퍼 오버플로우 익스플로잇)의 기초가 된다. Attack Lab에서는 어셈블리를 읽는 것에서 한 발 더 나아가, 스택 레이아웃을 직접 조작하는 코드를 작성해야 한다. Bomb Lab에서 어셈블리와 친해지지 않으면 Attack Lab에서 막히게 된다.
