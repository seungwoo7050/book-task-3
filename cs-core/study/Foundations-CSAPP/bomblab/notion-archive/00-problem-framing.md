# Bomb Lab — 어셈블리를 읽어 폭탄을 해체하다

## 처음 바이너리를 열었을 때

Bomb Lab을 처음 받으면 눈앞에 놓이는 건 하나의 실행 파일이다. 소스코드는 없다. 이 바이너리를 실행하면 입력을 기다리고, 틀리면 "BOOM!!!"을 출력하며 터진다. 총 여섯 개의 페이즈를 순서대로 통과해야 폭탄이 해체된다.

처음 드는 생각은 "소스코드도 없이 어떻게 정답을 알아내지?" 인데, 그게 정확히 이 과제의 목표다. x86-64 어셈블리를 직접 읽고, 디버거로 실행 흐름을 추적하면서, 각 페이즈가 기대하는 입력을 역추론해야 한다.

## 이 과제가 가르치려는 것

Bomb Lab은 CS:APP 3장(기계 수준 프로그래밍)의 실습이다. 책에서 배운 어셈블리 명령어, 레지스터 규약, 호출 규약을 실제 바이너리 위에서 적용하는 것이 핵심이다. 단순히 어셈블리를 "읽을 수 있는" 수준을 넘어서, 어셈블리 코드를 보고 "이 코드가 원래 C에서 어떤 구조였는지" 역으로 추론하는 능력을 기른다.

여섯 개 페이즈는 각각 다른 프로그래밍 패턴을 어셈블리 수준에서 인식하도록 설계되어 있다:

- **문자열 비교**: 고정 문자열과의 일치 검사
- **루프 기반 수열**: 반복 관계를 만족하는 숫자들
- **스위치/점프 테이블**: 인덱스 기반 분기
- **재귀 탐색**: 재귀 함수의 반환 경로 인코딩
- **니블 룩업 테이블**: 비트 마스킹을 통한 간접 참조
- **연결 리스트 재정렬**: 포인터 체이닝과 순서 검증

그리고 숨겨진 시크릿 페이즈는 이진 탐색 트리(BST) 위의 경로 인코딩을 다룬다.

## 왜 "미니 봄" 구현이 필요했는가

공식 Bomb Lab 바이너리는 CMU의 교육 자산이다. 공개 레포에 원본 바이너리를 포함하거나 특정 인스턴스의 정답을 게시할 수 없다. 하지만 "과제를 했다"는 것을 보여주려면 실행 가능하고 테스트 가능한 코드가 있어야 한다.

그래서 같은 페이즈 패턴을 재현하는 **companion mini-bomb**을 C와 C++로 직접 구현했다. 이 미니 봄은 공식 바이너리를 복제한 것이 아니라, 각 페이즈 패밀리의 핵심 구조(문자열 비교, 수열 검증, 스위치, 재귀, 룩업, 연결 리스트, BST)를 자체적으로 설계한 것이다.

## 도구와 워크플로우

Bomb Lab의 핵심 도구는 네 가지다:

| 도구 | 역할 |
|---|---|
| `gdb` | 브레이크포인트, 레지스터 검사, 메모리 덤프, 단계 실행 |
| `objdump -d` | 디스어셈블리 덤프 생성 |
| `strings` | 바이너리 내 문자열 리터럴 탐색 |
| `nm` | 심볼 테이블 조회 |

작업 순서는 항상 동일하다:
1. `break explode_bomb` — 안전 브레이크 설정
2. 페이즈 함수의 디스어셈블리 확인 — 입력 파싱 방식 파악
3. 제어 흐름 패턴 분류 — 비교, 루프, 점프 테이블 등
4. 의사코드 작성 — 어셈블리를 C 수준으로 번역
5. 후보 정답 작성 및 검증

## 작업 환경

- **호스트**: macOS (Apple Silicon)
- **검증 환경**: Docker (linux/amd64) — 공식 봄 바이너리는 x86-64 Linux ELF
- **공식 복원**: `make restore-official` → CMU 공식 셀프스터디 봄 다운로드
- **공식 검증**: `make verify-official` → Docker 안에서 정답 파일로 봄 실행
