# Performance Lab — 디버그 기록

## 32×32 대각선 블록에서 미스 폭발

### 증상

8×8 블로킹을 적용했는데 32×32에서 미스가 300을 넘었다. 비대각선 블록에서는 예상대로 동작했지만, 대각선 블록(ii == jj)에서 미스가 비정상적으로 많았다.

### 원인

s=5, E=1, b=5 캐시에서 32×32 `int` 매트릭스의 A[i][j]와 B[i][j]는 같은 캐시 세트에 매핑된다. 대각선 블록에서 `A[i][j]`를 읽고 `B[j][i]`를 쓸 때, `i == j`이면 A와 B가 같은 세트를 놓고 경쟁한다. 직접 매핑(E=1)이므로 경쟁 = 즉시 eviction이다.

### 수정

대각선 원소를 별도 변수에 저장해 두고, 나머지 원소를 모두 처리한 뒤 마지막에 쓰는 지연 쓰기 패턴을 적용했다. 이것으로 대각선 블록 내부의 A-B 충돌을 한 번으로 줄였다.

### 교훈

"같은 캐시 세트에 매핑되는 주소가 둘 이상일 때" 어떤 일이 벌어지는지를 손으로 추적해 봐야 한다. 직접 매핑 캐시에서는 두 주소가 같은 세트에 매핑되는 순간 핑퐁이 시작된다.

## 64×64에서 나이브 8×8 블로킹 실패

### 증상

32×32에서 잘 동작하던 8×8 블로킹을 64×64에 그대로 적용했더니 미스가 2,000을 넘었다.

### 원인

64×64 매트릭스에서 한 행은 `64 * 4 = 256` 바이트를 차지한다. 캐시 라인이 32바이트이고, 32개 세트가 있으므로 캐시 1주기는 `32 * 32 = 1024` 바이트 = 4행이다. 즉 행 i와 행 i+4는 같은 캐시 세트에 매핑된다.

8×8 블록의 상단 4행과 하단 4행이 같은 캐시 세트를 공유하므로, 블록 내부에서조차 conflict miss가 발생한다. 32×32에서는 8행 간격이 공유였지만, 64×64에서는 4행 간격으로 줄어든 것이다.

### 수정

8×8 블록을 4×4 사분면으로 분할하는 3단계 전략을 적용했다. B 행렬의 임시 영역을 활용해서 상단/하단 사분면 교환을 캐시 친화적으로 수행한다.

### 교훈

블로킹 크기를 결정할 때는 "캐시 1주기에 몇 행이 들어가는가"를 먼저 계산해야 한다. 매트릭스 크기가 바뀌면 블로킹 전략도 바꿔야 한다.

## LRU 카운터 초기화 실수

### 증상

캐시 시뮬레이터에서 첫 번째 구성(s=1, E=1, b=1)의 eviction 수가 기대값과 1 차이가 났다.

### 원인

`clock`을 0에서 시작하면, 모든 빈 라인의 `lru` 필드도 0이다. 첫 번째 접근에서 라인을 채우면 `lru = 0`이 되는데, 이것이 이미 비어 있는 라인의 초기값과 동일하다. 결과적으로 LRU 선택에서 "가장 오래된 것"과 "한 번도 사용하지 않은 것"의 구분이 불가능해졌다.

### 수정

`clock`을 1에서 시작하도록 변경했다. 이렇게 하면 유효하지 않은 라인(`valid = 0`)은 항상 `lru = 0`이고, 한 번이라도 접근한 라인은 `lru >= 1`이므로 구분된다.

### 교훈

카운터의 시작값은 "0과의 구분이 필요한가"로 결정해야 한다. 초기화되지 않은 상태와 첫 번째 접근 상태가 동일한 값을 가지면 버그가 된다.

## 계측 전치에서 A와 B의 주소 충돌

### 증상

전치 벤치마크의 미스 수가 손 계산과 다르게 나왔다. 특히 32×32에서 블로킹을 적용해도 미스가 줄지 않았다.

### 원인

A와 B 매트릭스의 시작 주소를 동일하게(0부터) 설정했더니, 모든 원소에서 A와 B가 같은 캐시 세트에 매핑되었다. 이것은 공식 환경에서 A와 B가 다른 주소 영역에 있는 것과 완전히 다른 충돌 패턴을 만든다.

### 수정

A는 주소 0부터, B는 주소 `1 << 20` (1MB)부터 시작하도록 오프셋을 적용했다. 이 오프셋은 공식 환경에서 두 매트릭스가 별도의 메모리 영역에 할당되는 것을 근사한다.

### 교훈

캐시 시뮬레이션에서는 데이터의 절대 주소가 결과를 결정한다. 두 배열의 상대적 위치가 충돌 패턴을 만들기 때문에, 벤치마크에서 주소 배치를 공식 환경과 유사하게 맞추는 것이 정확도의 핵심이다.
