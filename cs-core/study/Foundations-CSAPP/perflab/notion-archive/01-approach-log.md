# Performance Lab — 접근 기록

## Part A: 캐시 시뮬레이터 구현

### 주소 분해의 첫 번째 원칙

캐시 시뮬레이터의 핵심은 주소 분해다. 64비트 주소가 주어지면:

```
tag = address >> (s + b)
set_index = (address >> b) & ((1 << s) - 1)
block_offset = address & ((1 << b) - 1)   # 시뮬레이터에서는 사용하지 않음
```

이 세 줄이 캐시 동작의 전부다. 나머지는 이 분해 결과를 가지고 상태 테이블을 업데이트하는 것뿐이다.

### LRU 구현 — 카운터 방식

LRU 교체 정책을 구현하는 방법은 여러 가지가 있다. 연결 리스트로 접근 순서를 관리할 수도 있고, 타임스탬프 카운터를 쓸 수도 있다. 여기서는 글로벌 클럭 카운터(`cache->clock`)를 선택했다. 모든 접근에서 해당 라인의 `lru` 필드를 현재 클럭으로 갱신하고, eviction이 필요하면 가장 작은 `lru` 값을 가진 라인을 선택한다.

이 방식의 장점은 구현이 단순하다는 것이다. 단점은 카운터가 오버플로할 수 있다는 것인데, `uint64_t`로 선언했으므로 실질적으로 문제가 되지 않는다.

### 접근 유형 처리

트레이스 파일의 접근 유형은 세 가지다:
- `I`: instruction fetch — 무시
- `L`: data load — cache_access 1회
- `S`: data store — cache_access 1회
- `M`: data modify — cache_access 2회 (load + store)

`M`이 2회라는 것이 핵심이다. 첫 번째 접근에서 miss가 나더라도 두 번째 접근은 반드시 hit이 된다. 이 규칙만 정확하게 구현하면 시뮬레이터는 완성이다.

### 검증 — study.trace 오라클

공식 트레이스 파일을 재배포하지 않기 위해, 자체 작성한 `study.trace`를 사용한다. 세 가지 캐시 구성에 대해 기대값을 하드코딩하고 테스트:

| 구성 | hits | misses | evictions |
|---|---|---|---|
| s=1, E=1, b=1 | 5 | 10 | 8 |
| s=2, E=1, b=2 | 6 | 9 | 7 |
| s=5, E=1, b=5 | 10 | 5 | 0 |

세 번째 구성(s=5)에서 eviction이 0인 것은 충분히 큰 캐시가 모든 워킹 세트를 담을 수 있기 때문이다.

## Part B: 행렬 전치 최적화

### 32×32 — 8×8 블로킹과 대각선 지연 쓰기

32×32 매트릭스에서 나이브 전치는 약 1,183회의 캐시 미스를 발생시킨다. 8×8 블로킹으로 줄일 수 있지만, 대각선 블록(ii == jj)에서 문제가 생긴다.

이유: 공식 캐시는 s=5, E=1, b=5 — 32세트, 직접 매핑, 캐시 라인 32바이트. 32×32 `int` 매트릭스에서 A[i][j]와 B[i][j]는 같은 캐시 세트에 매핑된다. 대각선 블록에서는 A[i][j]를 읽고 B[j][i]를 쓸 때, i==j이면 A와 B가 같은 세트를 경쟁한다.

해결: 대각선 원소를 마지막에 쓰는 지연 쓰기(deferred diagonal write) 패턴:

```c
for (j = jj; j < jj + 8; ++j) {
    int value = load_a(32, A, i, j, cache);
    if (ii == jj && i == j) {
        diag_value = value;
        diag_index = i;
    } else {
        store_b(32, B, j, i, value, cache);
    }
}
if (diag_index >= 0) {
    store_b(32, B, diag_index, diag_index, diag_value, cache);
}
```

결과: 284 misses (임계값 300 이하 달성).

### 64×64 — 사분면 전략

64×64에서는 8행마다 같은 캐시 세트에 매핑된다. 나이브 8×8 블로킹으로는 블록 내부에서도 conflict miss가 발생한다.

핵심 아이디어: 8×8 블록을 4×4 사분면 4개로 나누고, B 행렬의 "아직 사용하지 않은" 영역을 임시 저장소로 활용한다.

1단계: A의 상단 4행(8열 전부)을 읽되, 좌측 4열은 B의 정위치에, 우측 4열은 B의 임시 위치에 저장
2단계: A의 하단 좌측 4열을 읽으면서, B의 임시 위치에 있는 데이터를 B의 올바른 위치로 옮김
3단계: A의 하단 우측 4열을 B의 정위치에 저장

이 3단계 댄스는 코드가 복잡하지만, 캐시 미스를 극적으로 줄인다. 결과: 1,176 misses (임계값 1,300 이하 달성).

### 61×67 — 일반 블로킹

비정방 매트릭스에서는 정교한 전략보다 충분히 큰 블록 크기(16×16)로 블로킹하고 경계를 가드하는 것이 가장 효과적이었다.

```c
for (ii = 0; ii < N; ii += 16) {
    for (jj = 0; jj < M; jj += 16) {
        for (i = ii; i < N && i < ii + 16; ++i) {
            for (j = jj; j < M && j < jj + 16; ++j) {
                B[j][i] = A[i][j];
            }
        }
    }
}
```

결과: 1,989 misses (임계값 2,000 이하 달성).

## 계측 프레임워크 설계

공식 평가 드라이버를 대체하기 위해, `TransposeCache` 구조체로 s=5, E=1, b=5 직접 매핑 캐시를 시뮬레이션한다. 전치 커널 함수가 매트릭스 원소에 접근할 때마다 `load_a()`, `store_b()`, `load_b()` 래퍼를 호출하고, 각 래퍼가 주소를 계산해서 캐시 시뮬레이터를 구동한다.

A 매트릭스는 주소 0부터, B 매트릭스는 주소 1MB(1<<20)부터 배치하여 충돌 패턴이 공식 환경과 유사하도록 했다.
