# Performance Lab — 문제 프레이밍

## 이 랩은 무엇을 묻는가

Performance Lab(실제로는 Cache Lab의 구조를 따르는 성능 최적화 과제)은 두 개의 독립적인 파트로 구성된다.

**Part A**는 캐시 시뮬레이터를 직접 구현하라는 과제다. 메모리 접근 트레이스 파일을 읽고, 주어진 캐시 파라미터(s, E, b)에 따라 hit, miss, eviction 횟수를 정확하게 세는 프로그램을 작성해야 한다. 캐시의 동작 원리를 코드로 증명하는 것이다.

**Part B**는 행렬 전치 함수를 캐시 친화적으로 최적화하라는 과제다. 정확성은 기본이고, 그 위에서 캐시 미스 횟수를 특정 임계값 아래로 줄여야 한다. 32×32, 64×64, 61×67 세 가지 크기에 대해 각각 다른 전략이 필요하다.

두 파트를 합치면 "캐시가 실제로 어떻게 동작하는지 시뮬레이터로 이해하고, 그 이해를 바탕으로 코드를 최적화하라"가 된다.

## 왜 이 랩이 까다로운가

Part A 자체는 비교적 단순하다. 주소에서 set index와 tag를 추출하고, LRU 정책으로 victim을 선택하면 된다. 진짜 어려운 것은 Part B다.

행렬 전치는 `B[j][i] = A[i][j]`라는 한 줄짜리 로직이다. 하지만 이 한 줄이 캐시에서는 재앙이 될 수 있다. A를 행 순서로 읽으면 캐시 라인의 공간 지역성을 활용하지만, B에 열 순서로 쓰면 매번 다른 캐시 세트에 접근하게 된다. 32×32 매트릭스에서는 대각선 블록에서 A와 B가 같은 캐시 세트에 매핑되어 충돌 미스가 폭발한다.

64×64는 더 심하다. 32바이트 캐시 라인에 `int` 8개가 들어가므로, 8행마다 같은 캐시 세트에 매핑되고, 나이브 8×8 블로킹이 자기 자신을 evict한다. 이것을 해결하려면 "B 행렬의 일부 영역을 임시 버퍼로 사용하는" 정교한 사분면 전략이 필요하다.

## 선행 지식

- 캐시 구조: set-associative, direct-mapped, LRU 교체 정책
- 주소 분해: tag, set index, block offset
- 공간/시간 지역성(spatial/temporal locality)
- 행렬 메모리 레이아웃: 행 우선(row-major) 저장
- 블로킹: 캐시 라인 크기에 맞춘 타일 분할

## 이 프로젝트의 전략

공식 핸드아웃의 평가 드라이버와 트레이스 파일은 로컬에만 존재하므로, 독립적으로 검증 가능한 컴패니언 모델을 구축했다:

1. `problem/` — 자체 작성한 starter 파일과 study-owned `study.trace` 보관
2. `c/`와 `cpp/` — 캐시 시뮬레이터 + 계측 전치 벤치마크
3. 전치 벤치마크는 공식 캐시 파라미터(s=5, E=1, b=5)를 시뮬레이션하면서 미스 횟수를 측정

이 구조 덕분에 공식 드라이버 없이도 "정확성 + 미스 임계값" 판정이 가능하다.
