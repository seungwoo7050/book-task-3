# Data Lab — 회고: 비트 연산이 사고방식을 바꾸다

## 과제를 마치고 돌아보면

Data Lab을 끝내고 나서 가장 크게 달라진 건, 코드를 읽을 때 "이 연산이 비트 수준에서 어떻게 동작하는가"를 자연스럽게 떠올리게 된 것이다. 이전에는 `x < 0`을 쓰면 그냥 "음수인지 확인"이라고만 생각했는데, 이제는 "부호 비트를 보는 것"임을 자동으로 인식하게 되었다.

이건 CS:APP 2장을 읽었을 때는 생기지 않았던 변화다. 읽으면 이해하지만 체화되지는 않는다. 비트 연산만으로 조건 분기, 비교, 변환을 직접 조립하면서 비로소 체화가 된 것 같다.

## 예상과 달랐던 것들

### 검증 환경 구축이 풀이보다 어려웠다

과제를 시작하기 전에는 퍼즐 풀이에 대부분의 시간을 쓸 거라고 예상했다. 실제로는 **Docker 기반 검증 파이프라인을 만드는 데** 더 많은 시간이 들었다. `dlc`가 Linux/amd64 전용 바이너리라는 사실, Apple Silicon에서의 QEMU 에뮬레이션 오버헤드, `-m32` 플래그 비호환 — 이런 문제들은 퍼즐 자체와는 관련이 없지만, "내 풀이가 맞다"는 확신을 얻기 위해 반드시 해결해야 했다.

하지만 이 과정도 배움이 아예 없었던 건 아니다. Docker 빌드, 크로스 플랫폼 Makefile, 아키텍처 감지 패턴 — 이런 것들은 시스템 프로그래밍을 하면서 반복적으로 마주치는 기술이다. 의도하지 않은 부수적 학습이었지만, 결과적으로 쓸모가 있었다.

### 패턴이 반복된다는 걸 깨닫는 순간

처음 몇 개의 퍼즐은 각각이 완전히 새로운 문제처럼 느껴졌다. 그런데 5~6번째 퍼즐을 풀 때쯤, "이건 아까 썼던 부울-마스크 변환이잖아"라는 인식이 생기기 시작했다. 그때부터 속도가 완전히 달라졌다.

이 경험은 비트 연산에만 해당하는 게 아니라, 프로그래밍 일반에도 적용되는 패턴 같다. 충분한 수의 구체적 문제를 풀어야 추상적 패턴이 보이고, 패턴이 보이면 새로운 문제에 대한 접근 속도가 급격히 빨라진다.

### 부동소수점 퍼즐은 정수 퍼즐과 전혀 다른 근육을 쓴다

정수 퍼즐에서는 "제한된 연산자로 원하는 결과를 어떻게 조립할까"가 핵심이었다면, 부동소수점 퍼즐에서는 "입력이 어느 범주에 속하는지 어떻게 정확히 분류할까"가 핵심이었다.

IEEE 754의 정규/비정규/특수값 분류를 직접 코드로 구현하면서, 왜 이 규격이 이렇게 복잡한 구간 구분을 갖고 있는지를 체감했다. 특히 비정규수의 존재 이유 — 0 근처에서 급격한 언더플로우 대신 점진적 정밀도 감소를 제공하는 것 — 는 코드로 다루기 전까지는 교과서의 문장에 불과했다.

## 다음에 비슷한 과제를 한다면

- **검증 환경을 먼저 만든다.** 풀이를 시작하기 전에 "내 답이 맞는지 확인하는 방법"부터 확보해야 한다. Data Lab에서는 Docker 파이프라인이 이에 해당했다.
- **경계값 테스트를 처음부터 작성한다.** 공식 `btest`에 의존하지 않고, 0, -1, INT_MIN, INT_MAX, NaN, ±∞ 등을 직접 명시한 테스트를 먼저 만든다.
- **"이 성질이 유일한가?"를 항상 묻는다.** `isTmax`에서 -1에 발목 잡힌 경험은, 비트 성질 기반 판별에서 가장 중요한 검증 질문을 남겨주었다.

## 이 과제가 커리큘럼에서 갖는 위치

Data Lab은 CS:APP의 첫 번째 실습이면서, 이후 Bomb Lab(어셈블리 리버스 엔지니어링)과 Attack Lab(버퍼 오버플로우 익스플로잇)의 기초가 된다. 비트 수준에서 데이터를 읽는 능력이 없으면 어셈블리 코드를 해석할 때 걸리게 되고, 스택 레이아웃을 이해할 때도 막히게 된다.

이 과제를 통해 "비트 단위의 언어"를 갖추게 되었고, 그 언어가 이후 과제들의 기본 어휘가 된다.
