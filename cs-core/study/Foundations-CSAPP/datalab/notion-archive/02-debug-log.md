# Data Lab — 디버깅 이야기: 정답인 코드가 실패할 때

## Docker 안에서 정답이 타임아웃 나다

가장 당황했던 순간은 Docker 안에서 `btest`를 돌렸는데, **분명히 정답인 솔루션이 타임아웃으로 실패**한 것이었다.

처음에는 코드에 무한루프라도 있나 의심했다. 하지만 같은 솔루션을 네이티브 Linux x86_64 환경에서 돌리면 문제없이 통과한다는 걸 확인했다. 원인은 **Apple Silicon 위의 QEMU amd64 에뮬레이션** 때문이었다.

`btest`의 기본 타임아웃은 10초(`-T 10`)인데, 에뮬레이션 오버헤드 때문에 `howManyBits`나 `floatFloat2Int` 같은 복잡한 퍼즐에서 시간이 부족했다. 해결은 단순했지만, 원인을 찾기까지가 머리 아팠다:

```bash
# 기본 실행 (타임아웃 발생)
./btest

# 에뮬레이션 환경용 수정
./btest -T 20
```

Makefile의 `verify-official` 타겟에 `-T 20`을 하드코딩해서 고정했다. 중요한 건 이게 에뮬레이션 환경의 성능 특성 때문이지, 솔루션 자체의 성능 문제가 아니라는 점을 명확히 구분하는 것이었다.

이 경험에서 배운 건, 크로스 플랫폼 검증 파이프라인을 만들 때 에뮬레이터의 성능 프로파일을 반드시 고려해야 한다는 것이다. 네이티브에서는 1초면 끝나는 테스트가 에뮬레이션에서는 20초를 넘길 수 있고, 그러면 정답인 코드가 거짓으로 실패한다. 타임아웃 같은 값은 환경에 따라 파라미터화하거나 최소한 문서화해야 한다.

## isTmax: -1이라는 복병

`isTmax(x)`는 "x가 INT_MAX인가?"를 판별하는 문제다. 첫 접근은 INT_MAX가 가진 특이한 비트 성질 — `x + 1`을 했을 때 모든 비트가 반전되는 것 — 을 이용하는 것이었다:

```c
// 첫 시도
return !(~(x + (x + 1)));
```

INT_MAX(0x7FFFFFFF)에서는 잘 동작했다. 그런데 테스트에서 `-1`을 넣었을 때도 1을 반환하는 걸 발견했다.

왜일까? `-1 + 1 = 0`이고, `~((-1) + 0) = ~(-1) = 0`이므로 `!0 = 1`이 되어 통과해버린다. INT_MAX만 가진 성질이라고 생각했는데, -1도 같은 패턴을 보인 것이다.

수정은 `x + 1`이 0인 경우(즉 x가 -1인 경우)를 명시적으로 배제하는 한 줄을 추가하는 것이었다:

```c
int y = x + 1;
return !(~(x + y)) & !!y;
```

`!!y`는 y가 0이면 0, 아니면 1을 반환하므로, -1 케이스가 깔끔하게 걸러진다.

비트 성질 기반 판별에서 가장 위험한 가정은 "이 성질이 유일할 것이다"라는 것이다. 항상 "이 성질을 가진 다른 값이 있는가?"를 확인해야 한다. 이건 이후 퍼즐에서도 반복적으로 적용한 검증 습관이 되었다.

## isLessOrEqual: 오버플로우를 무시하면 생기는 일

`isLessOrEqual(x, y)`에서는 직관적으로 "y - x가 0 이상이면 true"를 비트로 확인하려 했다:

```c
// 첫 시도
int diff = y + (~x + 1);
return !(diff >> 31);
```

대부분의 입력에서 잘 동작했지만, `isLessOrEqual(INT_MIN, INT_MAX)`에서 실패했다. `INT_MAX - INT_MIN`의 수학적 결과는 양수지만, 32비트 `int` 범위를 넘어서 오버플로우가 발생하면서 음수가 되기 때문이다.

이 문제의 해법은 **부호가 다른 경우와 같은 경우를 아예 분리**해서 처리하는 것이다. 부호가 다르면 뺄셈을 하지 않고 부호만 보면 되고, 부호가 같으면 뺄셈에서 오버플로우가 발생하지 않으므로 결과를 신뢰할 수 있다:

```c
int sx = x >> 31;
int sy = y >> 31;
int diff_sign = sx ^ sy;
// 부호가 다르면: x가 음수이면 무조건 x ≤ y
// 부호가 같으면: y - x의 결과 부호로 판단
return (diff_sign & !!sx) | ((!diff_sign) & diff_nonneg);
```

2의 보수 뺄셈은 같은 부호일 때만 안전하다는 것 — 이게 이 디버깅에서 얻은 가장 중요한 교훈이었다.

## floatFloat2Int: 비트 31에서 벌어지는 미묘한 문제

`floatFloat2Int`에서 오버플로우 판정을 `exp > 31`로 잡았다가 `btest`에서 실패했다.

문제는 `exp == 31`일 때였다. significand에는 항상 implicit 1이 포함되어 있으므로 `frac << (31 - 23) = frac << 8`을 하면 비트 31(부호 비트 자리)까지 밀린다. 양수인 float인데 `int`로 변환하면 음수가 되어버리는 것이다.

경계를 `exp > 30`으로 수정하면, 이런 케이스를 `0x80000000u`(오버플로우 표시)로 올바르게 반환한다:

```c
if (exp > 30) {
    return 0x80000000u;
}
```

비트 시프트와 타입 경계가 동시에 작용하는 코드에서는, 시프트 결과가 부호 비트를 건드리는 경우를 반드시 별도로 확인해야 한다. "exp > 31이면 오버플로우"라는 직관이 맞는 것처럼 느껴지지만, implicit bit 때문에 실제 경계는 한 칸 낮다.

## Apple Silicon에서 `-m32` 플래그가 통하지 않는다

로컬에서 `problem/Makefile`을 `make`했을 때 `-m32` 플래그 관련 링크 에러가 나왔다. Apple Silicon macOS의 Xcode 기본 컴파일러는 32비트 타겟을 지원하지 않기 때문이다.

Makefile에 아키텍처 감지 분기를 추가해서 해결했다:

```makefile
UNAME_S  := $(shell uname -s)
UNAME_M  := $(shell uname -m)
ARCH_FLAGS := -m32

ifeq ($(UNAME_S),Darwin)
ifeq ($(UNAME_M),arm64)
ARCH_FLAGS :=
endif
endif
```

Apple Silicon에서는 `-m32`를 빼고 64비트로 빌드한다. 솔루션 자체가 `int`만 사용하기 때문에 32/64비트 차이가 비트 연산 결과에 영향을 주지는 않지만, 공식 `dlc` 검증은 반드시 Docker(linux/amd64) 안에서 수행하는 것으로 정리했다.

여기서 배운 건, "로컬에서 make가 통과한다"와 "공식 도구가 통과한다"는 별개의 검증이라는 것이다. 크로스 플랫폼 C 프로젝트에서 아키텍처 플래그는 항상 조건부로 설정해야 하고, 공식 검증은 타겟 환경에서 반드시 따로 돌려야 한다.
