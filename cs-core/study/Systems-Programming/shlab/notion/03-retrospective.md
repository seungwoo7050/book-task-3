# Shell Lab — 회고

## 시그널은 왜 어려운가

Shell Lab을 시작하기 전에는 시그널 핸들링을 "핸들러 함수를 등록하면 끝"이라고 생각했다. 실제로 코드를 작성해보니, 시그널의 어려움은 핸들러 함수를 작성하는 것이 아니라 "핸들러가 실행되는 시점을 제어하는 것"에 있었다.

`SIGCHLD`가 `fork`와 `addjob` 사이에 도착하는 레이스, `printf`가 데드락을 일으키는 재진입 문제, `waitpid`의 이중 회수 — 이 모든 문제의 공통점은 "비동기 이벤트와 메인 흐름의 간섭"이다. 프로그래머가 작성한 순서대로 실행된다는 동기 프로그래밍의 가정이 시그널 앞에서 무너진다.

이 경험이 알려준 것은 시그널 프로그래밍의 핵심 원칙이다:

1. **임계 구역 보호**: 공유 자료구조(잡 테이블)에 접근하는 구간에서는 해당 시그널을 차단한다.
2. **시그널 안전 함수만 사용**: 핸들러에서는 POSIX 시그널 안전 함수 목록에 있는 것만 호출한다.
3. **한 곳에서만 회수**: 자식 회수는 핸들러에서 하거나 메인에서 하거나, 둘 중 하나만 선택한다.
4. **errno 보존**: 핸들러 진입 시 `errno`를 저장하고 복원한다.

## 프로세스 그룹의 중요성

`setpgid(0, 0)`이 왜 필요한지를 처음에는 이해하지 못했다. 그냥 시그널을 전달하면 되지 않나?

문제는 Ctrl-C를 누르면 커널이 터미널의 전경 프로세스 그룹 전체에 `SIGINT`를 보낸다는 것이다. 쉘과 자식이 같은 프로세스 그룹에 있으면, 쉘도 `SIGINT`를 받고 종료해 버린다.

`setpgid(0, 0)`으로 자식을 별도 프로세스 그룹에 넣으면, 쉘은 시그널을 받아서 핸들러에서 `kill(-pid, SIGINT)`로 전달만 하는 중개자 역할을 한다. 이것이 모든 Unix 쉘의 기본 아키텍처다.

## sigsuspend의 우아함

`waitfg()`를 처음에는 바쁜 대기(`while (fgpid(jobs) == pid) {}`)로 구현했다. CPU를 100% 먹었다. `sleep(1)`을 넣으면 반응이 1초 느려졌다.

`sigsuspend()`는 이 딜레마를 해결한다. "시그널이 도착할 때까지 원자적으로 대기"하므로, CPU를 소모하지 않으면서도 시그널 도착 즉시 깨어난다. 그리고 시그널 마스크를 일시적으로 변경하면서 대기하므로, 시그널 수신과 대기 사이의 레이스도 방지한다.

이것은 `poll()`이나 `select()` 같은 I/O 멀티플렉싱과 같은 철학이다: "이벤트가 올 때까지 효율적으로 대기하라."

## 테스트 하네스에 대한 생각

공식 trace/driver가 없으므로, FIFO 기반 직접 하네스를 만들어야 했다. 네 가지 테스트 케이스(basic_echo, bg_jobs, sigint, stop_fg)로 핵심 시나리오를 커버한다.

이 하네스의 장점은 쉘을 블랙박스로 테스트한다는 것이다. 실제 쉘 프로세스를 실행하고, 실제 시그널을 보내고, 실제 출력을 확인한다. 단위 테스트로는 할 수 없는, 시그널 핸들링의 통합 검증이 가능하다.

단점은 타이밍에 의존한다는 것이다. 비동기 프로세스의 상태 전이를 정확하게 기다리는 메커니즘이 아니라, `sleep`으로 추정하기 때문에 느린 시스템에서는 실패할 수 있다.

## 다음에 다시 한다면

- `sio_puts()`에 날짜/시각 출력을 추가하겠다. 시그널 핸들러의 실행 순서를 디버깅할 때, 각 메시지의 시간을 알면 레이스를 파악하기 훨씬 쉽다.
- 테스트 하네스에서 `sleep` 대신 동기화 메커니즘을 사용하겠다. 예를 들어, 쉘이 프롬프트를 출력하는 것을 감지하면 "이제 잡이 시작됐다"고 판단할 수 있다.
- 파이프라인(`|`)과 리다이렉션(`>`, `<`)을 추가 구현하겠다. 공식 과제에는 포함되지 않지만, 실제 쉘의 동작을 이해하는 데 큰 도움이 될 것이다.
