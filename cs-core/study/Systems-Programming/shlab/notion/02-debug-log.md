# Shell Lab — 디버그 기록

## fork/addjob 레이스 — 잡 테이블 불일치

### 증상

배경 잡을 빠르게 여러 개 실행하면, 가끔 `jobs` 명령에서 이미 종료된 잡이 남아 있거나, 반대로 아직 실행 중인 잡이 보이지 않았다.

### 원인

`fork()` 후 `addjob()` 전에 자식이 종료하면 `SIGCHLD`가 도착한다. 핸들러가 `deletejob()`을 호출하지만, 아직 잡이 추가되지 않았으므로 삭제 대상을 찾지 못한다. 이후 부모가 `addjob()`을 호출하면, 이미 죽은 프로세스의 PID가 잡 테이블에 영구적으로 남는다.

### 수정

`fork()` 전에 `SIGCHLD`를 `sigprocmask(SIG_BLOCK, ...)`로 차단하고, `addjob()` 완료 후에 복원한다. 이렇게 하면 핸들러는 잡이 테이블에 추가된 후에만 실행된다.

### 교훈

"fork와 관련된 모든 공유 자료구조 업데이트는 SIGCHLD 차단 구간 안에서 수행해야 한다." 이것이 Shell Lab의 핵심 규칙이다.

## 시그널 핸들러에서 printf 사용 — 데드락

### 증상

`sigchld_handler()`에서 `printf`로 "Job terminated" 메시지를 출력하면, 쉘이 간헐적으로 멈췄다.

### 원인

`printf`는 내부적으로 `stdout` 버퍼의 락을 잡는다. 메인 루프에서 `printf`를 호출하는 도중에 시그널이 도착하면, 핸들러에서 또 `printf`를 호출하려 하고, 같은 락을 이중으로 잡으려는 데드락이 발생한다. `printf`는 async-signal-safe하지 않다.

### 수정

핸들러에서 `sio_puts()`와 `sio_putl()`만 사용한다. 이 함수들은 `write()` 시스템 콜만 호출하므로 시그널 안전하다.

### 교훈

시그널 핸들러에서 호출할 수 있는 함수는 POSIX 시그널 안전 함수 목록에 있는 것만이다. 이 제약을 무시하면 간헐적 데드락이라는 가장 디버깅하기 어려운 버그가 된다.

## waitpid 이중 회수 문제

### 증상

전경 잡이 종료할 때 "No child processes" 에러가 간헐적으로 출력되었다.

### 원인

초기 구현에서 `waitfg()`가 `waitpid(pid, ...)`로 직접 대기했다. 그런데 `SIGCHLD` 핸들러도 `waitpid(-1, ...)`로 같은 자식을 회수한다. 핸들러가 먼저 자식을 회수하면, `waitfg()`의 `waitpid`가 "그런 자식 없음" 에러를 반환한다.

### 수정

`waitfg()`를 `sigsuspend()` 기반으로 변경했다. 직접 `waitpid`를 호출하지 않고, 핸들러가 잡 테이블을 갱신하면 `fgpid(jobs) != pid` 조건으로 루프를 탈출한다.

```c
while (fgpid(jobs) == pid) {
    sigsuspend(&empty);
}
```

### 교훈

자식 회수는 한 곳에서만 해야 한다. 핸들러에서 회수하는 방식을 선택했으면, 메인 코드에서는 잡 상태만 확인해야 한다.

## 자식 프로세스의 시그널 마스크 상속

### 증상

`/bin/sleep` 같은 외부 프로그램이 Ctrl-C에 반응하지 않았다.

### 원인

부모에서 `SIGCHLD`를 차단한 상태로 `fork()`하면, 자식도 `SIGCHLD`가 차단된 채로 태어난다. 여기까지는 `SIGCHLD`만 영향을 받지만, `sigfillset(&mask_all); sigprocmask(SIG_BLOCK, &mask_all, NULL)` 단계에서 모든 시그널이 차단되면, 자식이 `execvp` 전에 마스크를 복원하지 않는 경우 실행되는 프로그램도 모든 시그널이 차단된다.

### 수정

자식에서 `execvp` 전에 반드시 시그널 마스크를 복원한다:

```c
if (pid == 0) {
    sigprocmask(SIG_SETMASK, &prev, NULL);  // fork 이전 마스크로 복원
    setpgid(0, 0);
    execvp(argv[0], argv);
    _exit(1);
}
```

### 교훈

`fork()`로 만든 자식은 부모의 시그널 마스크를 그대로 물려받는다. 부모가 시그널을 차단한 이유(레이스 방지)는 자식에게는 적용되지 않으므로, `execvp` 전에 반드시 복원해야 한다.

## FIFO 테스트에서 타이밍 문제

### 증상

`sigint` 테스트 케이스가 CI에서 간헐적으로 실패했다. 쉘에 시그널을 보내기 전에 전경 잡이 아직 시작되지 않았다.

### 수정

FIFO에 명령을 쓴 뒤 `sleep 1`로 충분한 대기 시간을 주었다. 이상적으로는 쉘이 "잡을 시작했다"는 이벤트를 동기적으로 기다려야 하지만, FIFO 기반 테스트에서는 적절한 슬립이 실용적인 해결책이다.

### 교훈

비동기 프로세스를 테스트할 때는 "상태 전이를 기다리는 메커니즘"이 필수다. 슬립은 차선책이지만, 전용 동기화 없이는 불가피하다.
