# Proxy Lab — 문제 프레이밍

## 이 랩은 무엇을 묻는가

Proxy Lab은 HTTP 웹 프록시를 직접 구현하는 과제다. 세 가지 마일스톤이 순서대로 쌓인다:

1. **순차 전달**: 클라이언트의 HTTP GET 요청을 파싱하고, 원본 서버에 연결해서 요청을 전달하고, 응답을 클라이언트에 돌려준다.
2. **동시 처리**: 매 연결마다 별도 스레드를 생성해서, 여러 클라이언트를 동시에 서비스한다.
3. **캐싱**: 응답을 메모리에 저장해서, 같은 URL에 대한 두 번째 요청은 원본 서버에 접속하지 않고 캐시에서 서비스한다. LRU 교체 정책으로 총 캐시 크기를 제한한다.

이 세 마일스톤을 합치면 "네트워크 프로그래밍, 동시성, 공유 상태 관리"라는 시스템 프로그래밍의 핵심 주제가 모두 들어 있다.

## 왜 이 랩이 까다로운가

첫 번째 난관은 HTTP 파싱이다. 클라이언트로부터 받는 요청은 absolute-form URI(`http://host:port/path`)를 포함한다. 이것을 호스트, 포트, 경로로 분해하고, HTTP/1.0 형식의 요청으로 재구성해야 한다. 특정 헤더(`Host`, `User-Agent`, `Connection`, `Proxy-Connection`)는 무조건 프록시가 결정한 값으로 덮어쓰고, 나머지는 그대로 전달해야 한다.

두 번째 난관은 동시성에서 오는 공유 상태 문제다. 캐시는 모든 스레드가 공유하는 전역 자료구조다. 한 스레드가 캐시에 쓰는 동안 다른 스레드가 읽으면 데이터 레이스가 발생한다. 뮤텍스로 보호해야 하지만, 캐시 락을 쥔 채 클라이언트 I/O를 하면 다른 모든 요청이 블록된다.

세 번째 난관은 강건성이다. 원본 서버가 다운되거나, 클라이언트가 갑자기 연결을 끊거나, `SIGPIPE`가 발생할 수 있다. 프록시는 이런 상황에서 죽지 않고 다음 요청을 계속 서비스해야 한다.

## 선행 지식

- 소켓 프로그래밍: `socket`, `bind`, `listen`, `accept`, `connect`
- HTTP/1.0 프로토콜: 요청 라인, 헤더, 몸체
- POSIX 스레드: `pthread_create`, `pthread_detach`, `pthread_mutex_lock`
- Robust I/O: `rio_readlineb`, `rio_writen` (CS:APP csapp.c 라이브러리)
- 연결 리스트: LRU 캐시의 순서 관리

## 이 프로젝트의 전략

1. `problem/` — CS:APP의 `csapp.c`/`csapp.h` 헬퍼와 starter proxy 보관
2. `c/`와 `cpp/` — 처음부터 작성한 프록시 구현
3. `tests/` — Python 기반 로컬 원본 서버 + curl 기반 하네스로 전체 마일스톤 검증
4. 공식 드라이버 없이도 전달, 헤더 재작성, 캐시, 동시성, 실패 복구를 모두 검증하는 구조
