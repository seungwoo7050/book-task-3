# readme-contract

이 문서는 루트, 트랙, 프로젝트 README가 따라야 할 공개 표면 계약을 정의한다.

## 루트 README 계약

- 첫 문단에서 레포의 성격을 `multi-track study archive`로 명확히 밝힌다.
- `트랙 요약` 표를 둔다.
- `프로젝트 카탈로그` 표를 둔다.
- 각 프로젝트 행에는 최소한 `문제`, `이 레포의 답`, `검증 시작점`, `상태`가 포함돼야 한다.
- 루트에서 임의 프로젝트의 `문제`, `답`, `검증`으로 2클릭 이내에 도달할 수 있어야 한다.

## 트랙 README 계약

- 고정 섹션 순서는 `트랙 한줄 설명` -> `프로젝트 지도 표` -> `권장 순서` -> `검증 원칙` -> `공개 경계`다.
- `프로젝트 지도 표`는 각 프로젝트의 `문제`, `이 레포의 답`, `검증 시작점`, `상태`를 한 줄로 요약한다.
- 트랙 소개는 짧게 유지하고, 개별 프로젝트의 긴 설명을 대신하지 않는다.

## 프로젝트 README 계약

- 고정 섹션 순서는 `프로젝트 한줄 요약` -> `한눈에 보기` -> `디렉터리 역할` -> `검증 빠른 시작` -> `공개 경계/현재 한계`다.
- `한눈에 보기` 표의 최소 컬럼은 `문제`, `중요 제약`, `이 레포의 답`, `검증 시작점`, `배우는 개념`, `상태`다.
- README 첫 화면에서 구현 세부 reasoning을 길게 풀지 않는다.
- `누구를 위한 문서`, `포트폴리오 힌트`, 장문 배경 설명은 프로젝트 `docs/`나 `notion/`으로 내린다.

## 링크와 명령 표기

- 링크는 현재 저장소 안의 실제 경로만 가리킨다.
- `검증 시작점`은 보통 `problem/README.md`, 구현 디렉터리 `README.md`, 또는 canonical test entrypoint를 가리킨다.
- 명령은 복붙 가능한 block으로 적고, 성공 신호가 중요하면 한 줄 설명을 붙인다.
- 공식 자산이 필요한 경우 `restore-official`, `verify-official` 같은 local-only 경계를 명시한다.

## 상태 표기

- `verified`: 현재 문서화된 명령으로 재검증 가능한 상태
- `verified (local-only asset)`: 공식 자산을 로컬에 복원해야 전체 검증이 완료되는 상태
- `public verified`: 현재 저장소 안의 공개 자산만으로 검증이 끝나는 상태

루트와 트랙 표에서는 너무 긴 설명 대신 위 상태 문자열을 그대로 사용한다.
