# Documentation Style Guide

이 가이드는 이 레포의 모든 공개 문서가 같은 목적과 같은 톤을 유지하도록 만드는 기준입니다. 기준일은 2026-03-10입니다.

## 기본 목적

- 문서는 정답집처럼 보이면 안 됩니다.
- 문서는 구현을 따라가는 사람이 “왜 이렇게 설계했는가”를 먼저 이해하도록 도와야 합니다.
- 문서는 학습자에게 친절해야 하며, 동시에 더 나은 공개용 포트폴리오 레포를 설계할 수 있는 힌트를 줘야 합니다.

## 문서 첫 화면 규칙

- 첫 문단에서 이 프로젝트가 무엇을 가르치는지 바로 말합니다.
- 상태 나열보다 학습 목표와 읽는 순서를 먼저 보여 줍니다.
- 한국어를 기본으로 쓰되, 기술 용어는 필요한 곳에 영어 원어를 병기합니다.

## README 규칙

- 루트/트랙 README는 “소개글”보다 “학습 시작 안내서”에 가깝게 씁니다.
- 프로젝트 README는 최소한 다음을 포함합니다.
  - 무엇을 배우는가
  - 읽기 전에 알면 좋은 것
  - 어떤 파일부터 읽으면 좋은가
  - 검증 명령
  - 구현에서 집중할 포인트
  - 포트폴리오로 발전시키는 힌트

## 문제 문서 규칙

- `problem/README.md`는 “무엇을 구현해야 하는가”를 분명히 적습니다.
- 현재 레포에 없는 과거 로컬 경로를 현재 구조처럼 약속하지 않습니다.
- provenance는 아래 네 가지를 설명형 문장으로 남깁니다.
  - 원래 어떤 학습 주제였는가
  - 원래 어떤 구현 형태였는가
  - 현재 프로젝트에서 무엇을 유지했고 무엇을 재구성했는가
  - 원본 소스 트리는 현재 레포에 포함되지 않는다는 사실

## 개념 문서와 참고자료 규칙

- `docs/README.md`는 링크 목록이 아니라 읽는 이유를 설명하는 안내서여야 합니다.
- `docs/references/README.md`는 bare path 목록이 아니라 설명형 reference index여야 합니다.
- 각 참고자료는 최소한 제목, 출처 유형, 확인일, 참고 이유, 반영 내용을 포함합니다.

## 학습 노트 정책

- `notion/`은 현재 공개용 학습 노트이자 레포 백업 문서입니다.
- `notion-archive/`는 이전 세대 문서를 보존하는 아카이브입니다.
- 새 `notion/`은 복붙형 장문보다 문제 프레이밍, 접근 로그, 디버그 로그, 회고, 지식 인덱스를 기준으로 재구성합니다.

## 친절한 톤 규칙

- 독자가 처음 본다는 가정으로 씁니다.
- “이건 당연하다”는 식의 생략을 피하고, 설계 의도를 짧게라도 설명합니다.
- 설명은 코드보다 먼저 나오되, 실제 파일 경로와 검증 명령으로 이어져야 합니다.
- 포트폴리오 힌트는 과장된 브랜딩보다 “무엇을 추가하면 설계 깊이가 드러나는가”에 집중합니다.
