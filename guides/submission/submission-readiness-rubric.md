# 제출 준비도 평가 루브릭

프로젝트가 캡스톤 제출 또는 포트폴리오 공개 준비가 됐는지 판정하는 스코어카드다.  
각 항목을 점수로 환산하고 통과 임계값을 만족해야 submit-ready로 선언한다.

---

## 평가 체계

### 채점 단위

| 점수 | 의미 |
|------|------|
| **0** | 없음 또는 미구현 |
| **1** | 부분 구현 / 명백한 결함 존재 |
| **2** | 완료 / 기준 충족 |

### 가중치 및 최대 점수

| 카테고리 | 항목 수 | 최대 점수 |
|----------|---------|-----------|
| 기능 완성도 | 3 | 6 |
| 재현 가능성 | 3 | 6 |
| 테스트 품질 | 3 | 6 |
| 운영 증거 | 4 | 8 |
| 문서 품질 | 3 | 6 |
| **합계** | **16** | **32** |

### 통과 임계값

| 수준 | 조건 | 설명 |
|------|------|------|
| **submit-ready** | 총점 ≥ 24 점 | AND 운영 증거 ≥ 6점 |
| **conditional** | 총점 20–23 점 | 특정 항목 보완 후 재검토 |
| **not-ready** | 총점 < 20 점 | 구조적 보완 필요 |

> **주니어 전환 지원 기준**: submit-ready 이상이어야 포트폴리오 프로젝트로 제시 가능하다.  
> 운영 증거 소항목(로드 테스트, 장애 주입)에서 최소 1점 이상이 없으면  
> 총점이 24점 이상이더라도 conditional로 처리한다.

## 카테고리 1 — 기능 완성도

| # | 항목 | 0 | 1 | 2 |
|---|------|---|---|---|
| 1.1 | 핵심 기능 end-to-end 동작 | 미구현 | 동작하지 않는 경로 존재 | 모든 핵심 경로 동작 |
| 1.2 | 에러 처리 | 처리 없음 | 일부 케이스만 처리 | 모든 예측 가능한 에러 처리 |
| 1.3 | 데이터 정합성 | 검증 없음 | 일부 경계 케이스 누락 | 입력 검증 및 상태 일관성 유지 |

---

## 카테고리 2 — 재현 가능성

| # | 항목 | 0 | 1 | 2 |
|---|------|---|---|---|
| 2.1 | 단일 명령 기동 | 불가 | 여러 수동 단계 필요 | `make up` 또는 `docker compose up` 한 줄 |
| 2.2 | 환경 변수 문서화 | 없음 | 일부 누락 | `.env.example` 또는 전체 목록 README 포함 |
| 2.3 | 클린 빌드 검증 | 확인 안 됨 | 경고 존재 | 클린 환경에서 빌드·기동 성공 확인 |

---

## 카테고리 3 — 테스트 품질

| # | 항목 | 0 | 1 | 2 |
|---|------|---|---|---|
| 3.1 | 단위·통합 테스트 통과 | 없음 | 일부 실패 존재 | 모두 통과 |
| 3.2 | 테스트 커버리지 의미 있음 | 없음 또는 trivial | 핵심 경로 일부 누락 | 핵심 로직 경로 커버 |
| 3.3 | 로드 테스트 결과 첨부 | 없음 | smoke만 실행 | baseline 이상 실행 + 결과 첨부 |

> 로드 테스트 실행 방법: [load-testing.md](../testing/load-testing.md)

---

## 카테고리 4 — 운영 증거

| # | 항목 | 0 | 1 | 2 |
|---|------|---|---|---|
| 4.1 | 로드 테스트 (SLO 검증) | 없음 | 실행했으나 SLO 미달 또는 기록 미비 | baseline 통과 + SLO 달성 결과 첨부 |
| 4.2 | 장애 주입 훈련 | 없음 | 1개 시나리오 실행 (보고서 미작성) | 1개 이상 시나리오 + 보고서 첨부 |
| 4.3 | 모니터링 계측 | 없음 | `/health` 또는 로그만 존재 | `/metrics` + 구조화 로그 + 계측 체크리스트 통과 |
| 4.4 | Evidence Package | 없음 | 일부 항목 누락 | 변경 전후 메트릭 + 타임라인 + 런북 링크 |

> 운영 증거 포맷: [observability-reliability.md — Evidence Package](../devops/observability-reliability.md#5-evidence-package--readme-첨부-섹션)  
> 장애 주입 시나리오: [failure-injection.md](../devops/failure-injection.md)

## 카테고리 5 — 문서 품질

| # | 항목 | 0 | 1 | 2 |
|---|------|---|---|---|
| 5.1 | README 완성도 | 없음 또는 뼈대만 | 빌드·테스트 명령 일부 누락 | 빌드 + 테스트 + 실행 + 증거 섹션 완비 |
| 5.2 | 아키텍처 설명 | 없음 | 설명만 있고 다이어그램 없음 | 텍스트 도표 또는 이미지 다이어그램 첨부 |
| 5.3 | 알려진 한계 | 없음 | 막연한 TODO만 있음 | Known Limitations 섹션에 구체적으로 기재 |

## 보완 우선순위 결정 방법

총점이 낮을 때 어떤 항목부터 보완할지 결정하는 기준이다.

```
1. 0점 항목을 먼저 찾는다
   → 0점은 해당 요소가 완전히 없다는 신호다

2. 운영 증거 카테고리를 우선한다
   → 통과 임계값에 별도 조건이 붙어 있기 때문이다

3. 재현 가능성 2.1 을 일찍 해결한다
   → 기동이 한 줄이 아니면 다른 모든 검증이 느려진다

4. 테스트 품질 3.3 (로드 테스트) 를 건너뛰지 않는다
   → 운영 증거 4.1과 연계되기 때문이다
```
