# 문제지 문서 품질 기준

기존 `problem-subject-*`, `problem-subject-capstone/`, 각 프로젝트 `problem/README.md`, 프로젝트 `README.md`를 검토했을 때 좋은 문서는 `무슨 문제를 푸는가`와 `어디서 시작하고 어떻게 검증하는가`를 빠르게 보여 주었습니다. 반대로 약한 문서는 분류 이유와 제외 범위가 흐려져 정답 요약처럼 읽히는 경향이 있었습니다.

같은 체크리스트를 `problem-subject-capstone`에도 그대로 적용합니다. 다만 capstone 문서는 개별 문제를 나열하는 데서 끝나지 않고, 무엇을 통합해서 다시 풀게 되는지가 함께 드러나야 합니다.

아래 체크리스트를 현재 워크스페이스의 공용 기준으로 둡니다.

- 문제 목표가 한두 문장으로 분명한가.
- 시작 위치가 바로 보이는가.
  최소한 `problem/README.md` 또는 프로젝트 `README.md` 진입 경로가 즉시 드러나야 한다.
- 핵심 요구사항이 계약 형태로 읽히는가.
  구현 해설보다 `무엇이 통과 조건인가`가 먼저 보여야 한다.
- 검증 시작점이 바로 보이는가.
  테스트, 실행, 재현 경로 중 무엇을 먼저 보면 되는지 숨기지 않는다.
- 왜 essential 또는 elective인지 짧게 설명하는가.
  중요도 평가가 아니라 `서버 공통 필수인가 아닌가`를 분명히 적는다.
- 일부러 제외한 범위가 드러나는가.
  문제 경계 문서여야 하므로 이번 과제가 안 푸는 것도 함께 보여 준다.
- 정답 요약이 아니라 문제 경계 문서로 읽히는가.
  답안 구현, 해설, blog, notion은 스포일러 경계 뒤로 보낸다.
