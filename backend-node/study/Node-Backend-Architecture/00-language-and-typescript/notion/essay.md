# TypeScript로 백엔드를 시작하기 전에 해둬야 할 일

## 왜 이 과제가 첫 번째인가

백엔드 프로젝트를 시작하면 대부분 Express나 NestJS 같은 프레임워크부터 만진다. 그런데 막상 코드를 짜다 보면 언어 자체에서 막히는 순간이 생각보다 자주 온다. `async/await`의 에러 전파가 어떻게 되는지, `type`과 `interface`는 어떤 상황에서 쓰는 게 맞는지, 배열을 변환할 때 `map`과 `filter`를 어디서 체이닝하는 게 좋은지 — 이런 것들이 프레임워크 학습 중간에 튀어나오면 두 가지를 동시에 공부해야 하는 상황이 된다.

그래서 이 트랙은 프레임워크에 들어가기 전에, TypeScript의 핵심 문법과 비동기 패턴을 한 번 정리하고 가는 과제를 맨 앞에 두었다. 목표는 "언어가 병목이 되지 않는 상태"를 만드는 것이다.

## 과제를 설계할 때 고민한 것

처음에는 간단한 알고리즘 문제를 몇 개 풀면 될 거라고 생각했다. 하지만 백엔드 맥락에서 도움이 되려면, 나중에 실제로 반복할 패턴을 미리 연습하는 편이 낫다고 판단했다.

그래서 "도서 카탈로그"라는 도메인을 골랐다. 이 도메인은 이후의 REST API, CRUD, 인증 과제에서 계속 이어지기 때문에, 여기서 만든 타입 구조와 습관이 자연스럽게 다음 단계로 연결된다.

구체적으로 이런 기능을 넣었다:

- **태그 정규화**: 문자열 배열을 slug 형태로 변환하고 중복을 제거한다. `map`, `filter`, `Set`을 조합하는 작은 변환 파이프라인을 직접 만들어 보는 경험이다.
- **입력 → 내부 구조 변환**: `BookDraft`라는 외부 입력 타입을 `NormalizedBook`이라는 깨끗한 내부 구조로 바꾼다. 이건 나중에 DTO → Entity 변환의 원형이 된다.
- **비동기 병렬 호출과 부분 실패 처리**: inventory 서비스에 재고를 물어보는데, 일부 아이템이 실패해도 전체가 중단되면 안 된다. `Promise.all` 안에서 개별 `try/catch`로 실패를 흡수하는 패턴을 여기서 처음 연습한다.
- **CLI 인자 파싱과 종료 코드**: 마지막으로, 이 모든 걸 CLI로 묶어서 실행 가능한 프로그램으로 만든다. `process.argv`를 직접 파싱하고, 성공이면 0, 실패면 1을 반환하는 Unix 관례를 따른다.

## BookDraft와 NormalizedBook을 분리한 이유

처음에는 하나의 `Book` 타입으로 충분하다고 생각할 수 있다. 그런데 실제로 코드를 짜 보면, 외부에서 들어오는 데이터와 내부에서 믿고 쓰는 데이터는 성격이 다르다.

`BookDraft`는 사용자 입력이다. title 앞뒤에 공백이 있을 수 있고, tags에 중복이 들어올 수 있고, description이 없을 수도 있다. 반면 `NormalizedBook`은 시스템 내부에서 다른 함수들이 안심하고 쓸 수 있는 구조다. slug은 이미 만들어져 있고, tags는 정규화되어 있고, summary는 항상 존재한다.

이 분리는 사소해 보이지만, 뒤에서 DTO와 Entity를 분리하는 패턴의 출발점이 된다. `03-rest-api-foundations`에서 Express나 NestJS의 DTO 검증을 붙일 때, "입력과 내부 구조는 구분해야 한다"는 감각이 이미 잡혀 있으면 훨씬 수월하다.

## 비동기 에러 처리에서 배운 것

`fetchInventorySnapshot`은 slug 목록을 받아서 외부 서비스에 재고를 물어보는 함수다. 핵심은 **일부 슬러그가 실패해도 나머지 결과는 정상적으로 돌려줘야 한다**는 것이다.

처음에는 `Promise.all`이 하나라도 reject되면 전체가 실패한다는 점을 잊기 쉽다. 그래서 각 항목을 개별적으로 `try/catch`로 감싸고, 실패한 항목은 `inStock: null`과 `error` 메시지를 담아서 돌려주는 구조로 만들었다.

이 패턴은 나중에 외부 API 호출, DB 쿼리 배치 처리 등에서 계속 반복된다. "전체가 깨지는 것"과 "부분 실패를 허용하는 것"은 설계가 완전히 다르고, 이 과제에서 그 차이를 처음으로 체감하게 된다.

## CLI를 붙인 이유

비즈니스 로직만 만들고 끝내는 게 아니라 CLI까지 붙인 건, **"실행 가능한 프로그램을 만드는 감각"**을 일찍 잡기 위해서다.

`cli.ts`는 `process.argv`에서 `--title`, `--author`, `--year`, `--tags` 플래그를 직접 파싱한다. 라이브러리 없이 `readFlag`라는 작은 헬퍼로 처리하는데, 이 정도 규모에서는 외부 의존성을 추가하는 것보다 직접 만드는 편이 오히려 구조를 이해하기 좋다.

실행하면 이렇게 된다:

```bash
pnpm start -- --title "Node Patterns" --author "Alice" --year 2024 --tags "Node, Architecture"
```

정상이면 사람이 읽기 좋은 카드 형식으로 출력하고 종료 코드 0을 반환한다. 필수 인자가 빠지면 에러 메시지를 `stderr`에 쓰고 종료 코드 1을 반환한다. 이 구분은 Unix 관례이고, 나중에 스크립트와 CI에서 실패를 감지하는 기본 수단이 된다.

## 테스트 전략

Vitest를 테스트 러너로 선택했다. Jest와 API가 거의 같으면서 속도가 빠르고, TypeScript를 별도 변환 없이 바로 실행할 수 있다.

테스트는 크게 두 영역으로 나눴다:

1. **순수 함수 테스트**: `normalizeTags`, `toNormalizedBook`, `formatBookCard`는 입력과 출력이 명확한 순수 함수라서 단위 테스트로 커버한다.
2. **비동기 + 에러 경계 테스트**: `fetchInventorySnapshot`은 `vi.fn()`으로 모킹한 `InventoryClient`를 넣어서, 정상 응답과 에러 응답이 섞인 시나리오를 테스트한다.
3. **CLI 통합 테스트**: `runCli`에 가짜 `stdout`과 `stderr`를 넣고, 정상 입력과 불완전한 입력에 대해 종료 코드와 출력 내용을 확인한다.

## 이 과제의 위치와 다음 단계

이 과제를 마치면 TypeScript의 타입 설계, 배열 변환, 비동기 에러 처리, CLI 인자 파싱을 한 번씩 직접 해본 상태가 된다. 다음 과제인 `01-node-runtime-and-tooling`에서는 Node.js 런타임의 파일 시스템, 스트림, 환경 변수 같은 쪽으로 넘어간다.

여기서 만든 `BookDraft → NormalizedBook` 변환 패턴은 이후 `03-rest-api-foundations`에서 DTO 검증으로, `06-persistence-and-repositories`에서 Entity 매핑으로 자연스럽게 확장된다. 그래서 이 과제는 단순한 언어 연습이 아니라, 전체 트랙의 설계 언어를 세팅하는 역할을 한다.
