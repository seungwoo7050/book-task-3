# Node.js 런타임을 직접 만져보는 첫 번째 도구

## 왜 런타임부터 다루는가

이전 과제에서 TypeScript의 타입과 비동기 패턴을 잡았다면, 이번에는 그 코드가 실제로 돌아가는 환경인 Node.js 런타임을 직접 다룬다. `fs`로 파일을 읽고, `path`로 경로를 처리하고, `process.env`로 설정을 받고, `readline`로 스트림을 다루는 — 이런 것들은 프레임워크에 들어가면 추상화 뒤에 숨지만, 그 전에 한 번은 직접 써봐야 나중에 문제가 생겼을 때 추상화 아래를 들여다볼 수 있다.

그래서 이 과제는 "NDJSON 요청 로그를 읽어서 요약하는 CLI"를 만든다. 작은 도구지만, Node.js 런타임의 핵심 API를 대부분 건드리게 된다.

## NDJSON이라는 포맷을 선택한 이유

로그 파일 형식으로 NDJSON(Newline Delimited JSON)을 골랐다. 이건 한 줄에 하나의 JSON 객체가 들어가는 형식이다:

```json
{"timestamp":"2026-03-07T09:00:00.000Z","userId":"u1","route":"/books","status":200}
{"timestamp":"2026-03-07T09:00:10.000Z","userId":"u2","route":"/books","status":200}
```

이 형식의 장점은 **한 줄씩 읽으면서 바로 파싱할 수 있다**는 것이다. 일반 JSON 배열이면 파일 전체를 메모리에 올려야 파싱이 되지만, NDJSON은 `readline`으로 한 줄씩 처리할 수 있다. 로그 파일이 수 GB가 되어도 메모리 사용량은 거의 일정하다.

실무에서도 로그 파이프라인, ETL 스크립트, 데이터 처리 도구에서 NDJSON을 자주 만나게 되는데, 이 과제에서 그 처리 패턴을 처음 연습한다.

## 스트리밍으로 파일을 읽는 구조

`readRequestLog` 함수가 파일 읽기를 담당한다. 구조는 이렇다:

1. `path.resolve`로 상대 경로를 절대 경로로 변환한다
2. `fs.stat`으로 파일 존재 여부를 먼저 확인한다
3. `createReadStream`으로 파일을 열고
4. `readline.createInterface`로 줄 단위 읽기를 설정한 뒤
5. `for await...of`로 한 줄씩 파싱한다

여기서 중요한 결정이 하나 있었다. JSON 파싱이 실패하면 **몇 번째 줄에서 실패했는지**를 에러 메시지에 포함시켰다. `lineNumber`를 직접 카운팅해서 `Invalid JSON at line 3: Unexpected token` 같은 메시지를 만든다. 이건 사소해 보이지만, 실제로 깨진 로그 파일을 디버깅할 때 "어디가 문제인지" 바로 알 수 있느냐 없느냐의 차이는 크다.

## 환경 변수로 출력 형식을 바꾸는 이유

이 CLI는 `REPORT_FORMAT` 환경 변수로 출력 형식을 선택한다. `text`면 사람이 읽기 좋은 형태, `json`이면 다른 프로그램이 파싱하기 좋은 형태로 출력한다.

```bash
# 사람이 읽는 용도
pnpm start -- ../problem/data/request-log.ndjson

# 다른 도구와 연결하는 용도
REPORT_FORMAT=json pnpm start -- ../problem/data/request-log.ndjson
```

환경 변수를 사용한 건 두 가지 이유에서다:

첫째, CLI 플래그보다 **컨텍스트 설정**에 가깝기 때문이다. 파일 경로는 "이번에 뭘 처리할지"를 결정하는 인자고, 출력 형식은 "이 도구를 어떤 맥락에서 쓰는지"를 결정하는 설정이다. 전자는 매번 바뀌고 후자는 보통 고정이라서, 환경 변수로 분리하는 게 자연스럽다.

둘째, 이후 과제에서 `process.env`를 설정과 비밀 관리에 쓰는 패턴이 계속 나온다. `PORT`, `DATABASE_URL`, `JWT_SECRET` 같은 환경 변수를 다루기 전에, 간단한 `REPORT_FORMAT`으로 먼저 감을 잡는 것이다.

## 요약 로직을 순수 함수로 분리한 판단

`summarizeRequests` 함수는 레코드 배열을 받아서 통계를 계산한다. 경로별 호출 수(`perRoute`), 고유 사용자 수(`uniqueUsers`), 에러 수(`errorCount`), 총 요청 수(`totalRequests`)를 한 번 순회로 뽑아낸다.

이 함수를 파일 읽기와 분리한 건, **테스트를 쉽게 만들기 위해서**다. `summarizeRequests`는 순수 함수라서 파일 시스템 의존 없이 테스트할 수 있다. 반면 `readRequestLog`는 실제 파일에 접근하기 때문에 fixture 파일이 필요하다.

이 패턴은 나중에 controller와 service를 분리하는 것, repository를 추상화하는 것과 같은 맥락이다. "외부 의존이 있는 코드"와 "순수 계산 코드"를 분리하면 테스트 가능성과 재사용성이 동시에 올라간다.

## CLI 구조: stdout, stderr, 종료 코드

CLI의 진입점인 `runCli`는 이전 과제의 패턴을 그대로 따르되, 비동기라는 차이가 있다:

- `stdout`과 `stderr`를 직접 주입받아서, 테스트에서 `vi.fn()`으로 바꿔 넣을 수 있다
- `process.env`도 직접 주입받아서, 테스트에서 특정 환경 변수 조합을 시뮬레이션할 수 있다
- 성공이면 `0`, 실패면 `1`을 반환한다

`require.main === module` 가드로 직접 실행과 import를 구분하는 것도 이전 과제와 동일하다. 이 관례는 이후 과제에서 계속 유지된다.

## 테스트에서 확인한 것

테스트는 크게 두 영역이다:

1. **리포트 로직**: 실제 fixture 파일(`problem/data/request-log.ndjson`)을 읽고, 레코드 수, 유니크 유저 수, 에러 수, 경로별 통계가 맞는지 확인한다.
2. **CLI 동작**: `REPORT_FORMAT=json`으로 실행하면 JSON 출력이 나오는지, 잘못된 형식(`yaml`)을 넣으면 에러 메시지와 함께 실패하는지 확인한다.

fixture 파일에는 5개의 레코드가 들어 있고, 3명의 사용자가 `/books`, `/books/1`, `/health` 경로에 접근하며, 404와 500 에러가 하나씩 포함되어 있다. 작지만 모든 분기를 커버하기에 충분하다.

## 이 과제의 위치와 다음 단계

이 과제를 마치면 Node.js의 파일 시스템, 스트림, 경로 처리, 환경 변수를 모두 한 번씩 써본 상태가 된다. 다음 과제인 `02-http-and-api-basics`에서는 이 런타임 위에서 HTTP 서버를 직접 만드는 쪽으로 넘어간다.

여기서 익힌 스트리밍 패턴은 나중에 대용량 응답 처리, 파일 업로드, 로그 파이프라인에서 다시 만나게 된다. 그리고 환경 변수로 동작을 제어하는 패턴은 `08-production-readiness`의 config loader까지 이어지는 기본 습관이 된다.
