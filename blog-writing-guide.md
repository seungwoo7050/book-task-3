# blog-writing-guide

이 문서는 `/Users/woopinbell/work/book-task-3/` 워크스페이스에서 프로젝트 개발 과정을 `blog/` 시리즈로 남길 때 따를 공용 기준을 정리한다.

이 가이드가 다루는 것은 단순한 문체 보정이 아니다.  
`blog/`를 어떤 장르로 쓸지, 어떤 단위에 적용할지, 본문을 어떤 형식으로 쓸지, 코드와 CLI를 어디까지 남길지까지 고정하는 문서다.

핵심 원칙은 아래 다섯 가지다.

- 적용 단위는 `독립 프로젝트 단위`다.
- `blog/`의 기본 장르는 `탐색형 학습 로그`다.
- 본문은 `날짜/세션 기반 chronological timeline`을 기본으로 쓴다.
- `핵심 소스코드`는 본문 중간 inline으로 남긴다.
- `CLI`는 `정리된 전체` 형식으로 chronological하게 전부 기록한다.

## 문서 정체성

`blog/`는 다 알고 난 뒤 매끈하게 정리하는 회고문이 아니다.  
기본 장르는 `모르는 상태에서 알아가며 남기는 탐색형 학습 타임라인`이다.

이 가이드가 원하는 글은 아래 흐름을 숨기지 않는다.

- 무엇을 아직 모르고 있었는가
- 어떤 가설을 세웠는가
- 어디서 틀렸는가
- 무엇을 버렸는가
- 어떤 검증으로 확신을 얻었는가
- 다음에 무엇이 남았는가

즉, 글의 중심은 `정답 요약`이 아니라 `무지 -> 가설 -> 실패 -> 수정 -> 검증 -> 다음 질문`이다.

`읽히는 개발 서사`라는 표현은 계속 유효하다.  
다만 여기서 말하는 서사는 결과를 다 알고 난 뒤의 매끄러운 해설이 아니라, 당시 시점의 오해와 판단이 살아 있는 서사여야 한다.

명시적으로 금지하는 느낌도 있다.

- 처음부터 원인과 해답을 모두 알고 있었던 것처럼 쓰는 문장
- 실패와 탐색을 지운 뒤 결과만 남긴 회고문
- README를 길게 늘인 문서체

사용자가 흔히 `Opus처럼 더 읽히게 써라`고 말할 때 기대하는 방향은 여기에 가깝다.

- 흔한 AI 요약문보다 자연스럽게 읽히는 기술 글
- 딱딱한 보고서보다 장면 전환이 살아 있는 문장
- 결과 설명보다 판단의 축적이 보이는 서술

한 문장으로 줄이면 이렇다.

`문서처럼 정확해야 하지만, 읽을 때는 이미 다 알고 쓰는 해설문처럼 보이면 안 된다.`

## 시리즈 적용 단위

이 가이드의 기본 적용 단위는 `독립 프로젝트 단위`다.

여기서 `독립 프로젝트 단위`란 이름이 꼭 `lab`일 필요가 없는, 아래 조건을 만족하는 가장 작은 학습 프로젝트 디렉터리를 뜻한다.

- 자기 `README.md`가 있다
- 자기 검증 명령 또는 entrypoint가 있다
- 문제 범위와 구현 범위를 독립적으로 설명할 수 있다

기본 단위는 보통 아래와 같은 형태다.

- `blog/<track>/<project>/`
- `Foundations-CSAPP/datalab`
- `Systems-Programming/proxylab`
- `Operating-Systems-Internals/filesystem-mini-lab`
- `Programming-Languages-Foundations/parser-interpreter`

중요한 경계는 아래와 같다.

- 기본값은 `track 전체`가 아니다
- 기본값은 `repo 전체`가 아니다
- `lab`이라는 말은 기준 용어로 쓰지 않는다
- `track 전체`나 `repo 전체` 시리즈는 사용자가 명시적으로 요청한 경우에만 예외로 만든다

즉 이 가이드는 기본적으로 `cs-core/study` 전체를 한 시리즈로 쓰라는 문서가 아니다.  
기본값은 언제나 `하나의 독립 프로젝트`다.

## 문서 층 역할 구분

블로그 글을 쓰기 전에 다른 문서 층과 역할이 겹치지 않게 해야 한다.

- `README.md`: 짧은 공개 표면. 문제, 범위, 상태, 검증 진입점을 빠르게 보여 준다.
- `docs/`: 오래 남길 개념 설명, 참고 자료, 구조 설명을 둔다.
- `notion/`: 압축된 재구현, 재검증, 재학습 노트를 둔다.
- `notion-archive/`: 초안, 긴 기록, 폐기된 시도, 이전 판을 보존한다.
- `blog/`: `독립 프로젝트 단위`의 긴 chronological 학습 로그를 둔다.

즉 `blog/`는 `notion/`을 대체하지 않는다.

- `notion/`은 압축 rebuild note다
- `blog/`는 더 길고 더 chronological하며 더 불확실성이 드러나는 public learning log다

또 하나 분명히 적는다.

- `blog/`는 `track 요약문`이 아니다
- `blog/`는 `repo 전체 회고문`이 아니다
- `blog/`는 특정 독립 프로젝트의 처음부터 끝까지를 따라가는 시리즈다

## 본문 기본 형식

파일 구조의 번호 규칙과, 파일 안의 서술 형식은 분리해서 본다.

- 파일 구조는 `00`, `10`, `20`, `30` 같은 번호 체계를 쓴다
- 파일 내부 서술은 `날짜/세션 기반 chronological timeline`을 기본으로 쓴다

기본 형식은 아래에 가깝다.

````md
# 프로젝트 개발 타임라인

## 2026-03-12
### 09:30
- 목표:
- 진행:
- 이슈:
- 판단:
- 다음:

### 14:10
- 진행:
- 이슈:
- 원인 추정:
- 조치:
- 검증:

### 20:40
- 정리:
  - ...
````

정확한 시각 근거가 없으면 아래 fallback을 쓴다.

````md
# 프로젝트 개발 타임라인

## Day 3
### Session 2
- 목표:
- 진행:
- 이슈:
- 판단:
- 검증:
- 다음:
````

한 엔트리에서 사용할 필드는 아래를 기본 집합으로 본다.

- `목표`
- `진행`
- `이슈`
- `원인 추정`
- `판단`
- `조치`
- `검증`
- `다음`

이 필드들은 `선택형`이다.  
모든 엔트리에 전부 억지로 넣지 않는다. 중요한 건 템플릿을 다 채우는 것이 아니라, 당시 흐름이 끊기지 않게 남는 것이다.

## 날짜와 시각 규칙

정확한 날짜와 시각은 `근거가 있을 때만` 사용한다.

- 실제 작업 로그
- commit 시각
- note timestamp
- 저장된 terminal history
- notion/dev log의 시계열 기록

이런 근거가 없으면 날짜와 시각을 지어내지 않는다.

- 날짜가 없으면 `Day 1`, `Day 2`
- 시각이 없으면 `Session 1`, `Session 2`
- 더 거칠게만 구분 가능하면 `Phase 1`, `Phase 2`

실시간 기록을 바탕으로 나중에 재구성하더라도 chronology는 유지해야 한다.  
즉, 순서를 바꿔 더 깔끔하게 보이게 만드는 것보다, 당시 흐름을 보존하는 쪽을 우선한다.

금지 예시는 아래와 같다.

- 실제 근거 없이 `2026-03-12 09:30` 같은 시각을 붙이는 것
- 나중에 이해한 순서대로 장면을 재배치해 실제 탐색 순서를 지우는 것

## 탐색형 문체

이 가이드는 `이미 다 아는 상태에서 쓰는 글`을 원하지 않는다.

반드시 남겨야 할 질문은 아래와 같다.

- 처음엔 무엇을 몰랐는가
- 처음 가설은 무엇이었는가
- 어디서 틀렸는가
- 무엇을 버렸는가
- 무엇이 확신의 계기였는가
- 이 시점에서 아직 무엇이 미해결인가

권장하는 시점 표지는 아래와 같다.

- `처음엔`
- `이 시점에서는`
- `당시에는`
- `처음 가설은`
- `나중에 보니`
- `이때까지는`

좋은 문장은 보통 이런 식이다.

- `처음엔 allocator가 split 로직에서만 꼬인다고 생각했는데, trace를 다시 보니 free list unlink 시점부터 불변식이 깨지고 있었다.`
- `이 시점에서는 cache miss를 줄이는 문제라고만 봤지만, 실제로는 diagonal conflict를 피하는 접근 순서가 핵심이었다.`
- `나중에 보니 signal 처리 자체보다, mask를 언제 풀고 언제 wait 상태로 들어가느냐가 race의 중심이었다.`

나쁜 문장은 보통 이런 식이다.

- `문제의 원인은 free list unlink였다. 이를 수정했다.`
- `signal race가 발생하므로 mask 순서를 바꿨다.`
- `cache miss를 줄이기 위해 block 전략을 사용했다.`

이런 문장은 정보는 맞아도, `당시 무엇을 모르고 있었는가`가 사라져 학습 로그처럼 읽히지 않는다.

## 핵심 소스코드 기록

핵심 소스코드는 반드시 본문 중간 inline으로 기록한다.

여기서 `핵심 소스코드`란 전체 구현이 아니라, 판단이 갈린 지점을 보여 주는 짧은 코드 조각을 뜻한다.

- 결정적이었던 함수
- 버그가 났던 분기
- 상태 전이가 드러나는 부분
- invariant를 강제하는 짧은 블록
- recovery, cache hit, signal discipline처럼 설계 핵심이 드러나는 부분

원칙은 아래와 같다.

- 전체 파일 복붙 금지
- 코드 로그 전용 별도 파일을 기본값으로 두지 않음
- 짧고 결정적인 조각만 본문에 넣음
- 코드 뒤에는 반드시 `왜 이 코드가 그 시점에 중요했는가`를 문장으로 적음

예:

```c
if (GET_SIZE(HDRP(next_bp)) > 0 && !GET_ALLOC(HDRP(next_bp))) {
    remove_free_block(next_bp);
    size += GET_SIZE(HDRP(next_bp));
}
```

이 코드를 넣었다면 바로 뒤에 이런 설명이 따라야 한다.

`처음엔 coalesce 자체가 잘못됐다고 생각했지만, 실제로는 merge 이전에 free list에서 기존 블록을 떼어 내지 않아 같은 블록이 두 번 남는 게 문제였다. 그래서 이 짧은 unlink 조각이 이 시점의 핵심 증거가 됐다.`

한 줄 규칙으로 줄이면 이렇다.

`코드는 핵심만 남기고, 그 코드가 왜 중요했는지는 문장으로 설명한다.`

## CLI 기록 원칙

CLI는 `핵심만` 남기는 대상이 아니다.  
실행한 명령은 `전체를 chronological하게` 남긴다.

다만 그대로 raw dump를 붙이지는 않는다. 형식은 `정리된 전체`다.

- 실제 실행한 명령은 빠짐없이 남긴다
- 순서는 실제 chronology를 따른다
- 핵심 출력은 verbatim으로 남긴다
- 반복적이거나 긴 출력은 요약할 수 있다
- raw terminal transcript 전체 덤프는 금지한다

기본 배치는 각 날짜/세션 엔트리 내부의 `실행 로그` 또는 `CLI` 블록이다.

예:

```bash
$ make clean
$ make
$ ./mdriver -f traces/short1-bal.rep
$ ./mdriver -f traces/short2-bal.rep
$ pytest -q
```

출력은 이렇게 다룬다.

- 최종 pass/fail, 수치, 상태 문자열 같은 검증 신호는 그대로 남긴다
- 같은 형식의 반복 출력은 `short trace 2개 통과`처럼 요약할 수 있다
- 디버깅 중 실제 판단에 영향을 준 에러 메시지는 verbatim으로 남긴다

한 줄 규칙은 이렇다.

`코드는 핵심만, CLI는 전체.`

## 번호 규칙

번호는 장식이 아니라 `순서와 삽입 가능성`을 위한 규칙이다.

- `00`은 고정적으로 시리즈 지도다
- `10`, `20`, `30`은 주제 챕터가 아니라 `연속된 타임라인 구간`이다
- 상한은 없다
- 중간 삽입이 필요하면 `15`, `25`, `35`를 쓴다
- 가능하면 기존 파일은 재번호하지 않는다

즉:

- `00`만 역할이 고정이다
- `10`부터 `60`이 고정 세트인 것은 아니다
- `90`이 끝이라는 규칙도 없다
- 필요하면 `100`, `110`, `120`까지 계속 간다

## 예전 구조는 보조 틀로만 사용

예전 문서형 구조였던 아래 틀은 완전히 금지하지는 않는다.

1. `당시 목표`
2. `실행`
3. `문제와 판단`
4. `검증 신호`
5. `다음 단계`

다만 이 틀은 이제 `기본 형식`이 아니다.  
필요할 때 참고 가능한 보조 틀일 뿐이다.

이유는 단순하다.

- 이 틀은 글을 빠르게 정리하는 데는 좋다
- 하지만 그대로 고정하면 실시간 탐색의 호흡보다 회고형 요약문으로 기울기 쉽다

따라서 기본값은 여전히 `날짜/세션 기반 타임라인`이다.

## 자료를 어떻게 글로 바꿀 것인가

원천 자료가 있다고 바로 좋은 학습 로그가 되지는 않는다.  
보통 아래 순서로 다시 써야 한다.

1. `notion-archive/`에서 실제 작업 순서와 당시 판단을 찾는다.
2. `notion/`에서 현재판 핵심 요약과 rebuild 관점을 확인한다.
3. 프로젝트 `README.md`, `problem/README.md`, `docs/`, 구현 README에서 사실과 검증 명령을 교차 확인한다.
4. 실제 chronology를 기준으로 날짜/세션 엔트리를 다시 세운다.
5. 각 엔트리 안에 핵심 코드와 전체 CLI를 끼워 넣는다.
6. 마지막으로 결과를 다 아는 해설문처럼 보이는 문장을 걷어 낸다.

좋은 변환 기준은 아래와 같다.

- 기록은 chronology를 유지한다
- 글은 당시 시점의 긴장감을 남긴다
- 코드는 판단이 갈린 지점만 남긴다
- CLI는 전체를 남기되 dump처럼 보이지 않게 정리한다
- 검증 전의 불확실성과 검증 후의 확신을 구분해서 적는다

## Canonical Example

아래 예시는 이 가이드가 기본값으로 기대하는 형식이다.

````md
# 프로젝트 개발 타임라인

## 2026-03-12
### 09:30
- 목표: `realloc`이 어디서 깨지는지부터 잡는다. 이 시점에서는 split 이후 payload copy가 문제라고 생각했다.
- 진행: `make test`를 다시 돌리고, 실패 trace만 따로 분리했다.
- 이슈: `binary2-bal.rep`에서 같은 block이 free list에 두 번 보였다.
- 원인 추정: 처음 가설은 copy 길이 계산 오류였는데, 로그를 다시 보니 free list unlink 순서가 더 수상했다.
- 판단: copy 로직을 먼저 고치기보다, `coalesce`와 `place` 앞뒤의 free list 변화를 먼저 확인해야 했다.

CLI:

```bash
$ make clean
$ make
$ ./mdriver -f traces/binary2-bal.rep
$ ./mdriver -f traces/short1-bal.rep
```

검증:

- `binary2-bal.rep`는 여전히 실패
- `short1-bal.rep`는 통과
- 즉 전체 allocator가 무너진 것이 아니라 특정 free-list 경로에서만 깨진다고 봤다

이 시점의 핵심 코드는 아래였다.

```c
remove_free_block(next_bp);
size += GET_SIZE(HDRP(next_bp));
```

처음엔 merge 계산식만 의심했지만, 실제로는 merge 전에 기존 free block을 리스트에서 떼어 내지 않아 중복 엔트리가 생기고 있었다. 이 짧은 조각이 중요한 이유는, 당시 추측을 버리고 문제를 `coalesce 이전 상태 정리`로 다시 보게 만든 증거였기 때문이다.

### 14:10
- 진행: `coalesce` 직전과 직후의 block 상태를 로그로 비교했다.
- 조치: `next_bp`와 `prev_bp`가 free일 때 unlink가 항상 먼저 일어나게 순서를 바꿨다.
- 검증: `binary2-bal.rep` 통과, `pytest -q` 전체 통과.
- 다음: 이제 `realloc` 최적화보다 trace별 utilization이 왜 낮은지 봐야 했다.

CLI:

```bash
$ ./mdriver -f traces/binary2-bal.rep
$ pytest -q
```

출력:

```text
5 passed
```

### 20:40
- 정리:
  - 오늘 핵심은 `realloc copy`가 아니라 `free list invariant`였다.
  - 처음 가설은 틀렸지만, 실패 trace를 분리한 덕분에 범위를 빨리 줄였다.
  - 다음 세션에서는 correctness가 아니라 utilization과 throughput을 따로 본다.
````
