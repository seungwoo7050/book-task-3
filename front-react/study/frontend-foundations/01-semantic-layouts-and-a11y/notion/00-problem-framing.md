# 문제 정의: 왜 시맨틱 레이아웃부터 시작했는가

## 출발점

프론트엔드 학습의 첫 단계를 정할 때, React 컴포넌트나 상태 관리부터 시작하는 게 보통이다. 하지만 이 경로는 의도적으로 다른 출발점을 택했다. "화면에 보이는 것"보다 "화면이 무엇을 말하는가"를 먼저 다루기로 한 것이다.

이유는 실무에서 반복적으로 마주치는 문제 때문이다. 디자인 시스템이 아무리 정교해도, 시맨틱 구조와 접근성이 빠지면 결국 스크린리더 사용자는 페이지를 탐색할 수 없고, 키보드만 쓰는 사용자는 포커스를 잃어버린다. 이걸 나중에 고치려면 마크업 구조 자체를 뜯어야 할 때가 많다.

## 왜 설정 셸인가

학습용 화면으로 마케팅 랜딩 페이지나 블로그를 택할 수도 있었다. 하지만 설정/대시보드 형태의 셸을 선택한 이유가 있다.

- **form 요소가 자연스럽게 등장한다.** label, help text, error text, fieldset 같은 접근성 패턴을 한 화면에서 모두 검증할 수 있다.
- **landmark가 분명하다.** header, nav, main, aside가 역할별로 깔끔하게 나뉘어 시맨틱 구조의 효과를 바로 체감할 수 있다.
- **반응형 전환이 의미 있다.** 단순히 한 컬럼으로 쌓이는 게 아니라, reading order가 바뀌었을 때 내용이 여전히 논리적인지 검증할 수 있는 구조다.

## 범위를 좁힌 이유

이 프로젝트에서 **일부러 하지 않은 것**들이 있다.

- 라우팅은 넣지 않았다. 페이지 전환이 아니라 한 화면의 의미 구조에 집중하고 싶었다.
- localStorage나 서버 저장도 없다. 폼 제출은 UI 피드백까지만 다루고, 데이터 흐름은 다음 프로젝트로 넘겼다.
- custom select나 toggle 위젯을 직접 만들지 않았다. 네이티브 요소의 접근성을 먼저 이해하고 나서, 커스텀 요소의 ARIA 패턴을 다루는 게 순서라고 판단했다.

이 제한 덕분에 이 프로젝트는 "의미 구조가 올바른가"와 "키보드로 모든 걸 할 수 있는가"라는 두 질문에만 집중할 수 있었다. 그리고 그 제한이 만드는 빈자리가 자연스럽게 다음 프로젝트(`02-dom-state-and-events`)의 출발점이 된다.
