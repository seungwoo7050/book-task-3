# 디버그 기록: 비동기에서 만난 함정들

## 오래된 응답이 최신 결과를 덮어쓰다

가장 먼저 만난 문제가 이것이다. 검색창에 "inc"를 타이핑하면 세 번의 요청이 나간다: "i", "in", "inc". mock service는 검색어 길이에 비례해서 지연을 넣기 때문에(`140 + query.search.length * 25`), "i" 요청이 가장 빨리 응답한다.

AbortController 없이는, "inc" 결과가 화면에 먼저 뜨고, 나중에 "i" 결과가 도착해서 덮어쓰는 일이 벌어진다. 사용자는 "inc"를 검색했는데 화면에는 "i" 결과가 보이는 셈이다.

AbortController로 이전 요청을 취소하고, requestTracker로 오래된 응답을 무시하니 이 문제가 사라졌다. **비동기 UI에서는 "응답 순서 = 요청 순서"라는 가정이 성립하지 않는다**는 교훈이었다.

## AbortError를 에러 UI로 표시한 실수

AbortController로 요청을 취소하면 `AbortError`가 발생한다. 처음에는 이 에러도 catch해서 에러 UI를 보여 줬다. "요청이 취소되었습니다"라는 에러 메시지와 Retry 버튼이 뜨는 아이러니한 상황이 벌어졌다.

AbortError는 사용자의 의도적인 취소(새 검색어 입력)의 결과이므로, **에러가 아니라 정상 흐름**이다. catch 블록에서 `(error as Error).name === "AbortError"` 체크를 추가해서, abort된 요청은 조용히 무시하게 수정했다.

## 목록 로드 후 상세 자동 로드의 타이밍

목록이 로드되면 첫 번째 항목이 자동으로 선택되고, 그 항목의 상세를 로드해야 한다. 처음에는 `loadList()` 안에서 `loadDetail()`을 호출했는데, 두 요청이 동시에 진행되면서 상태가 꼬이는 경우가 생겼다.

해결은 `loadList()` 내부에서, 목록 응답을 받은 후에 `selectedId`를 결정하고, 그 다음에 `await loadDetail(selectedId)`를 호출하는 순차 실행이었다. 병렬이 아니라 순차로 해야 하는 이유는, 목록의 결과가 있어야 어떤 항목의 상세를 로드할지 결정할 수 있기 때문이다.

## 필터 변경 시 선택된 항목이 사라지는 문제

이전 프로젝트의 `reconcileSelection()`과 비슷한 문제가 여기서도 나타났다. 카테고리를 "Runbooks"로 바꾸면, 이전에 선택한 "Escalation policy handbook"(policy)이 목록에서 사라진다. 선택된 ID는 남아 있지만 목록에 없으니 디테일 패널이 어긋난다.

`loadList()` 응답 처리에서, 새 목록에 현재 `selectedId`가 있으면 유지하고, 없으면 첫 번째 항목으로 자동 전환하는 로직을 넣었다.

## mock service의 지연 시간 조정

처음에 mock service의 지연을 500ms로 설정했더니, E2E 테스트가 너무 느려졌다. 반면 지연을 0으로 만들면 loading 상태가 화면에 아예 나타나지 않아서 loading UI를 테스트할 수 없었다.

결국 목록은 `140 + search.length * 25`ms, 상세는 `90ms`로 설정했다. 사람이 눈으로 볼 때는 loading 상태가 잠깐 보이면서 UX를 확인할 수 있고, 테스트에서는 적당히 빨라서 타임아웃이 걸리지 않는 균형점이다.
