# 회고: 프레임워크 없이 비동기 UI를 만든다는 것

## foundations 트랙의 마지막이라는 위치

세 개의 프로젝트를 끝내고 보니, 의도적으로 설계된 학습 경로가 보인다.

- **01**: 정적 구조. HTML이 "무엇을 말하는가"
- **02**: 동적 상태. 이벤트가 "무엇을 바꾸는가"
- **03**: 비동기 흐름. 네트워크가 "무엇을 불확실하게 만드는가"

이 세 프로젝트를 거치면서, 프레임워크 없이 UI를 만드는 것이 **가능하지만 점점 더 힘들어진다**는 걸 직접 체감했다. 특히 비동기 상태가 추가되면 포커스 관리, 상태 정합성, 렌더링 최적화가 급격히 복잡해진다.

## 가장 큰 수확

**"비동기 상태는 boolean이 아니라 enum이다."** 이 한 문장이 이 프로젝트의 핵심이다.

`isLoading: boolean`으로는 "요청 성공했지만 결과 없음"과 "요청 자체가 실패함"을 구분할 수 없다. `AsyncState`를 5가지로 나누고, 각 상태에 다른 UI를 매핑하니, 사용자에게 "지금 무슨 일이 일어나고 있는지"를 훨씬 명확하게 전달할 수 있었다.

또 하나는 **AbortController와 request tracker의 이중 안전장치 패턴**이다. 물리적 취소(abort)와 논리적 무시(token check)를 함께 쓰면, 네트워크 응답 순서에 관계없이 화면에는 항상 최신 요청의 결과만 반영된다. 이 패턴은 React의 `useEffect` cleanup이나 TanStack Query 같은 라이브러리가 내부적으로 하는 일이기도 하다.

## mock service라는 설계 패턴

mock service를 `ExplorerService` 인터페이스로 추상화한 것은 테스트에서 큰 도움이 됐다.

- 단위 테스트에서는 지연 없는 mock을 주입해서 빠르게 검증
- E2E 테스트에서는 실제 mock service(지연 포함)를 사용해서 실제와 비슷한 UX 검증
- "Simulate failure" 플래그로 에러 시나리오를 확정적으로 재현

이 패턴은 실무에서도 MSW(Mock Service Worker) 같은 도구와 같은 원리다. **서비스 경계에서 인터페이스를 끊어 놓으면, 구현을 바꿔 끼울 수 있다.**

## 아쉬운 점과 넘기는 질문

- **debounce가 없다.** 글자마다 요청이 나가는데, 실무에서는 debounce를 걸어서 타이핑이 멈춘 뒤에 요청을 보내는 게 일반적이다.
- **pagination이 없다.** 5개의 하드코딩된 항목이라 필요 없었지만, 실제로는 스크롤이나 "Load more" 패턴이 필요하다.
- **optimistic update가 없다.** 이 앱은 읽기 전용이라 mutation이 없다. 쓰기 작업이 있으면 optimistic update, rollback, retry가 추가로 필요하다.

이 질문들은 더 이상 `frontend-foundations` 트랙에서 다루지 않는다. 대신 `react-internals` 트랙이 렌더링 최적화를, `frontend-portfolio` 트랙이 제품 수준의 비동기 패턴을 이어받는다.

## 프레임워크의 필요성을 체감하다

세 프로젝트를 통해 가장 반복적으로 느낀 불편함은:

1. `innerHTML` 교체 후 포커스가 사라지는 문제
2. 상태와 DOM의 동기화를 수동으로 관리해야 하는 부담
3. 비동기 흐름의 cleanup을 매번 직접 작성해야 하는 번거로움

이것들이 바로 React가 해결하려는 문제들이다. Virtual DOM은 (1)을 해결하고, 선언적 렌더링은 (2)를 해결하고, useEffect cleanup은 (3)을 해결한다. 이 체감 없이 React를 쓰면, 왜 그런 API가 존재하는지 설명하기 어렵다.
