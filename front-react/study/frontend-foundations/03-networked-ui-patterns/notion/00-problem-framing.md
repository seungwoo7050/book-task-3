# 문제 정의: 서버 없이 서버 같은 UI를 만들다

## 비동기라는 빈자리

`02-dom-state-and-events`에서는 상태 동기화와 이벤트 위임을 다뤘지만, 모든 데이터가 하드코딩되어 있었다. 검색하면 즉시 결과가 나오고, 실패란 없었다. 실무 프론트엔드에서 가장 많은 문제가 생기는 지점 — 네트워크 지연, 실패, 재시도, 취소 — 을 전혀 경험하지 못한 것이다.

이 프로젝트의 출발 질문은 단순했다: **"서버가 느리거나 실패할 때, UI는 무엇을 보여 줘야 하는가?"**

## 왜 directory explorer인가

학습용 비동기 앱으로 채팅이나 SNS 피드를 택할 수도 있었지만, **운영 도구의 내부 문서 디렉토리**를 선택했다. 이유는:

- **목록 + 상세 패턴**이 자연스럽다. 왼쪽에 검색 결과 목록, 오른쪽에 선택한 문서의 상세 내용. 이 구조에서 목록 로딩과 상세 로딩이 독립적으로 발생한다.
- **검색과 필터가 요청을 트리거한다.** 글자를 타이핑할 때마다 새 요청이 발생하므로, 이전 요청을 취소해야 하는 상황이 자연스럽게 만들어진다.
- **실패와 재시도를 시뮬레이션할수 있어야 한다.** "Simulate failure" 버튼을 두면, 다음 요청이 반드시 실패하게 만들 수 있어서 에러 UI와 retry 흐름을 직접 테스트할 수 있다.

## 핵심 질문

1. **비동기 상태를 어떻게 분류하는가?** 단순히 loading/loaded가 아니라, idle/loading/success/empty/error로 나눠야 UX가 달라지는지.
2. **빠르게 연속된 요청에서 어떤 응답이 화면에 반영되어야 하는가?** (race condition 방지)
3. **진행 중인 요청을 어떻게 취소하는가?** (AbortController)
4. **실패 후 재시도는 어떤 UX로 제공하는가?**

## mock service라는 판단

실제 서버를 띄우지 않고 `service.ts`에 mock service를 만들었다. `wait()` 함수로 인위적인 지연을 넣고, AbortSignal을 존중하도록 구현했다. 이 접근의 장점은:

- 네트워크 환경에 의존하지 않고 비동기 흐름을 정확히 재현할 수 있다
- `simulateFailure` 플래그로 실패 시나리오를 확정적으로 테스트할 수 있다
- 테스트에서 mock service를 주입할 수 있어서 E2E와 단위 테스트 모두 예측 가능하다
