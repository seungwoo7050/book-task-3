# 문제 정의: 화면이 "살아 있으려면" 무엇이 필요한가

## 이전 프로젝트에서 남은 질문

`01-semantic-layouts-and-a11y`에서는 올바른 마크업 구조와 접근성 패턴을 다뤘다. 하지만 그 프로젝트의 UI는 본질적으로 정적이었다. 폼을 제출하면 상태 메시지가 바뀌긴 하지만, 목록이 필터링되거나, 데이터가 편집되거나, 새로고침 뒤에 작업 맥락이 복원되는 일은 없었다.

실무에서 마주치는 UI는 다르다. 사용자가 필터를 바꾸면 목록이 다시 그려지고, 어떤 행을 선택하면 디테일 패널이 갱신되고, 제목을 수정하면 그 변경이 저장된다. 이 모든 것이 "상태(state)"와 "이벤트(event)"의 조합이다.

## 왜 task board인가

학습용 화면으로 to-do 리스트를 택할 수도 있었지만, 의도적으로 **task/workspace board** 형태를 선택했다.

- **필터, 정렬, 검색이 자연스럽게 등장한다.** 단순한 추가/삭제가 아니라, 같은 데이터를 다양한 관점으로 보여 주는 UI를 연습할 수 있다.
- **선택(selection)과 편집(inline edit)이 동시에 필요하다.** row를 선택하면 디테일이 바뀌고, 편집 모드에 들어가면 input이 나타나는 — 두 종류의 UI 상태가 공존한다.
- **상태 저장의 층이 나뉜다.** 검색/필터는 URL에, 편집 결과는 localStorage에, 편집 모드는 메모리에만. 이 분리를 명시적으로 연습할 수 있는 화면이다.

## 핵심 질문

이 프로젝트에서 답하고 싶었던 질문은 세 가지다.

1. **이벤트가 발생했을 때, 어떤 상태를 바꾸고 어떻게 화면을 다시 그리는가?** — React 없이, 순수 DOM 조작으로 이 사이클을 직접 돌려 보는 것.
2. **상태는 어디에 저장하고, 그 경계를 어떻게 나누는가?** — URL에 넣을 것과 localStorage에 넣을 것, 메모리에만 둘 것의 구분.
3. **이벤트 핸들러가 늘어날 때 어떻게 관리하는가?** — 개별 listener 대신 delegation으로 묶을 수 있는가.

## 범위 제한

- 서버 통신은 없다. 데이터는 하드코딩된 4개의 태스크 항목이다.
- 라우팅 라이브러리를 쓰지 않는다. `history.replaceState()`로 URL만 동기화한다.
- 컴포넌트 추상화를 하지 않는다. 전체 마크업을 하나의 함수에서 반환하는 구조를 유지한다.

이 제한이 불편하게 느껴지는 지점이 곧 `03-networked-ui-patterns`의 출발점이 된다. 서버 없이는 비동기 상태, 로딩, 에러, 재시도를 다룰 수 없기 때문이다.
