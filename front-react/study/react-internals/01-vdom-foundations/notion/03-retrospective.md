# 회고: VDOM이라는 추상화의 힘과 한계

## "UI를 데이터로 표현한다"는 것의 의미

이 프로젝트를 하기 전에는 VDOM을 "DOM 복사본을 메모리에 만들어 두는 최적화 기법" 정도로 이해했다. 하지만 직접 구현해 보니 핵심은 최적화가 아니라 **추상화**다.

VNode는 "UI가 어때야 하는지"를 선언적으로 표현한 데이터다. `createElement("div", { id: "root" }, ...)` 호출이 브라우저 DOM을 만드는 게 아니라, "이런 div가 있어야 한다"는 명세를 plain object로 만든다. 이 명세가 있어야 "이전 명세와 새 명세를 비교해서 최소한의 변경만 적용한다"는 reconciliation이 가능해진다.

foundations 트랙에서 `innerHTML` 교체의 한계를 체감했는데, 그 한계는 정확히 "이전 상태를 데이터로 기억하지 않았기 때문"이다. VDOM은 이전 상태를 기억하는 데이터 구조를 제공한다.

## updateDom의 네 단계가 가르쳐 준 것

prop 업데이트를 네 단계로 나누는 것이 과도한 설계처럼 보일 수 있지만, 실제로 이벤트 핸들러 중복 문제를 겪고 나서 이 구조의 가치를 체감했다.

**순서가 정해져 있으면 버그를 추론하기 쉽다.** "핸들러가 두 번 호출된다"는 증상이 있을 때, "이전 이벤트 제거가 새 이벤트 추가보다 먼저 실행되는가?"라는 질문으로 바로 좁혀질 수 있다. 순서가 없으면 가능한 원인이 너무 많다.

## 이 단계의 한계, 그리고 그것이 만드는 다음 질문

이 VDOM 구현은 정확히 세 가지를 하지 않는다:

1. **이전 트리와 새 트리를 비교하지 않는다.** 매번 전체를 처음부터 만든다. 작은 변경에도 전체 DOM을 재생성한다.
2. **렌더링을 중단하거나 분할하지 않는다.** 깊은 트리는 동기 재귀로 메인 스레드를 오래 점유한다.
3. **상태와 effect가 없다.** 컴포넌트가 자신의 상태를 가질 수 없고, 마운트/언마운트 시 정리 작업을 할 수 없다.

각각의 한계가 다음 프로젝트의 동기다:
- (1) → `02-render-pipeline`: diff/patch, keyed reconciliation
- (2) → `02-render-pipeline`: render phase / commit phase 분리
- (3) → `03-hooks-and-events`: useState, useEffect, delegated events

## 테스트 설계에 대한 소감

27개의 테스트를 작성하면서, 테스트가 "올바른 동작 확인" 기능뿐 아니라 **"이 함수가 무엇을 보장하는가"를 문서로 남기는** 역할도 한다는 걸 느꼈다.

`createElement`의 테스트만 봐도, 이 함수의 계약이 무엇인지 바로 읽힌다: null props를 빈 객체로 처리하고, primitive child를 TEXT_ELEMENT로 감싸고, VNode child는 그대로 유지하고, 깊은 중첩을 지원한다.
