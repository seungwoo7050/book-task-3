# 문제 정의: React를 쓰기 전에 React를 만들어 보다

## 왜 이 트랙이 필요한가

`frontend-foundations` 트랙을 끝내면서 가장 크게 느낀 것은 "프레임워크가 자동으로 해 줘야 하는 일들"이 있다는 것이었다. innerHTML 교체 후 포커스가 사라지고, 상태와 DOM의 동기화를 수동으로 관리해야 하고, 비동기 cleanup을 직접 작성해야 했다.

React가 이 문제들을 어떻게 해결하는지 API 수준에서는 알고 있다. `useState`, `useEffect`, JSX. 하지만 **내부에서 어떤 일이 벌어지는지** 설명할 수 없으면, 최적화가 왜 필요한지, 왜 특정 패턴이 잘못된지 말할 수 없다.

이 트랙은 React의 핵심 내부를 **가장 작은 단위부터 직접 구현**하면서, "왜 그렇게 만들었는가"를 체감하려는 시도다.

## 왜 VDOM부터 시작하는가

React의 모든 것은 Virtual DOM에서 시작한다. JSX가 함수 호출로 변환되고, 그 함수 호출이 plain object 트리를 만들고, 그 트리가 실제 DOM으로 물질화된다.

이 과정을 건너뛰고 reconciliation이나 hooks를 먼저 다루면, "무엇과 무엇을 비교하는가"의 기반이 없다. VDOM이 먼저 있어야 "이전 트리와 새 트리의 차이를 계산한다"는 말이 구체적인 의미를 갖는다.

## 이 프로젝트에서 답해야 하는 질문

1. **VNode는 왜 `{ type, props }` 형태인가?** — React가 UI를 데이터로 표현하는 방식의 최소 구조
2. **문자열과 숫자 child를 왜 `TEXT_ELEMENT`로 감싸는가?** — 렌더러가 모든 자식을 동일한 인터페이스로 순회하기 위해
3. **실제 DOM 생성과 prop 반영은 어떤 순서로 처리하는가?** — createDom → updateDom → children 재귀 → appendChild
4. **동기 재귀 render의 한계는 무엇인가?** — 큰 트리에서 메인 스레드 점유, 변경 집합 계산 없음

## 레거시 자산과의 관계

이 프로젝트는 기존 학습 자료(`problem/original/`)에서 출발하되, 새로운 TypeScript 구현(`ts/`)으로 재작성되었다. 원본의 아이디어를 가져오면서도, 타입 정의와 테스트 커버리지를 이 저장소의 기준에 맞게 새로 만들었다.

## 범위 제한

- **reconciliation은 없다.** 이전 트리와 새 트리를 비교하지 않는다. 매번 전체를 처음부터 만든다.
- **스케줄러는 없다.** 모든 렌더가 동기적으로 완료된다.
- **hooks와 effect는 없다.** 함수 컴포넌트 상태와 사이드 이펙트는 아직 없다.
- **이벤트 위임은 없다.** 각 DOM 요소에 직접 listener를 붙인다.

이 제한들이 각각 다음 프로젝트의 출발점이 된다.
