# 문제 정의 — 왜 로그 파싱으로 테스트를 연습하나

## 출발점

프로젝트 01에서 Go 문법을, 02에서 타입과 에러를 익혔다면, 이제 남은 건 "내가 쓴 코드가 맞는지 어떻게 확인하느냐"는 질문이다. Go에서 테스트는 언어와 도구 체인의 일부다. 별도 프레임워크를 설치할 필요 없이 `go test`만 돌리면 된다. 하지만 "테스트를 어떻게 구조화하느냐"는 별개의 스킬이다.

이 과제에서 익히고 싶은 것은 세 가지였다:
1. **table-driven test** — 입력과 기대값을 데이터로 나열하는 패턴
2. **benchmark** — 성능을 숫자로 측정하는 방법
3. **race detector** — 동시성 문제를 자동으로 찾는 도구

## 왜 로그 파싱인가

이 세 가지를 한 번에 연습하려면, "파싱 에러가 생길 수 있고, 성능이 의미 있고, 동시성이 필요한" 문제가 필요했다. 로그 분석이 딱 맞았다.

- 로그 라인 포맷이 잘못되면 파싱이 실패한다 → table-driven test로 여러 케이스를 나열하기 좋다
- 로그가 수천 줄이면 처리 속도가 중요하다 → benchmark를 넣을 구실이 생긴다
- 로그를 여러 goroutine에서 수집하면 race condition이 생길 수 있다 → Recorder를 race-safe하게 만드는 연습이 된다

## 의도적으로 뺀 것들

- **pprof나 trace**: Go의 프로파일링 도구는 강력하지만, 이 단계에서 넣으면 테스트 작성 자체에 집중하기 어렵다.
- **외부 로그 라이브러리**: zap이나 logrus 같은 라이브러리는 나중에 다룬다. 여기서는 순수 문자열 파싱만 한다.
- **파일 I/O**: 로그를 파일에서 읽는 건 입력 경로를 복잡하게 만든다. `[]string` 슬라이스를 직접 받는 것으로 단순화했다.
