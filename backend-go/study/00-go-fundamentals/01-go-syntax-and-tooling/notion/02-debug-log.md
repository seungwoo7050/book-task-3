# 디버그 기록 — 어디서 막혔고 어떻게 풀었나

## 단어 수가 기대와 안 맞는 문제

테스트를 처음 작성했을 때 자주 부딪히는 건 "단어 수가 하나 더 많거나 적은" 상황이다. 예를 들어 `"Go makes Go tooling approachable, and Go tests stay fast."` 라는 문장에서 총 단어 수가 10인지 9인지가 헷갈릴 수 있다.

이건 구두점 처리 방식에 따라 달라진다. `strings.Fields`로 공백만 기준으로 자르면 `"approachable,"`이 하나의 토큰이 된다. 반면 `FieldsFunc`로 구두점까지 구분자로 처리하면 `"approachable"`과 쉼표가 분리된다. 어떤 함수를 쓰느냐에 따라 결과가 달라지기 때문에, 테스트 기대값을 먼저 손으로 세 보는 게 정확했다.

실제로 기대값을 잘못 세서 테스트가 실패한 적이 있다. `go test`가 실패하면 메시지를 보고 실제 값과 기대값을 비교하면 되는데, 이 과정 자체가 "Go 테스트 루프를 경험하는 것"이기도 했다.

## map 순회 순서의 비결정성

처음에 `topWord`를 map을 직접 순회하면서 골랐는데, 같은 빈도의 단어가 2개 이상일 때 실행마다 다른 결과가 나왔다. 테스트가 때때로 통과하고 때때로 실패하는 flaky test 상태가 됐다.

```
--- FAIL: TestBuildSummary (0.00s)
    lesson_test.go:15: top word = "simple", want go
```

이걸 고치려면 map 순회 자체를 결정적으로 만들어야 했다. 키를 정렬해서 순회하는 `SortedKeys` 함수를 추가한 건 이 문제를 겪고 나서였다. 성능상 비용이 추가되지만, 입문 과제의 데이터 크기에서는 전혀 문제가 안 됐다.

## FieldsFunc에 빈 문자열이 포함되는 경우

`FieldsFunc`의 결과에 빈 문자열이 섞일 수 있다는 건 표준 라이브러리 문서를 꼼꼼히 읽지 않으면 놓치기 쉽다. 실제로는 `FieldsFunc`가 연속된 구분자를 하나로 합치기 때문에 빈 문자열이 잘 안 나오지만, 방어적으로 빈 문자열 필터링을 넣어 두었다.

```go
if field == "" {
    continue
}
```

이 코드가 없어도 현재 테스트는 통과하지만, 입력 데이터가 예상 밖일 때를 대비한 것이었다. 나중에 이 필터가 실제로 필요한 케이스는 발견하지 못했지만, 제거하지 않고 남겨 뒀다.
