# 문제 정의 — 왜 상품 카탈로그인가

## 출발점

프로젝트 01에서 Go 문법의 기초를 익혔다면, 그 다음에 마주치는 건 "값을 어떻게 구조화하고, 행동을 어떻게 분리하며, 에러를 어떻게 의미 있게 전달하느냐"는 질문이다. 이건 Go의 타입 시스템, interface, error 패턴을 한꺼번에 건드리는 문제다.

상품 카탈로그를 골랐다. 이유는 단순하다. 상품을 추가하고, 조회하고, 가격을 계산하는 세 가지 동작만으로 struct, method, interface, error를 전부 자연스럽게 연습할 수 있기 때문이다.

## 핵심 요구사항을 이렇게 잘라낸 이유

1. **SKU 중복 방지**: 같은 상품을 두 번 넣으면 에러. 이걸 sentinel error(`ErrDuplicateSKU`)로 만들면, `errors.Is`로 비교하는 패턴을 배운다.
2. **존재하지 않는 상품 조회**: custom error 타입(`NotFoundError`)으로 반환하면, `errors.As`로 구체적인 정보를 꺼내는 패턴을 배운다.
3. **할인 규칙 분리**: `PricingRule` interface를 만들면, "interface는 행동만 정의한다"는 Go의 철학을 체험한다.

이 세 가지를 하나의 예제에 모으면, 입문자가 타입 시스템에서 흔히 부딪히는 지점을 한 바퀴 돌게 된다.

## 의도적으로 뺀 것들

- 데이터베이스 저장은 없다. 카탈로그는 인메모리 map이다.
- JSON 직렬화도 없다. 타입과 에러에 집중하기 위해서다.
- interface를 여러 개 만들거나 조합하는 것도 넣지 않았다. 하나의 interface로 "이게 뭘 하는 건지"만 느끼면 충분했다.
