# 회고 — 세션과 JWT를 모두 구현해본 뒤

## 무엇을 만들었나

세션 기반 인증과 JWT 기반 인증을 동일한 서버에 구현했다. 사용자 두 명(admin, player), 엔드포인트 다섯 개, 미들웨어 네 가지. 외부 라이브러리는 `golang.org/x/crypto` 하나뿐이고, JWT 서명까지 직접 구현했다.

## 잘된 점

**JWT를 라이브러리 없이 만든 것**이 가장 큰 학습이었다. `header.payload.signature`라는 3-part 구조가 추상적으로 알고 있던 것과, 실제로 base64url 인코딩을 하고 HMAC-SHA256으로 서명하는 코드를 작성하는 것 사이에는 큰 차이가 있었다. padding 제거, URL-safe base64, 상수 시간 비교 같은 디테일을 체감했다.

**인증(Authentication)과 인가(Authorization)의 분리**가 깔끔하게 됐다. `requireAnyAuth`는 "너 누구야?"만 확인하고, `requireRole`은 "이 역할이 맞아?"만 확인한다. 이 분리가 없었다면 `/admin` 엔드포인트에서 세션 사용자와 JWT 사용자를 각각 처리하는 코드가 중복됐을 것이다.

## 아쉬운 점

**세션 저장소가 인메모리**다. 서버가 재시작되면 모든 세션이 사라진다. Redis 같은 외부 저장소를 연결하면 해결되지만, 이 프로젝트의 범위는 아니었다. 그러나 인터페이스로 추상화해두었다면 나중에 교체하기 훨씬 쉬웠을 것이다.

**사용자가 하드코딩**되어 있다. `initUsers`에서 bcrypt 해시를 미리 계산해 맵에 넣는 방식이라, 회원가입 기능은 없다. 05장의 Task API처럼 POST /register를 추가하면 가능하지만, 인증 자체에 집중하려고 의도적으로 생략했다.

**HTTPS가 없다**. HTTP 위에서 세션 쿠키와 JWT를 주고받는 건 실무에서는 치명적이다. `Secure` 플래그 없이 쿠키를 설정하면 중간자 공격에 취약하다. 이 부분은 배포 환경에서 TLS를 설정해야 한다.

## 프로젝트 06과의 연결

06에서 만든 미들웨어 체인 패턴(recoverPanic → logRequest → enableCORS → router)이 그대로 재활용됐다. 07에서는 이 체인에 `requireAnyAuth → requireRole`이 추가된 셈이다. 미들웨어를 `func(http.Handler) http.Handler` 시그니처로 통일한 덕분에, 인증 미들웨어도 기존 체인에 자연스럽게 끼워 넣을 수 있었다.

## 프로젝트 08로의 전달

08(sql-store-api)에서는 이 인증 로직 위에 실제 데이터 저장소를 얹게 된다. 하드코딩된 사용자가 DB로 이동하고, 세션도 DB나 Redis에 저장될 수 있다. 07에서 배운 bcrypt + JWT 패턴은 그대로 유지된다.
