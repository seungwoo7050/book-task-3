# 문제 정의 — 왜 세션과 JWT를 나란히 구현하나

## 출발점

API를 만들었으면 다음 질문은 "누가 이 API를 부르는지 어떻게 아느냐"다. 인증(authentication)은 "누구인가", 인가(authorization)는 "뭘 할 수 있는가"를 결정하는 단계다. 이 과제는 둘 다 다룬다.

세션과 JWT를 한 과제에서 나란히 구현한 이유가 있다. 실무에서는 둘 중 하나만 쓰는 게 보통이지만, 차이를 체감하려면 같은 요구사항을 두 방식으로 풀어 봐야 한다. "서버가 상태를 들고 있느냐(세션), 클라이언트가 서명된 토큰을 가지고 다니느냐(JWT)"의 차이가 코드로 어떻게 드러나는지 보는 게 핵심이다.

## 핵심 요구사항

1. **비밀번호 해싱**: bcrypt로 해시. 평문 저장은 절대 하지 않는다.
2. **세션 로그인**: POST로 자격 증명 → 서버에 세션 저장 → 쿠키로 토큰 반환.
3. **JWT 로그인**: POST로 자격 증명 → HMAC-SHA256 서명 토큰 반환.
4. **보호 리소스**: `/me/session`, `/me/jwt`는 각각 해당 인증 방식 필요.
5. **역할 기반 인가**: `/admin`은 admin 역할만 접근. player는 401이 아니라 403.

## 의도적으로 뺀 것들

- **외부 사용자 저장소**: 사용자를 인메모리 맵에 하드코딩했다. DB 연동은 08번에서.
- **refresh token**: JWT 만료 후 갱신은 이 과제 범위 밖이다.
- **OAuth/OIDC**: 외부 Identity Provider 연동은 하지 않는다.
