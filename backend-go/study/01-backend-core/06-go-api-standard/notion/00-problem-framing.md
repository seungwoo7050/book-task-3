# 문제 정의 — 왜 "표준 라이브러리만으로" 본격 API를 만드나

## 출발점

05번 과제에서 HTTP/REST의 기초를 닦았다. 하지만 거기서는 핸들러 하나에 모든 것이 들어 있었다. 실제 API를 만들려면 더 많은 것이 필요하다: 미들웨어, 에러 응답 통일, 환경변수 설정, graceful shutdown, 코드 구조 분리. 이 과제는 그 간극을 메운다.

핵심 제약은 "외부 프레임워크 없이 표준 라이브러리만 사용"하는 것이다. chi나 echo 같은 라우터를 쓰면 편하지만, "프레임워크가 해 주는 것들"을 직접 만들어야 비로소 그것들이 왜 필요한지 체감할 수 있다.

## 왜 Movie CRUD인가

Movie는 충분히 복잡한 데이터 모델이면서도 익숙한 도메인이다. 제목, 연도, 런타임, 장르 배열, 버전 관리까지 있으면 validation, partial update, pagination을 자연스럽게 연습할 수 있다.

특히 PATCH(부분 업데이트)를 포함시킨 건 의도적이다. PUT은 전체 교체, PATCH는 일부만 변경이라는 HTTP 의미론의 차이를 경험하게 만드는 것이 목적이다.

## Legacy 과제에서 온 것

이 프로젝트는 legacy 저장소의 `01-go-api-standard`에서 마이그레이션됐다. 문제 명세와 핵심 구조는 유지하되, Go 1.22의 새 라우팅과 `log/slog`를 적용하고, 코드 구조를 `cmd/api`, `internal/data`로 정리했다.

## 의도적으로 뺀 것들

- **영속 DB**: 인메모리 스토어를 사용한다. DB 연동은 08번 과제에서 한다.
- **인증/인가**: auth는 07번 과제에서 다룬다.
- **rate limiting**: 11번 과제에서 전용으로 다룬다.
