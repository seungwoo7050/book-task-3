# 문제 정의 — 캐시, 마이그레이션, 관측성의 교차점

## 이전 프로젝트의 한계

08(sql-store-api)에서 SQL 기반 CRUD를 만들었다. 모든 요청이 매번 DB를 조회한다. 읽기가 많고 쓰기가 드문 상품 목록 같은 시나리오에서는 불필요한 DB 부하가 발생한다. 또한 08의 마이그레이션은 Go 상수 두 줄(up/down)에 불과했고, 서버에서 무슨 일이 일어나는지 알 수 있는 관측 수단이 없었다.

## 핵심 과제

1. **Cache-aside 패턴**: 읽기 요청 시 캐시를 먼저 확인(hit)하고, 없으면 DB에서 읽어 캐시에 저장(miss). 쓰기 시 캐시를 무효화(invalidation).
2. **Migration up/down**: 08에 이어 스키마를 코드로 관리하되, seed 데이터도 함께 다룬다.
3. **관측성(Observability)**: 캐시 hit/miss/writes 메트릭 노출, trace ID 전파로 요청 추적 기반 마련.

## 왜 인메모리 캐시인가

실무에서는 Redis를 쓰지만, 캐시의 본질 — hit/miss 판단, invalidation 시점 — 은 인메모리 맵으로도 완전히 학습할 수 있다. Redis 클라이언트 설정, 연결 관리 같은 부수 복잡성을 빼고 캐시 패턴 자체에 집중한다.

## 관측성의 범위

이 프로젝트는 관측성의 세 축(로깅, 메트릭, 트레이싱) 중 기초만 다룬다:
- **로깅**: `log/slog`로 구조화된 로그 (trace ID 포함)
- **메트릭**: Prometheus-like 텍스트 포맷으로 `/metrics` 노출
- **트레이싱**: `X-Trace-ID` 헤더 전파 (분산 트레이싱의 첫 걸음)

OpenTelemetry, Jaeger 같은 도구는 범위 밖이다.
