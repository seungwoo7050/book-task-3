# 회고 — 이 과제에서 얻은 것

## SQL을 직접 쓰는 감각

ORM 없이 `database/sql`만 쓰면 SQL이 코드 안에 문자열로 있다. 처음에는 불편해 보이지만, "어떤 쿼리가 실행되는지"가 코드를 읽는 것만으로 완전히 드러난다. ORM이 생성하는 쿼리를 추측할 필요가 없다. 이 투명함이 디버깅을 쉽게 만든다.

## 스키마 설계가 코드 구조를 결정한다

`inventory` 테이블의 composite PK `(player_id, item_id)`를 정하고 나면, Go 코드에서 "중복 삽입을 어떻게 처리할까"가 자동으로 결정된다. `ON CONFLICT`를 쓸 수밖에 없다. 스키마를 먼저 잘 잡으면, 애플리케이션 로직이 단순해진다는 걸 체감했다.

## in-memory SQLite는 테스트에 강력하다

외부 DB 서버 없이 `go test`만으로 스키마, 시드, 쿼리, 트랜잭션을 전부 검증할 수 있다. CI에서도 별도 서비스 기동이 필요 없다. 다만 SQLite의 락 모델이나 격리 수준은 PostgreSQL과 다르기 때문에, 운영 환경과 차이가 있다는 건 인식해 둬야 한다.
