# 문제 정의 — 왜 API 전에 SQL 모델링을 먼저 하나

## 출발점

Go 기초(01~03)를 마치고 바로 HTTP API로 넘어갈 수도 있었다. 하지만 API를 만들면 결국 데이터를 어딘가에 저장해야 한다. 그때 스키마 설계 감각이 없으면, `database/sql`이나 ORM을 "그냥 돌아가게" 쓰게 되고, 나중에 데이터 무결성 문제가 터진다.

이 과제는 API 과제로 넘어가기 전의 브리지 역할이다. "테이블을 어떻게 나누고, 관계를 어떻게 표현하고, 트랜잭션으로 뭘 보호하는지"를 작은 예제로 먼저 경험해 두는 것이다.

## 왜 게임 상점인가

플레이어가 아이템을 사는 모델은 다대다 관계를 자연스럽게 만든다. 한 플레이어가 여러 아이템을, 한 아이템이 여러 플레이어에게 소유될 수 있다. 이걸 표현하려면 중간 테이블(`inventory`)이 필요하고, FK와 composite PK가 등장한다.

동시에 "구매"라는 행위는 트랜잭션으로 묶어야 한다. 아이템 조회와 인벤토리 추가가 원자적이어야 "반만 된 구매"가 생기지 않는다.

## 의도적으로 뺀 것들

- **실제 RDBMS**: PostgreSQL이나 MySQL 대신 SQLite in-memory를 사용했다. 외부 서비스 기동 없이 `go test`만으로 검증 가능하게 만드는 게 우선이었다.
- **마이그레이션 도구**: goose나 migrate 같은 도구는 08번 과제에서 다룬다. 여기서는 `CREATE TABLE`을 Go 코드에 직접 넣어서 스키마 자체에 집중했다.
- **ORM**: GORM이나 sqlx 같은 것 없이 순수 `database/sql`만 사용했다. SQL을 직접 쓰는 감각을 먼저 들이기 위해서다.
