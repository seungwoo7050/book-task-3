# 문제 정의 — 왜 이 네 개의 엔드포인트인가

## 출발점

SQL 모델링(04)을 마치고 나면 자연스럽게 "이 데이터를 HTTP로 노출하려면 어떻게 해야 하지?"라는 질문이 생긴다. 이 과제는 그 질문에 대한 가장 작은 답이다. Task라는 단순한 자원을 만들고, 조회하고, 목록을 페이지 단위로 반환하는 것. 이 정도면 REST API의 핵심 감각을 대부분 건드린다.

## 네 개의 엔드포인트를 고른 이유

1. **`GET /v1/healthcheck`** — 서비스가 살아 있는지 확인하는 최소 엔드포인트. 로드밸런서나 모니터링 도구가 이걸 호출한다. API를 만들 때 가장 먼저 작성하는 관례다.

2. **`POST /v1/tasks`** — 리소스 생성. 여기서 JSON 디코딩, validation(빈 제목 거부), 상태 코드 분리(201 vs 200), idempotency key를 전부 경험한다.

3. **`GET /v1/tasks`** — 목록 조회 + pagination. `page`와 `page_size` 쿼리 파라미터를 파싱하고 경계 조건을 처리하는 연습이 된다.

4. **`GET /v1/tasks/{id}`** — 단건 조회. path parameter 파싱과 404 응답을 다룬다.

## 왜 in-memory인가

이 과제에서는 데이터베이스를 연결하지 않았다. 슬라이스와 맵으로 충분했다. 영속 저장소를 넣으면 DB 연결, 마이그레이션, 트랜잭션 같은 관심사가 들어와서 "HTTP 감각 잡기"라는 본래 목적이 흐려진다. DB 연동은 08번 과제에서 한다.

## idempotency key를 여기에 넣은 이유

처음에는 idempotency key가 입문 과제에 과한 거 아닌가 싶었다. 하지만 "같은 요청을 두 번 보내면 어떻게 되는가?"는 현실 API에서 매우 흔한 질문이다. 네트워크 재시도, 프론트엔드의 더블 클릭 같은 상황을 상상하면, idempotency는 "나중에 배울 것"이 아니라 "처음부터 의식해야 할 것"이다.

구현 자체는 단순하다—메모리 맵에 키를 저장하고, 같은 키가 들어오면 기존 결과를 반환한다. 이 단순한 구현이 "idempotency란 뭔가"를 체감하게 만드는 데는 충분했다.
