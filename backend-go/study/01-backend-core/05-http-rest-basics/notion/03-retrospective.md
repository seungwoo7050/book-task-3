# 회고 — 이 과제에서 얻은 것

## Go 1.22의 라우팅은 프레임워크 없이도 충분했다

이전에는 gorilla/mux나 chi를 써야 `GET /v1/tasks/{id}` 같은 라우팅이 가능했다. Go 1.22에서 `ServeMux`가 이걸 지원하면서, 작은 API에서는 외부 라우터가 필요 없어졌다. 표준 라이브러리만으로 CRUD API를 완성할 수 있다는 건 의존성 관리 측면에서 큰 이점이다.

## 상태 코드를 의식적으로 고르는 습관

처음에는 "성공이면 200, 실패면 500"으로 단순하게 생각했다. 하지만 201 vs 200, 400 vs 422처럼 상태 코드를 세분화하면, 클라이언트에게 "무슨 일이 일어났는지"를 코드 한 줄로 전달할 수 있다. 이 감각은 이후 모든 API 과제에서 계속 쓰인다.

## httptest는 서버를 띄우지 않고 API를 검증한다

`httptest.NewRequest`와 `httptest.NewRecorder`로 실제 TCP 연결 없이 핸들러를 테스트할 수 있다. 서버를 `go run`으로 띄우고 curl로 확인하는 것도 유용하지만, 자동화된 테스트는 `httptest`로 하는 게 정확하고 빠르다.

## 다음 과제와의 연결

이 과제에서 만든 JSON API 패턴(라우팅, 핸들러, 상태 코드, validation)이 06번(go-api-standard)에서 더 정교한 미들웨어, 에러 처리, 구조 분리로 확장된다.
