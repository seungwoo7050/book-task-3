# 문제 정의 — 왜 분산 로그인가

## 분산 시스템의 기반 구조

Kafka, NATS, Pulsar — 이 시스템들의 공통점은 **커밋 로그(commit log)**다. 순서가 보장되고, 한 번 기록하면 수정하지 않는(append-only) 데이터 구조. 이벤트 소싱, 메시지 큐, 변경 데이터 캡처(CDC)의 핵심이다.

12에서 gRPC로 서비스 간 통신을 배웠다. 하지만 통신만으로는 분산 시스템이 만들어지지 않는다. 데이터를 영속적으로 저장하고, 복제하고, 장애 시 복구하는 메커니즘이 필요하다. 그 메커니즘의 기초가 커밋 로그다.

## 핵심 과제

네 계층으로 쌓아올린다:

| 계층 | 역할 |
|------|------|
| **Store** | 파일에 바이너리 레코드를 append-only로 저장 |
| **Index** | 오프셋 → 파일 위치 매핑 (mmap으로 빠른 조회) |
| **Segment** | Store + Index를 묶어 하나의 단위로 관리 |
| **Log** | 여러 Segment를 관리, 자동 rotation |

## 왜 직접 만드는가

Kafka를 운영하려면 Java, ZooKeeper, 브로커 설정을 알아야 한다. 하지만 "왜 Kafka가 빠른가?"를 이해하려면  append-only 쓰기, mmap 읽기, 세그먼트 rotation의 원리를 직접 구현해 봐야 한다.

## 외부 의존성 없음

Go 표준 라이브러리만 사용. `syscall.Mmap`/`Munmap`으로 메모리 매핑을 직접 다룬다. 파일 I/O, 바이너리 인코딩, `bufio.Writer` — 모두 stdlib.
