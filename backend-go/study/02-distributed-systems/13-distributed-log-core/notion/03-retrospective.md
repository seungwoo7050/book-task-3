# 회고 — 분산 로그의 뼈대를 만들며

## 4계층 아키텍처는 정말 필요했을까

Store → Index → Segment → Log. 이 4단계 추상화가 과하다고 느꼈던 순간이 있다. "그냥 하나의 파일에 쭉 쓰면 안 되나?" 하지만 각 계층이 하나의 책임만 지면서 테스트가 독립적으로 가능해졌다. Store 테스트에서 Index를 신경 쓸 필요가 없고, Segment 테스트에서 다중 Segment 관리를 걱정할 필요가 없다.

Kafka의 내부 구조도 이와 비슷하다. `.log` 파일(Store에 해당)과 `.index` 파일이 Segment 단위로 묶이고, Partition이 여러 Segment를 관리한다. 이 프로젝트에서 만든 Log가 Kafka의 Partition 하나에 해당한다.

## mmap을 선택한 이유와 대가

mmap은 커널이 페이지 캐시를 관리해주므로 별도의 캐싱 로직 없이 빠른 읽기가 가능하다. 하지만 대가가 있다.

첫째, 파일 크기를 미리 정해야 한다. Index를 생성할 때 `maxBytes`로 파일을 확장하고, Close할 때 실제 크기로 줄인다. 이 과정을 빠뜨리면 파일에 쓰레기 바이트가 남는다.

둘째, `syscall.Mmap`과 `syscall.Munmap`은 플랫폼 의존적이다. 이 프로젝트는 리눅스/맥에서만 동작한다. Windows에서는 `syscall.CreateFileMapping`을 써야 한다.

셋째, mmap된 영역 밖에 접근하면 SIGBUS가 발생하고 프로세스가 죽는다. recover로 잡을 수 없다.

## 바이너리 인코딩의 단순함

protobuf나 JSON 대신 `binary.BigEndian.PutUint64`로 길이를 인코딩했다. 8바이트 고정 헤더 + 가변 데이터. 파싱이 극도로 빠르고 구현이 간단하다. 하지만 스키마가 없으므로, 데이터 형식이 변경되면 마이그레이션이 어렵다. 범용 메시지 브로커라면 protobuf 같은 스키마 기반 인코딩이 낫겠지만, 이 프로젝트의 범위에서는 바이너리 인코딩이 적절한 선택이었다.

## 복제가 빠진 로그는 일기장이다

이 프로젝트에서 만든 것은 단일 노드 로그다. 서버가 죽으면 데이터는 디스크에 남지만, 디스크가 고장나면 끝이다. Raft 같은 합의 알고리즘으로 여러 노드에 복제해야 진정한 "분산" 로그가 된다.

하지만 복제를 구현하기 전에 단일 노드 로그의 동작을 완벽히 이해하는 것이 먼저다. Append, Read, Truncate, Reset — 이 기본 연산의 의미론을 확실히 잡아야 복제 프로토콜에서 "어떤 상태를 동기화할 것인가"를 결정할 수 있다.

## 테스트에서 배운 것

테스트 파일들(`store_test.go`, `index_test.go`, `segment_test.go`, `log_test.go`)을 통해 각 계층의 계약(contract)을 명확히 했다. 특히 `t.TempDir()`을 활용해 테스트 간 격리를 보장한 점이 중요하다. 파일 시스템에 의존하는 코드는 테스트 격리가 까다로운데, Go의 `t.TempDir()`이 테스트 종료 시 자동 정리를 해줘서 깔끔했다.
