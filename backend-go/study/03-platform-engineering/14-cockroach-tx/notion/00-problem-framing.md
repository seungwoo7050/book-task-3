# 문제 정의 — 분산 DB에서 "돈"을 다루는 것의 의미

## 왜 이 프로젝트인가

게임 플랫폼의 인벤토리 관리. 플레이어가 아이템을 구매하면 잔액 차감, 인벤토리 추가, 감사 로그 기록이 하나의 트랜잭션 안에서 이루어져야 한다. 단순해 보이지만, CockroachDB를 쓰면 세 가지 새로운 문제가 드러난다.

첫째, **직렬화 충돌**. CockroachDB는 Serializable Isolation을 기본으로 사용한다. 두 트랜잭션이 같은 행을 동시에 수정하면, 하나는 SQLSTATE `40001`을 받고 재시도해야 한다. PostgreSQL의 기본 Read Committed와는 완전히 다른 세계다.

둘째, **멱등성**. 네트워크 타임아웃 후 클라이언트가 같은 구매 요청을 다시 보낸다. 돈이 두 번 빠지면 안 된다. 멱등성 키로 이전 응답을 캐시해두고 중복 요청에 같은 응답을 돌려줘야 한다.

셋째, **낙관적 잠금**. 잔액을 차감할 때 `WHERE version = ?`로 다른 트랜잭션이 먼저 수정했는지 확인한다. 프로젝트 08에서 이미 다뤘지만, CockroachDB의 재시도 메커니즘과 결합하면 복잡도가 올라간다.

## 핵심 도전

```
[HTTP 요청] → [멱등성 키 확인] → [잔액 차감 (낙관적 잠금)] → [인벤토리 추가] → [감사 로그] → [멱등성 키 저장]
                                        ↕
                              CockroachDB가 40001을 던지면?
                              → 트랜잭션 전체를 처음부터 재시도
```

이 모든 것이 하나의 트랜잭션 안에서 일어나야 하므로, 트랜잭션 재시도 시 `fn(tx)` 전체가 다시 실행된다. Side effect가 있으면 안 된다.

## 프로젝트 08과의 차이

프로젝트 08에서는 SQLite + `database/sql` + 단일 프로세스였다. 동시성 문제는 `sync.Mutex`로도 해결할 수 있었다. 여기서는 CockroachDB라는 분산 DB가 동시성 제어를 담당하고, 애플리케이션은 DB가 주는 재시도 신호에 따라야 한다.

또한 프로젝트 08은 `modernc.org/sqlite`를 직접 사용했지만, 여기서는 `jackc/pgx/v5`를 `database/sql` 드라이버로 등록해서 사용한다. PostgreSQL 호환 프로토콜이라 SQLSTATE 에러 코드를 파싱해야 한다.
