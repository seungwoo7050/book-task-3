# 회고 — at-least-once는 타협이 아니라 설계다

## Exactly-Once의 환상

분산 시스템에서 "정확히 한 번" 메시지 전달은 이론적으로 불가능하지는 않지만, 현실적으로 매우 비싸다. Kafka 자체는 "exactly-once semantics"를 지원하지만, 이는 Kafka-to-Kafka를 의미한다. 외부 시스템(DB)과 Kafka 사이에서는 여전히 at-least-once + 멱등 Consumer가 실용적인 해답이다.

Outbox 패턴은 이 현실을 인정하는 설계다. DB 트랜잭션으로 일관성을 보장하고, Relay로 비동기 발행을 하고, Consumer에서 중복을 처리한다.

## 2단계 멱등성: 인메모리 + DB

Consumer의 2단계 중복 체크는 성능과 정확성의 균형이다. 인메모리 캐시는 빠르지만 휘발성. DB 테이블은 느리지만 영속적. 대부분의 중복은 캐시에서 걸러지고, 재시작 직후에만 DB를 조회한다.

프로덕션에서는 캐시에 TTL을 두거나, Redis 같은 외부 캐시를 사용할 수 있다. 하지만 이 프로젝트의 목적은 패턴의 본질을 이해하는 것이므로 `map + DB` 조합이 충분했다.

## Relay는 단일 프로세스여야 하는가

현재 Relay는 단일 고루틴이다. 여러 인스턴스를 실행하면? 같은 미발행 이벤트를 두 Relay가 동시에 가져가서 중복 발행할 수 있다.

해결 방법:
1. `SELECT ... FOR UPDATE SKIP LOCKED` — 이미 다른 Relay가 잡은 행을 건너뜀
2. 리더 선출 — 한 번에 하나의 Relay만 활성화
3. 파티셔닝 — aggregate_id 범위별로 Relay를 분리

이 프로젝트에서는 단일 Relay로 충분. 확장이 필요하면 `FOR UPDATE SKIP LOCKED`가 가장 간단.

## 프로젝트 13과의 연결

프로젝트 13에서 만든 append-only 로그의 개념이 Kafka 토픽과 겹친다. Store = 파티션 파일, Segment = 파티션의 세그먼트 파일, Index = 오프셋 인덱스. 차이는 Kafka가 네트워크를 통해 다중 프로듀서/컨슈머를 지원하고, 복제를 통해 내구성을 보장한다는 점이다.

## Cleanup의 필요성

Outbox 테이블이 무한히 커지면 성능에 영향을 준다. `Cleanup` 메서드로 발행 완료된 이벤트를 주기적으로 삭제한다. 그러나 감사 목적으로 보존이 필요하다면, 별도의 아카이브 테이블로 이동하는 것이 나을 수 있다.

Dead Letter Queue를 추가할 것이다. 현재는 핸들러 실패 시 무한 재시도인데, poison pill(파싱 불가능한 메시지)이 영원히 Consumer를 막는다. 일정 횟수 실패 후 별도 토픽으로 이동하면 나머지 메시지 처리를 계속할 수 있다.
