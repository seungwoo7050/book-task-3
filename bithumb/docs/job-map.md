# 채용 공고와 프로젝트 연결

이 트랙은 “AWS를 많이 써 봤다”는 주장보다, 보안 자동화 문제를 코드와 문서로 설명할 수 있는지를 보여 주는 데
집중합니다. 각 프로젝트는 클라우드 보안 주니어 공고에서 자주 보이는 책임을 작은 단위로 쪼개어 대응합니다.

| 공고에서 자주 보이는 책임 | 연결 프로젝트 | 설명 포인트 |
| --- | --- | --- |
| IAM 정책 이해와 최소 권한 | `01`, `04` | 정책 평가와 위험 finding을 둘 다 설명할 수 있습니다. |
| Terraform/설정 기반 misconfiguration 탐지 | `02`, `05` | plan JSON을 읽고 triage 가능한 규칙으로 바꾸는 흐름을 보여 줍니다. |
| 탐지 이후 조치 자동화 | `06` | finding을 dry-run 조치안으로 연결하는 관점을 보여 줍니다. |
| 로그 적재와 이상 행위 탐지 | `03`, `07` | 원본 로그를 적재하고 queryable한 구조로 바꾸는 감각을 증명합니다. |
| 컨테이너 보안 정책 점검 | `08` | Kubernetes manifest와 이미지 메타데이터 기반 규칙을 설명할 수 있습니다. |
| 예외, 증적, 감사 흐름 관리 | `09` | 거버넌스 관점의 보안 운영을 작은 모델로 설명합니다. |
| 보안 플랫폼/통합 시스템 구축 | `10` | 앞선 로직을 API, worker, 상태 저장소, 보고 흐름으로 통합합니다. |

## 포트폴리오로 옮길 때 강조할 점

- 기능 목록보다 “어떤 운영 질문에 답하는 프로젝트인지”를 먼저 씁니다.
- 테스트 통과 여부뿐 아니라, 어떤 입력과 어떤 한계를 전제로 했는지 함께 적습니다.
- 캡스톤 하나만 보여 주기보다, 캡스톤에 들어간 작은 판단 엔진들을 함께 묶어서 설명하는 편이 더 설득력 있습니다.
