# 문서 정책

## 목적

이 저장소의 문서는 학습 과정을 숨기지 않으면서도, 방문자가 `문제 -> 답 -> 검증 -> 한계`를 빠르게 파악하도록
돕는 것을 목표로 합니다. 문서는 코드 설명을 대신하는 장문 에세이가 아니라, 공개 표면과 재현 기준을 분명하게
만드는 장치여야 합니다.

## 공개 표면 계약

- 루트 `README.md`는 전체 프로젝트 매트릭스를 가져야 합니다.
- 트랙 `README.md`는 각 프로젝트의 `문제 / 답 / 검증 / 다음 단계 연결`을 보여 줘야 합니다.
- 프로젝트 `README.md`는 아래 순서를 고정합니다.
  - `## 풀려는 문제`
  - `## 내가 낸 답`
  - `## 입력과 출력`
  - `## 검증 방법`
  - `## 현재 상태`
  - `## 한계와 다음 단계`
  - `## 더 깊게 읽을 문서`

## 하위 문서 계약

- `problem/README.md`
  - `## 원래 문제`
  - `## 제공된 자료`
  - `## 제약`
  - `## 통과 기준`
  - `## 이번 프로젝트에서 일부러 제외한 것`
- `python/README.md`
  - `## 구현한 답의 범위`
  - `## 핵심 엔트리포인트`
  - `## 실행`
  - `## 테스트`
  - `## 대표 출력 예시`
  - `## 구현 메모`
- `docs/README.md`
  - 링크 목록보다 먼저 “이 문서 묶음이 설명하는 것”을 적습니다.
  - 읽는 순서와 핵심 질문은 뒤에 배치합니다.
- `notion/README.md`
  - 프로젝트 README를 보완하는 공개 학습 기록입니다.
  - 빠른 재현 경로, 디버깅 근거, 결정 이유를 중심으로 유지합니다.
  - 프로젝트의 첫 진입점 역할은 맡기지 않습니다.

## 언어와 주석 원칙

- 설명 문서는 한국어를 기본으로 쓰고, 기술명·패키지명·CLI 명령·설정 키·파일 경로는 원문을 유지합니다.
- 코드 식별자, JSON 키, SQL, severity, control ID, CLI 출력 문자열은 영어를 유지합니다.
- 코드 주석과 docstring은 한국어 기준으로 정리하되, “무엇을 하는가”보다 “왜 이렇게 했는가”를 설명할 때만 추가합니다.
- 무의미한 package docstring은 삭제하거나 한국어 한 줄 설명으로 치환합니다.

## 증거와 재현 원칙

- 모든 명령은 레포 루트 기준으로 적습니다.
- 추상적인 감상보다 실제 파일 경로, 입력, 출력, 검증 명령을 우선합니다.
- 문서에 적는 검증 명령은 실제로 실행 가능한 경로만 사용합니다.
- `verified` 같은 상태 표시는 테스트나 데모 기준과 함께 유지합니다.

## notion 정책

- `notion/`은 공개용 백업 문서로 레포에 포함합니다.
- 새 `notion/`은 `README.md`, `00-problem-framing.md`, `01-approach-log.md`, `02-debug-log.md`, `03-retrospective.md`, `04-knowledge-index.md`, `05-reproduction-guide.md`를 기본 세트로 사용합니다.
- `05-reproduction-guide.md`는 가장 빠른 재현 경로와 기대 결과를 적습니다.
- 기존 `notion/`을 다시 쓸 때는 먼저 기존 폴더를 `notion-archive/`로 보존합니다.
