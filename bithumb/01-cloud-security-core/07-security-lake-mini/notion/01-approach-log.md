# 접근 기록

## 핵심 선택

- detection rule 다섯 개를 개별 파이프라인이 아니라 하나의 SQL 분류 흐름으로 묶었습니다.
- alert 구조를 finding 구조와 일부러 분리했습니다. 행위 기반 탐지에서는 `event_name`, `actor`, `occurred_at`가 더 중요합니다.
- 적재 후 Parquet를 유지해 다음 단계의 재사용성과 도구 호환성을 열어 두었습니다.

## 버린 대안

- 외부 룰 파일이나 복잡한 detection framework는 도입하지 않았습니다.
- 여러 로그 소스를 동시에 상관 분석하는 구조는 capstone으로 미뤘습니다.
- severity 점수화까지 넣는 대신, 먼저 control_id가 정확히 나오는지를 우선했습니다.

## 다음 프로젝트와의 연결

이 프로젝트는 `alert`를 다루지만, capstone에서는 결국 이 결과가 다른 finding과 한 흐름에 들어갑니다. 따라서 지금 단계에서
“행위 기반 탐지와 정적 분석 결과는 무엇이 다른가”를 구분하는 것이 중요합니다.

## 다시 써도 유지할 기준

- query 하나가 어떤 운영 질문을 대신하는지 설명할 수 있어야 합니다.
- fixture와 기대 결과가 명확해야 합니다.
- lake 적재와 탐지가 한 번의 흐름으로 보이도록 유지해야 합니다.
