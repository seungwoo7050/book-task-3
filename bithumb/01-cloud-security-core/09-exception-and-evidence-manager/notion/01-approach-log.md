# 접근 기록

## 핵심 선택

- 예외, 증거, 감사 이력을 서로 다른 데이터 구조로 나눴습니다.
- suppression 판단은 현재 시간과 `approved` 상태, `expires_at`를 함께 보게 했습니다.
- audit event는 별도 리스트에 append-only로 남겨 “무엇이 언제 일어났는가”를 따로 추적할 수 있게 했습니다.

## 버린 대안

- 예외를 단순 boolean 플래그로 두는 방식은 포기했습니다. 승인자와 만료, 사유를 설명할 수 없기 때문입니다.
- DB를 바로 붙이는 대신 메모리 모델로 먼저 시작했습니다. 이 과제의 목적은 비즈니스 로직 고정입니다.
- 예외와 증거를 하나의 자유형 딕셔너리로 합치지 않았습니다. 이후 capstone 확장을 생각하면 분리가 더 낫습니다.

## 다음 프로젝트와의 연결

현재 구현은 메모리 모델이지만, 필드 구조는 capstone의 DB 모델로 옮기기 좋게 설계되어 있습니다. `scope_type`, `scope_id`,
`expires_at`, `approved_by` 같은 필드를 지금 단계에서 분명하게 가져가는 이유가 여기에 있습니다.

## 다시 써도 유지할 기준

- 예외는 “무시”가 아니라 “근거 있는 억제”여야 합니다.
- audit는 append-only 성격을 잃지 않아야 합니다.
- expiry와 approval을 함께 보지 않으면 suppression 설명이 부정확해집니다.
