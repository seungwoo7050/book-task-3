# 회고

## 이 프로젝트가 실제로 증명한 것

- 이 프로젝트는 보안 운영에서 중요한 것이 finding 생성만이 아니라, 예외를 어떤 근거로 승인하고 언제 만료시키는가라는 점을 분명히 증명했습니다.
- 메모리 모델만으로도 create, approve, suppress, append evidence, append audit event 흐름을 충분히 고정할 수 있음을 보여 줍니다.
- append-only audit event는 나중에 DB로 옮겨도 의미가 유지되는 핵심 개념입니다.

## 이번 버전이 의도적으로 단순화한 것

- 영속화, revoke, 역할 기반 승인, 알림, 외부 증적 저장소 연동은 아직 없습니다.
- 한 프로세스 안에서만 state가 유지되므로 서비스 재시작 이후 복구는 다루지 않습니다.
- evidence payload는 매우 작고, 실제 파일 해시나 무결성 검증은 없습니다.

## 학습자가 여기서 반드시 가져가야 할 판단

- 학습자는 suppression을 “위험을 지운다”가 아니라 “기한과 근거를 가진 예외 상태”로 이해해야 합니다.
- expiry와 approval을 같은 boolean으로 뭉치지 않고 분리해 둔 덕분에, 만료 후 상태를 설명하기가 쉬워집니다.
- 이 프로젝트의 성공 기준은 많은 기능 추가가 아니라, 예외 관리의 핵심 상태 전이를 명확히 보이는 데 있습니다.

## 공개 기록으로 확장할 때 보강할 증거

- 공개용 기록에서는 audit event 세 개가 어떤 순서로 쌓이는지 예시를 적어 두면 매우 효과적입니다.
- `is_suppressed`가 시간이 지나면 false로 바뀌는 장면을 캡처해 두면 expiry 개념을 설명하기 쉽습니다.
- capstone에서 같은 개념이 DB와 API로 확장되는 지점을 연결하면 이 프로젝트의 가치가 더 분명해집니다.
