# 회고

## 이 프로젝트가 실제로 증명한 것

- 이 프로젝트는 로그 파일을 읽는 것만으로는 부족하고, 이후 detection query가 기대는 공통 필드 구조를 먼저 만들어야 한다는 점을 증명했습니다.
- CloudTrail과 VPC Flow Logs처럼 형식이 다른 입력도 `occurred_at`, `event_name`, `actor`, `resource_id` 같은 공통 질문 축으로 정리할 수 있음을 보여 줍니다.
- DuckDB와 Parquet를 함께 남기면 “즉시 질의”와 “재사용 가능한 산출물”을 동시에 확보할 수 있습니다.

## 이번 버전이 의도적으로 단순화한 것

- 실제 CloudTrail 전체 필드나 VPC Flow Logs 원본 텍스트 포맷은 모두 담지 않았습니다.
- 대용량 처리, 파티셔닝, 스키마 진화 같은 운영형 ETL 고민은 제외했습니다.
- 현재 단계에서는 alert 생성까지 하지 않고, 정규화와 집계 질의까지만 책임집니다.

## 학습자가 여기서 반드시 가져가야 할 판단

- 보안 로그 학습 초반에는 “모든 필드를 보존하는 것”보다 “어떤 질문을 가능하게 할 것인가”를 먼저 결정하는 편이 더 중요합니다.
- 테스트에서 이벤트 집계와 시간 범위 필터를 함께 검증하는 이유는, ETL이 실제 분석 질문에 답할 수 있는지 확인하기 위해서입니다.
- 이 프로젝트의 성공 기준은 화려한 시각화가 아니라, 뒤의 security lake mini가 바로 받아 쓸 수 있는 최소 구조를 고정하는 데 있습니다.

## 공개 기록으로 확장할 때 보강할 증거

- 공개용 기록에서는 `ingested 4 records` 같은 총량 메시지보다, 어떤 event_name과 actor 집계가 나와야 하는지를 같이 적는 편이 더 실효적입니다.
- 공통 필드를 왜 골랐는지 표로 정리하면, “정규화”가 단순 변환이 아니라 질문 설계라는 점이 더 잘 드러납니다.
- 다음 프로젝트와 연결할 때는 Parquet 파일 존재 여부보다, 그 파일이 detection query의 입력이라는 사실을 먼저 보여 주는 것이 좋습니다.
