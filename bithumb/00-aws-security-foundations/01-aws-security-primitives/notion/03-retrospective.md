# 회고

## 이 프로젝트가 실제로 증명한 것

- 이 프로젝트는 policy evaluation에서 중요한 것이 최종 boolean 하나가 아니라, 각 statement가 왜 매칭되거나 제외됐는지 설명하는 기록이라는 점을 증명했습니다.
- 테스트 세 개만으로도 allow 성립, explicit deny 우선, no allow statement matched라는 세 가지 핵심 규칙을 재현할 수 있어 학습 단위가 지나치게 크지 않다는 것도 보여 줍니다.
- 순수 함수 기반 엔진을 택했기 때문에 뒤 프로젝트에서 “평가 결과를 finding으로 포장하는 계층”을 얹기 쉬운 구조가 됐습니다.

## 이번 버전이 의도적으로 단순화한 것

- Condition, Principal, NotAction, NotResource, 조직 단위 정책처럼 IAM의 전체 복잡도는 의도적으로 넣지 않았습니다.
- AWS API 호출이나 실제 계정 상태를 읽지 않고, JSON fixture만으로 결론을 냅니다. 이 단순화 덕분에 재현성은 높지만 현실 반영 범위는 제한됩니다.
- 리소스 문자열과 액션 와일드카드 매칭만 다루기 때문에 서비스별 예외 규칙은 포함되지 않습니다.

## 학습자가 여기서 반드시 가져가야 할 판단

- 학습자 입장에서는 “평가 엔진을 얼마나 크게 만드는가”보다 “근거를 얼마나 설명 가능하게 남기는가”가 더 중요합니다.
- negative case가 없는 보안 엔진은 신뢰하기 어렵기 때문에, `test_request_denied_when_no_allow_statement_matches` 같은 케이스를 같은 비중으로 읽어야 합니다.
- 이 프로젝트의 성공 기준은 AWS IAM 전체 재현이 아니라, 뒤 단계의 analyzer가 기대는 최소한의 평가 semantics를 흔들림 없이 고정하는 데 있습니다.

## 공개 기록으로 확장할 때 보강할 증거

- 공개용 문서로 옮길 때는 allow/deny 결과 표보다 `decision.reason`과 `matches[].reason`을 함께 캡처해 두는 편이 좋습니다.
- CLI 출력 예시 하나와 세 개의 테스트 이름을 나란히 두면, 구현과 검증이 같은 규칙을 가리킨다는 점을 더 쉽게 보여 줄 수 있습니다.
- 뒤 프로젝트인 `04-iam-policy-analyzer`에서 어떤 필드가 그대로 재사용되는지 연결 그림을 추가하면 레포 전체 흐름이 더 선명해집니다.
