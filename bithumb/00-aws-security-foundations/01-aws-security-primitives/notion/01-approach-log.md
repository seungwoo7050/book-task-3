# 접근 기록

## 핵심 선택

- 엔진을 외부 상태 없는 순수 함수로 두었습니다. 같은 입력이면 항상 같은 판단이 나오고, 테스트 fixture만 있으면 즉시 검증할 수 있습니다.
- wildcard 매칭은 표준 라이브러리 `fnmatch`에 기대는 방향을 택했습니다. 정책 규칙을 정규표현식으로 다시 만드는 것보다 의도가 분명합니다.
- 최종 결과만 돌려주지 않고 statement별 결과를 함께 남기도록 설계했습니다. 이 결정이 나중에 CLI의 `explain` 흐름과 연결됩니다.

## 버린 대안

- Condition, SCP, Permission Boundary를 처음부터 넣는 방식은 포기했습니다. 학습 순서를 망가뜨리는 과한 확장이라고 판단했습니다.
- 실제 AWS SDK 기반 권한 검증은 배제했습니다. 이 트랙은 로컬 fixture만으로도 재현 가능한 학습 구조를 유지해야 합니다.
- “허용이 하나라도 있으면 허용” 같은 단순 규칙은 explicit deny 우선순위를 제대로 설명하지 못하므로 채택하지 않았습니다.

## 다음 프로젝트와의 연결

이 프로젝트에서 중요한 것은 정책 문법이 아니라 판단 흐름입니다. `04-iam-policy-analyzer`는 같은 정책 입력을 받아
“허용되는가?” 대신 “얼마나 위험한가?”를 묻습니다. 따라서 지금 단계에서 statement 매칭과 deny 우선순위를 정확히
잡아 두는 것이 뒤 프로젝트의 품질을 결정합니다.

## 다시 써도 유지할 기준

- 설명 가능성이 성능보다 우선입니다.
- fixture만으로 재현 가능해야 합니다.
- 테스트 이름만 봐도 어떤 정책 규칙을 검증하는지 드러나야 합니다.
