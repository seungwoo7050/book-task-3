# 회고

## 이 프로젝트가 실제로 증명한 것

- 이 프로젝트는 Terraform을 “인프라를 실제로 띄우는 도구” 이전에 “선언형 상태를 분석 가능한 JSON으로 바꾸는 도구”로도 학습할 수 있음을 증명했습니다.
- insecure와 secure 예제를 쌍으로 유지하면, 단순히 취약한 예제를 모으는 것보다 훨씬 강한 비교 학습이 가능하다는 점도 확인됩니다.
- 후속 프로젝트가 필요로 하는 것은 클라우드 계정이 아니라 안정적인 입력 스키마라는 사실이 분명해집니다.

## 이번 버전이 의도적으로 단순화한 것

- remote state, module graph, provider alias, workspace 전략처럼 Terraform 운영 난이도를 높이는 요소는 의도적으로 제외했습니다.
- 실제 apply 이후 drift나 cloud-side side effect는 다루지 않습니다. 대신 `terraform show -json` 결과를 신뢰 가능한 중간 산출물로 취급합니다.
- AWS 계정 자격 증명은 `study2` 더미 값으로 고정돼 있어 실제 리소스와 연결되지 않습니다.

## 학습자가 여기서 반드시 가져가야 할 판단

- 이 프로젝트의 핵심은 Terraform 문법 숙달이 아니라, 보안 엔진이 읽기 좋은 fixture를 얼마나 안정적으로 만드는가에 있습니다.
- 재현성 관점에서는 `-refresh=false`와 local-only plan 생성이 가장 중요한 설계 선택입니다.
- 학습자가 여기서 놓치기 쉬운 포인트는 secure 예제도 반드시 같은 흐름으로 통과시켜야 한다는 점입니다. 취약한 입력만 있으면 규칙 엔진 품질을 평가할 수 없습니다.

## 공개 기록으로 확장할 때 보강할 증거

- 공개용 기록에서는 `main.tf` diff보다 `tfplan.json`의 리소스 타입 차이를 먼저 보여 주는 편이 더 설득력 있습니다.
- CSPM 엔진이 읽는 실제 필드 이름을 몇 개 골라 캡처하면, 왜 이 프로젝트가 다음 프로젝트의 기반인지 더 쉽게 설명할 수 있습니다.
- Terraform 미설치 환경에서 테스트가 skip되는 조건도 함께 적어 두면 재현성 문서로서 더 정직해집니다.
