# 접근 기록

## 핵심 선택

- Terraform을 배포 도구보다 “분석 가능한 선언”으로 다루기로 했습니다.
- insecure와 secure를 같은 리소스 종류로 맞추어 두고, 설정값만 바뀌게 설계했습니다.
- plan JSON을 직접 아티팩트로 남기고, Python 검증 코드가 그 결과를 읽게 했습니다.

## 버린 대안

- 실제 AWS 계정에 apply해서 콘솔 결과를 확인하는 방식은 포기했습니다. 재현성과 비용 측면에서 학습 효율이 떨어집니다.
- Terraform 지식 전부를 다루지 않고, S3/SG/IAM policy처럼 보안 차이가 분명한 리소스만 남겼습니다.
- Shell 스크립트만으로 검증을 끝내지 않고 pytest 경로를 남겨, 이후 전체 `make test-integration` 흐름에 포함되도록 했습니다.

## 다음 프로젝트와의 연결

이 프로젝트의 핵심 산출물은 Terraform 코드 자체보다 plan JSON입니다. `05-cspm-rule-engine`은 바로 이 구조를
읽어 finding을 만듭니다. 그래서 이 단계에서는 “어떤 리소스를 선언했나”보다 “어떤 JSON 필드로 바뀌는가”를
계속 의식하는 것이 중요합니다.

## 다시 써도 유지할 기준

- 로컬에서 끝까지 재현 가능해야 합니다.
- insecure/secure 비교가 한눈에 보여야 합니다.
- 보안 분석 관점의 입력 구조가 우선입니다.
