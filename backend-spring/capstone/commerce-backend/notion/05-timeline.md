# Timeline — 소스코드에서는 드러나지 않는 것들

## 프로젝트 초기화

이 캡스톤은 랩들과 동일한 scaffold 구조에서 시작했지만, 도메인 패키지 구조가 다르다.

```
src/main/java/com/webpong/study2/app/
├── commerce/
│   ├── api/           ← CommerceController, CommerceAuthController
│   ├── application/   ← CommerceService
│   └── domain/        ← 3 Entities, 3 Repositories
└── global/            ← 공통 모듈 (다른 랩들과 동일)
```

```bash
./gradlew --version    # Gradle 8.13
./gradlew build        # 초기 빌드 확인
```

## Docker Compose 설정

이 캡스톤의 compose.yaml은 모든 랩의 인프라를 합쳐놓은 형태이다:

```bash
docker compose up -d

# 5개 서비스가 기동된다
# postgres  — 5432 (commerce DB)
# redis     — 6379 (현재 코드에서 미사용)
# mailpit   — 1025(SMTP), 8025(UI) (현재 코드에서 미사용)
# redpanda  — 9092 (현재 코드에서 미사용)
# app       — 8080 (Spring Boot)
```

Redis, Mailpit, Redpanda가 compose.yaml에 있지만 코드에서 연결하지 않는다. "인프라는 준비되었으나 코드 연결은 v2의 과제"라는 상태이다.

## Flyway 마이그레이션

```
src/main/resources/db/migration/
├── V1__init.sql       ← 빈 마커 파일
└── V2__commerce.sql   ← 세 테이블 생성
```

V2__commerce.sql이 핵심이다:

```sql
-- 상품: 이름, 가격(12자리 소수점 2자리), 재고, 낙관적 락 버전
create table commerce_products (
    id bigint generated by default as identity primary key,
    name varchar(255) not null,
    price numeric(12, 2) not null,
    stock integer not null,
    version bigint not null default 0
);

-- 장바구니: 고객 이메일 + 상품 ID + 수량
create table commerce_cart_items (
    id bigint generated by default as identity primary key,
    customer_email varchar(255) not null,
    product_id bigint not null,
    quantity integer not null
);

-- 주문: 고객 이메일 + 총 수량 + 상태
create table commerce_orders (
    id bigint generated by default as identity primary key,
    customer_email varchar(255) not null,
    total_quantity integer not null,
    status varchar(50) not null
);
```

주목할 점: `commerce_cart_items.product_id`에 FK 제약이 없다. 엔티티 관계를 JPA 레벨에서 관리하지 않고, `CommerceService`에서 `productRepository.findById()`로 직접 조회한다.

## 테스트 시나리오

`CommerceApiTest.catalogCartAndOrderFlowWork()`는 전체 구매 흐름을 하나의 테스트로 검증한다:

```bash
./gradlew test --tests "*.CommerceApiTest"

# 테스트 흐름:
# 1. POST /auth/login              → accessToken 받기
# 2. POST /admin/products           → Monitor 상품 등록 (price: 199.99, stock: 4)
# 3. GET  /products                 → 상품 목록에 Monitor 존재 확인
# 4. POST /cart/items               → 장바구니에 Monitor 1개 추가
# 5. POST /orders?customerEmail=... → checkout → status: PLACED
# 6. GET  /admin/orders             → 주문 목록에 buyer@example.com 확인
```

H2 in-memory DB에서 실행되므로 PostgreSQL 없이 통과한다.

```bash
make test      # 전체 테스트
make lint      # Spotless + Checkstyle
make smoke     # health check
```

## application.yml 프로파일

- **기본 (H2)**: 로컬 개발과 테스트 용. `jdbc:h2:mem:study2-lab;MODE=PostgreSQL`
- **docker**: PostgreSQL 16 연결. `docker compose up` 시 자동 적용

```yaml
spring:
  config:
    activate:
      on-profile: docker
  datasource:
    url: jdbc:postgresql://${POSTGRES_HOST}:${POSTGRES_PORT}/${POSTGRES_DB}
```

## Git 이력에서 주목할 커밋 순서

1. **scaffold 생성**: global/ 패키지 + build.gradle.kts + compose.yaml
2. **V2__commerce.sql**: 세 테이블 정의
3. **엔티티 3개**: CommerceProductEntity, CartItemEntity, OrderEntity
4. **리포지토리 3개**: JpaRepository 확장
5. **CommerceService**: createProduct, listProducts, addCartItem, checkout, listOrders
6. **CommerceController + CommerceAuthController**: API 엔드포인트
7. **CommerceApiTest**: 전체 흐름 통합 테스트
8. **문서화**: docs/README.md, notion/ 파일들
