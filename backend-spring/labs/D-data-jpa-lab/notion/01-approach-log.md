# Approach Log — 설계 방향을 잡기까지

## 세 가지 선택지 앞에서

JPA 랩을 어떤 방향으로 만들 것인지 고민할 때, 세 가지 길이 보였다.

**첫 번째 길: Querydsl을 처음부터 전면적으로 사용하는 방식.** 동적 검색, 복잡한 조인, 조건 조합까지 모두 보여주면 학습량은 많아지겠지만, scaffold 단계의 코드가 지나치게 복잡해진다. 코드를 처음 읽는 사람이 "JPA 기본 구조"를 파악하기 전에 Querydsl의 `BooleanExpression`과 `JPAQueryFactory`에 먼저 압도당할 위험이 있었다.

**두 번째 길: 순수 CRUD만 노출하는 방식.** `save()`, `findAll()`, `findById()`, `delete()` — 이것만으로도 동작하는 API는 만들 수 있다. 하지만 이 정도라면 Spring Data JPA의 진짜 tradeoff를 전혀 보여주지 못한다. entity와 service의 경계가 왜 중요한지, 동시 수정은 어떻게 처리하는지 — 이런 질문이 빠진 CRUD는 학습 목적에 맞지 않았다.

**세 번째 길: 하나의 aggregate를 중심으로 entity/repository/service 구조를 뚜렷하게 보여주는 방식.** 범위는 작지만, 각 계층이 왜 필요한지, 어디에서 책임이 나뉘는지가 코드에 드러난다. 여기에 낙관적 락과 페이지네이션을 더하면, CRUD를 넘어서는 설계 결정이 자연스럽게 포함된다.

세 번째 길을 택했다.

## 선택한 설계 방향

### 패키지 구조: 계층이 보이는 코드

`data/domain/`에 `ProductEntity`와 `ProductRepository`, `data/application/`에 `DataApiService`, `data/api/`에 `DataApiController`를 배치했다. JPA entity가 직접 API 응답으로 나가지 않도록 `ProductResponse`라는 record를 service 안에 두어서, entity → response 변환이 어디서 일어나는지 명확하게 했다.

이 구조가 중요한 이유는, entity에 비즈니스 로직을 밀어 넣으면 JPA 매핑 관심사와 도메인 규칙이 뒤섞이기 때문이다. `ProductEntity`에는 `changePrice()`라는 도메인 메서드가 하나 있는데, 이것이 entity가 가질 수 있는 행동의 최소 단위다. 가격 변경의 검증(version 확인)은 service가 맡는다.

### 영속성 전략: Flyway + JPA validate

스키마는 Flyway 마이그레이션(`V2__lab_products.sql`)이 관리하고, Hibernate는 `ddl-auto: validate` 모드로 검증만 수행한다. 이 조합은 "스키마 변경은 SQL로 명시적으로, ORM 매핑은 validation으로 안전하게"라는 원칙을 따른다. `V1__init.sql`은 공유 scaffold의 마커 테이블이고, `V2__lab_products.sql`이 이 랩의 실제 스키마다.

```sql
create table if not exists lab_products (
    id bigint generated by default as identity primary key,
    name varchar(255) not null,
    price numeric(12, 2) not null,
    version bigint not null default 0
);
```

`price`에는 `numeric(12, 2)`를 사용해서 소수점 정밀도를 보장한다. `version` 컬럼은 기본값 0으로 시작하며, JPA의 `@Version`과 연동된다.

### 보안 경계: 의도적으로 열어 둔다

`SecurityConfig`는 CSRF를 비활성화하고 모든 요청을 permitAll로 설정한다. 이 랩의 목적은 데이터 영속성 설계이지 인증이 아니다. 보안을 다루는 것은 A-auth-lab이나 B-federation-security-lab의 역할이다.

### OSIV: 명시적으로 꺼 둔다

`spring.jpa.open-in-view: false`를 설정했다. Open Session In View는 controller 레이어에서도 lazy loading이 동작하게 해주는 편의 기능이지만, 트랜잭션 경계를 모호하게 만든다. 이 랩에서는 "트랜잭션은 service 레이어에서 시작하고 끝난다"는 원칙을 유지하기 위해 OSIV를 끈다.

## 폐기한 아이디어들

- **전체 카탈로그 그래프를 첫 scaffold에 넣기**: product → category → review라는 3단 관계를 처음부터 구현하려 했으나, entity 수가 늘어나면 N+1과 fetch join 같은 새로운 문제가 동시에 등장한다. 지금은 하나의 entity로 기본 구조를 확립하는 것이 순서상 맞다.
- **순수 Querydsl showcase**: Querydsl을 설치하고 동적 검색을 전면에 내세우는 방향. 하지만 ProductRepository가 `JpaRepository<ProductEntity, Long>`만 상속하는 현재 단계에서는 보여줄 복잡도가 충분하지 않다.

## 근거 자료

이 방향을 잡는 과정에서 docs/README.md와 problem/README.md를 확인했다. docs/README.md는 현재 구현된 범위(Flyway-managed `lab_products`, JPA entity/repository/service, page-based listing, optimistic locking)와 아직 다루지 않는 것(Querydsl 심화, soft delete, N+1)을 명시하고 있어서, 문서와 코드 사이의 정합성을 유지하는 데 기준이 되었다.
