# Timeline — D-data-jpa-lab 개발 과정

이 문서는 소스 코드에서 드러나지 않는 개발 과정을 시간 순서대로 기록한다. CLI 명령어, 패키지 설치, 설정 변경, 디버깅 과정 등 "코드를 읽는 것만으로는 알 수 없는 것들"을 담는다.

## Phase 1: 프로젝트 초기 설정

### Gradle 프로젝트 scaffold 구성

공유 scaffold를 기반으로 D-data-jpa-lab 디렉토리를 생성했다. build.gradle.kts는 다른 랩과 동일한 기본 의존성(Spring Boot starters, Flyway, Redis, Kafka, Mailpit 등)을 포함하며, 여기에 JPA 특화 의존성을 추가했다.

```bash
# build.gradle.kts에 JPA 관련 의존성 확인
# spring-boot-starter-data-jpa — JPA + Hibernate
# querydsl-jpa:5.1.0:jakarta — Querydsl JPA 지원
# querydsl-apt:5.1.0:jakarta — Querydsl annotation processor
# jakarta.annotation-api — annotation processing 지원
# jakarta.persistence-api — JPA annotation processing 지원
```

Querydsl의 annotation processor 설정이 중요한 부분이었다. `querydsl-apt`를 `annotationProcessor`로 등록하고, `jakarta.annotation-api`와 `jakarta.persistence-api`도 `annotationProcessor`에 추가해야 `Q` 클래스가 정상 생성된다.

```kotlin
annotationProcessor("com.querydsl:querydsl-apt:${querydslVersion}:jakarta")
annotationProcessor("jakarta.annotation:jakarta.annotation-api")
annotationProcessor("jakarta.persistence:jakarta.persistence-api")
```

## Phase 2: 스키마 설계와 Flyway 마이그레이션

### V1__init.sql — 공유 마커 테이블

```sql
-- src/main/resources/db/migration/V1__init.sql
create table if not exists study2_marker (
    id bigint primary key,
    context varchar(100) not null
);
```

이 파일은 모든 랩에서 공유하는 scaffold의 일부다. Flyway가 정상 동작하는지 확인하는 마커 역할이다.

### V2__lab_products.sql — 제품 테이블

```sql
-- src/main/resources/db/migration/V2__lab_products.sql
create table if not exists lab_products (
    id bigint generated by default as identity primary key,
    name varchar(255) not null,
    price numeric(12, 2) not null,
    version bigint not null default 0
);
```

설계 결정 포인트:
- `generated by default as identity`: PostgreSQL의 IDENTITY 컬럼. H2의 PostgreSQL 호환 모드에서도 동작한다.
- `numeric(12, 2)`: 금액에 부동소수점(float, double)을 쓰지 않고 정확한 소수점 표현을 사용한다.
- `version bigint not null default 0`: JPA `@Version`과 연동되는 낙관적 락 컬럼. 새 레코드의 기본값은 0이다.

## Phase 3: JPA Entity 설계

### ProductEntity 작성

```bash
# 패키지 구조
# data/domain/ProductEntity.java
# data/domain/ProductRepository.java
# data/application/DataApiService.java
# data/api/DataApiController.java
```

entity를 domain 패키지에, repository도 domain 패키지에, service를 application 패키지에, controller를 api 패키지에 배치했다. 이 구조는 Hexagonal Architecture의 축약판이다.

Entity 설계 시 고민한 점들:
- `protected ProductEntity() {}` — JPA spec이 요구하는 no-arg constructor. `protected`로 해서 외부에서 기본 생성자를 직접 호출하지 못하게 했다.
- setter를 두지 않고 `changePrice()` 메서드를 제공했다. 가격 변경이라는 의도가 메서드 이름에 드러난다.
- `BigDecimal price` — Java에서 금액을 다룰 때의 표준 타입. `@Column(precision = 12, scale = 2)`로 DB 컬럼과 정확히 매핑한다.

## Phase 4: application.yml 설정

### 로컬 프로파일 (기본)

```yaml
spring:
  datasource:
    url: jdbc:h2:mem:study2-lab;MODE=PostgreSQL;DB_CLOSE_DELAY=-1;DB_CLOSE_ON_EXIT=FALSE
  jpa:
    open-in-view: false
    hibernate:
      ddl-auto: validate
```

핵심 설정 세 가지:
1. **H2 PostgreSQL 호환 모드**: `MODE=PostgreSQL`로 H2가 PostgreSQL 문법을 지원하게 한다. `generated by default as identity` 같은 PostgreSQL 전용 문법이 H2에서도 동작한다.
2. **OSIV 비활성화**: `open-in-view: false`는 Spring Boot의 기본값(`true`)을 명시적으로 끈다. 기본값을 그대로 두면 WARNING 로그가 출력된다.
3. **DDL validate**: Flyway가 관리하는 스키마와 entity 클래스의 매핑이 일치하는지만 확인한다.

### Docker 프로파일

```yaml
spring:
  config:
    activate:
      on-profile: docker
  datasource:
    url: jdbc:postgresql://${POSTGRES_HOST}:${POSTGRES_PORT}/${POSTGRES_DB}
```

Docker 환경에서는 PostgreSQL 16에 직접 연결한다. compose.yaml에서 `POSTGRES_HOST`, `POSTGRES_PORT`, `POSTGRES_DB` 환경변수를 주입한다.

## Phase 5: Service와 Controller 구현

### DataApiService — 트랜잭션 경계의 핵심

```bash
# @Transactional이 붙은 메서드 목록
# create(name, price) — @Transactional
# list(page, size) — @Transactional(readOnly = true)
# updatePrice(productId, version, price) — @Transactional
```

`list()` 메서드에 `@Transactional(readOnly = true)`를 사용한 이유: Hibernate가 dirty checking을 수행하지 않으므로 조회 성능이 조금 더 낫고, 의도를 코드에 명시할 수 있다.

`updatePrice()`에서 version을 수동 비교하는 이유: JPA `@Version`의 `OptimisticLockException`은 flush 시점에 발생하는데, 이것을 그대로 두면 500 에러로 처리될 수 있다. 수동 비교를 먼저 하면 `IllegalArgumentException`으로 `GlobalExceptionHandler`가 400 Bad Request + ProblemDetail 응답을 만들어준다.

### DataApiController — 입력 검증

```bash
# 요청/응답 record 정의
# CreateRequest(@NotBlank name, @Min(0) price)
# UpdateRequest(@Min(0) price, long version)
```

`@Validated`를 controller 클래스에 적용했다. `@NotBlank`, `@Min(0)` 같은 Bean Validation 어노테이션이 동작한다. 검증 실패 시 `GlobalExceptionHandler`의 `handleValidation()`이 ProblemDetail을 반환한다.

## Phase 6: 테스트 작성과 실행

### DataApiTest — 전체 흐름 검증

```bash
# 테스트 실행
./gradlew test

# 또는 Makefile 사용
make test
```

`productCrudAndConflictCheckWork()` 테스트의 흐름:

```
1. POST /api/v1/products {"name":"Keyboard","price":129.99}
   → 200 OK, id와 version:0 반환

2. GET /api/v1/products
   → 200 OK, content[0].name = "Keyboard"

3. PATCH /api/v1/products/{id} {"price":149.99,"version":0}
   → 200 OK, price = 149.99 (version은 1로 증가)

4. PATCH /api/v1/products/{id} {"price":159.99,"version":0}
   → 400 Bad Request, code = "bad_request" (stale version)
```

4번 단계가 중요하다. version 0으로 다시 요청하면 이미 version이 1로 증가했으므로 거부된다. 이것이 낙관적 락의 핵심 동작이다.


## Phase 8: 문서 정리와 keyword inflation 점검

### docs/README.md 작성

구현 범위를 세 영역으로 나누었다:
- "Implemented now" — 현재 코드에서 동작이 검증된 기능
- "Important simplifications" — 코드에 있지만 깊이가 제한된 부분
- "Next improvements" — 아직 구현하지 않은 확장 방향

이 구분이 keyword inflation을 방지하는 핵심이다. Querydsl이 build.gradle.kts에 있다고 해서 "Querydsl을 구현했다"고 적지 않는다.

### notion/ 문서 체계

각 문서의 역할:
- 00-problem-framing: 왜 이 랩이 존재하는가
- 01-approach-log: 어떤 설계 방향을 택했는가
- 02-debug-log: 어떤 문제를 만나고 어떻게 대응했는가
- 03-retrospective: 무엇이 나아지고 무엇이 아직 약한가
- 04-knowledge-index: 재사용 가능한 개념과 용어 정리
- 05-timeline: 이 문서. 소스 코드에서 보이지 않는 개발 과정
