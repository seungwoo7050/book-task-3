# Timeline — E-event-messaging-lab 개발 과정

이 문서는 소스 코드에서 드러나지 않는 개발 과정을 시간 순서대로 기록한다. CLI 명령어, 인프라 설정, 설계 판단의 배경 등 "코드를 읽는 것만으로는 알 수 없는 것들"을 담는다.


## Phase 2: Outbox 테이블 스키마 설계

### V2__outbox_events.sql 작성

```sql
-- src/main/resources/db/migration/V2__outbox_events.sql
create table if not exists outbox_events (
    id bigint generated by default as identity primary key,
    aggregate_type varchar(100) not null,
    aggregate_id varchar(100) not null,
    event_type varchar(100) not null,
    payload varchar(2000) not null,
    status varchar(50) not null
);
```

스키마 설계 시 고민한 점들:

- **aggregate_type + aggregate_id**: 이 두 필드가 있으면 "어떤 도메인 객체에서 이벤트가 발생했는가"를 추적할 수 있다. 나중에 topic routing이나 이벤트 필터링에 사용할 수 있다.
- **payload varchar(2000)**: 단순화된 선택. 프로덕션에서는 `jsonb`(PostgreSQL) 또는 별도의 이벤트 스토어를 고려해야 한다.
- **status varchar(50)**: 현재는 PENDING과 PUBLISHED만 사용하지만, FAILED, RETRY_PENDING 등으로 확장할 여지를 남겼다.
- **created_at이 없다**: 이벤트 생성 시점을 기록하는 timestamp 컬럼이 없다. 후속 개선에서 추가할 필요가 있다.

## Phase 3: OutboxEventEntity 작성

### Entity 설계

```bash
# 패키지 구조
# events/domain/OutboxEventEntity.java
# events/domain/OutboxEventRepository.java
# events/application/EventMessagingService.java
# events/api/EventMessagingController.java
```

Entity 설계 포인트:
- `protected OutboxEventEntity() {}` — JPA no-arg constructor
- setter 대신 `markPublished()` 메서드 하나만 제공. 상태 전환의 의도가 메서드 이름에 드러난다.
- 5개의 필드: aggregateType, aggregateId, eventType, payload, status. 모두 String 타입으로 단순화했다.

### Repository에 커스텀 쿼리 추가

```java
public interface OutboxEventRepository extends JpaRepository<OutboxEventEntity, Long> {
    List<OutboxEventEntity> findByStatus(String status);
}
```

`findByStatus("PENDING")`으로 발행 대기 중인 이벤트를 조회한다. Spring Data JPA의 메서드 이름 기반 쿼리 생성이 사용된다.

## Phase 4: Service 구현 — 세 개의 메서드

### emitOrderPlaced — 이벤트 생성

```java
@Transactional
public EventResponse emitOrderPlaced(String orderId) {
    OutboxEventEntity entity = outboxEventRepository.save(
        new OutboxEventEntity("ORDER", orderId, "ORDER_PLACED",
            "{\"orderId\":\"" + orderId + "\"}", "PENDING"));
    return EventResponse.from(entity);
}
```

payload를 JSON 문자열로 직접 구성한다. Jackson의 `ObjectMapper`를 사용하지 않은 것은 의도적 단순화이다. 프로덕션에서는 DTO를 직렬화하는 것이 안전하다.

### publishPending — 상태 전환

```java
@Transactional
public List<EventResponse> publishPending() {
    List<OutboxEventEntity> pending = outboxEventRepository.findByStatus("PENDING");
    pending.forEach(OutboxEventEntity::markPublished);
    return pending.stream().map(EventResponse::from).toList();
}
```

이 메서드가 "발행"이라고 불리지만, 실제로 Kafka에 메시지를 보내지는 않는다. JPA의 dirty checking이 `markPublished()` 호출로 변경된 status를 트랜잭션 종료 시 자동으로 UPDATE한다.

### list — 조회

```java
@Transactional(readOnly = true)
public List<EventResponse> list() {
    return outboxEventRepository.findAll().stream().map(EventResponse::from).toList();
}
```

## Phase 5: Controller와 API 설계

### 엔드포인트 구조

```
POST /api/v1/orders/{orderId}/events  — 주문 이벤트 생성, PENDING 상태로 저장
POST /api/v1/outbox-events/publish    — PENDING 이벤트들을 PUBLISHED로 전환
GET  /api/v1/outbox-events            — 전체 이벤트 목록 조회
```

첫 번째 엔드포인트는 RESTful 리소스 계층(`/orders/{orderId}/events`)을 따른다. 주문(order)의 하위 리소스로 이벤트를 생성한다는 의미이다.

## Phase 6: Compose 스택 — Redpanda 추가

### compose.yaml에 Redpanda 컨테이너 추가

```yaml
redpanda:
  image: redpandadata/redpanda:v24.3.8
  command:
    - redpanda start --overprovisioned --smp 1 --memory 512M
    - --kafka-addr INTERNAL://0.0.0.0:9092,EXTERNAL://0.0.0.0:19092
    - --advertise-kafka-addr INTERNAL://redpanda:9092,EXTERNAL://localhost:19094
  ports:
    - "19094:19092"
```

Redpanda 설정 포인트:
- `--overprovisioned --smp 1 --memory 512M`: 로컬 개발용으로 리소스를 제한한다
- **INTERNAL / EXTERNAL 리스너**: 컨테이너 간 통신은 `redpanda:9092`, 호스트에서의 접근은 `localhost:19094`
- 앱 컨테이너의 환경변수: `KAFKA_BOOTSTRAP_SERVERS: redpanda:9092`

```bash
# Docker Compose 실행
docker compose up -d

# Redpanda 상태 확인 (redpanda 컨테이너에서)
# rpk cluster info
```

## Phase 9: 문서와 keyword inflation 점검

D-data-jpa-lab과 같은 패턴으로, 현재 구현과 다음 단계를 명확히 분리하는 문서를 작성했다.

이 랩에서 특히 주의한 것은 "Kafka를 쓴다"는 표현의 범위이다. build.gradle.kts에 `spring-kafka`가 있고 compose.yaml에 Redpanda가 있지만, 실제로 Kafka 토픽에 메시지를 발행하는 코드는 없다. 이 간극을 문서에 솔직하게 적는 것이 keyword inflation을 막는 핵심이다.
