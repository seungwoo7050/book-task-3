# Approach Log — Outbox를 먼저 배우는 이유

## 세 가지 선택지

이벤트 메시징 랩을 어떻게 구성할 것인지 고민할 때, 세 가지 방향이 있었다.

**첫 번째 길: Kafka API를 직접 사용하는 방식.** `KafkaTemplate.send()`를 호출하면 빠르게 "Kafka를 쓴다"는 결과를 얻을 수 있다. 하지만 이 방식은 DB 트랜잭션과 메시지 발행 사이의 경계를 무시한다. 비즈니스 로직이 성공하고 DB에 커밋된 후에 Kafka 발행이 실패하면 어떻게 되는가? 이 질문이 빠진 채로 Kafka를 쓰는 것은 위험하다.

**두 번째 길: Outbox를 먼저 두는 방식.** DB 트랜잭션 안에서 outbox 테이블에 이벤트를 저장하고, 별도의 프로세스가 outbox에서 읽어 Kafka에 발행하는 구조. 구현할 것이 더 많지만, "메시지 발행의 안전한 경계"라는 핵심 개념을 코드로 보여줄 수 있다.

**세 번째 길: DLQ와 재시도까지 한 번에 넣는 방식.** 가장 완성도 높은 구현이지만, scaffold 수준에서는 메시지 브로커의 신뢰성 메커니즘과 DB 트랜잭션 경계를 동시에 설명해야 해서 복잡도가 급격히 올라간다.

두 번째 길을 택했다.

## 선택한 설계 방향

### 패키지 구조: event의 생명주기가 보이는 코드

`events/domain/`에 `OutboxEventEntity`와 `OutboxEventRepository`, `events/application/`에 `EventMessagingService`, `events/api/`에 `EventMessagingController`를 배치했다. D-data-jpa-lab과 같은 계층 구조를 따르면서, "이벤트"를 다루는 도메인의 특성이 드러나게 했다.

`OutboxEventEntity`는 다섯 개의 필드를 가진다:
- `aggregateType`: 이벤트가 발생한 도메인 (예: "ORDER")
- `aggregateId`: 해당 도메인의 식별자 (예: "ORDER-1")
- `eventType`: 발생한 이벤트의 종류 (예: "ORDER_PLACED")
- `payload`: 이벤트 데이터를 담은 JSON 문자열
- `status`: PENDING 또는 PUBLISHED

entity에는 `markPublished()` 메서드가 하나 있다. 상태를 PUBLISHED로 바꾸는 것이 이 entity가 가진 유일한 행동이다. 이것은 의도적이다 — outbox entity의 책임은 "상태를 안전하게 전환하는 것"이지, 실제 발행 로직을 포함하는 것이 아니다.

### 영속성 선택: outbox row를 durable handoff boundary로 사용

핵심 아이디어는 "메시지 발행을 DB 트랜잭션의 일부로 만들자"는 것이다.

```java
@Transactional
public EventResponse emitOrderPlaced(String orderId) {
    OutboxEventEntity entity = outboxEventRepository.save(
        new OutboxEventEntity("ORDER", orderId, "ORDER_PLACED",
            "{\"orderId\":\"" + orderId + "\"}", "PENDING"));
    return EventResponse.from(entity);
}
```

이 메서드가 성공하면 outbox 행이 DB에 커밋된다. 이 행이 존재한다는 것 자체가 "이 이벤트는 반드시 발행되어야 한다"는 계약이다. 실제 발행이 나중에 실패해도, outbox에 PENDING 행이 남아 있으므로 재시도할 수 있다.

### 스키마 설계: outbox_events 테이블

```sql
create table if not exists outbox_events (
    id bigint generated by default as identity primary key,
    aggregate_type varchar(100) not null,
    aggregate_id varchar(100) not null,
    event_type varchar(100) not null,
    payload varchar(2000) not null,
    status varchar(50) not null
);
```

`payload`를 `varchar(2000)`으로 둔 것은 단순화이다. 프로덕션에서는 `jsonb` 타입이나 별도의 이벤트 스토어를 고려해야 한다. 현재는 학습 목적으로 페이로드 구조를 JSON 문자열로 보여주는 것이 목적이다.

### 보안과 인증: 이 랩의 관심사가 아니다

SecurityConfig는 모든 요청을 permitAll로 열어 둔다. 이 랩의 초점은 메시징 경계이지 인증이 아니다.

### Compose 스택: Redpanda 포함

Docker Compose에 Kafka 호환 브로커인 Redpanda v24.3.8이 포함되어 있다. Redpanda는 JVM 없이 Kafka 프로토콜을 구현하므로 로컬 개발 환경에서 가볍게 실행할 수 있다.

현재 코드가 실제로 Redpanda에 메시지를 보내지는 않지만, compose.yaml에 브로커가 포함되어 있으므로 scheduled publisher를 추가하면 바로 연결할 수 있는 상태다. `KAFKA_BOOTSTRAP_SERVERS: redpanda:9092`가 앱에 주입된다.

## 폐기한 아이디어들

- **Direct publish only**: `KafkaTemplate`으로 바로 발행하는 방식. DB 트랜잭션과 메시지 발행 사이의 경계가 무시되므로 폐기했다.
- **Full retry/DLQ orchestration을 scaffold 필수로 두기**: 재시도 횟수, backoff 전략, DLQ 토픽 설정까지 포함하면 스프링 설정만으로 상당한 분량이 된다. scaffold 단계에서는 outbox라는 핵심 개념에 집중하는 것이 맞다.
