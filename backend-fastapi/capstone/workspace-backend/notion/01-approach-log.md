# 접근 로그

## 고려한 선택지

- 개별 랩 코드를 직접 import 해서 조립하는 방식
- 각 개념을 capstone 도메인에 맞게 다시 구현하는 방식
- 인증, 인가, 알림, 실시간 전달을 일부만 통합하는 방식
- 하나의 협업 도메인에서 전부 다시 엮는 방식

## 채택한 방향

- 랩 코드를 공용 패키지로 만들지 않고, capstone 안에서 개념을 다시 구현했다.
- 사용자, 워크스페이스, 프로젝트, 댓글, 알림을 하나의 협업형 도메인으로 묶었다.
- 큐와 실시간 전달을 모두 넣되, 프런트엔드와 클라우드 인프라는 제외했다.

## 버린 대안

- 랩 코드를 그대로 import 하는 방식은 "통합 설계"보다 "예제 모음"에 가까워 보여서 채택하지 않았다.
- 너무 많은 기능을 추가하는 방향은 capstone의 경계를 흐리므로 피했다.

## 트레이드오프

- 장점: 앞선 랩의 패턴을 통합 구조로 재해석하는 능력을 보여 줄 수 있다.
- 단점: 각 랩의 단순함이 사라져 설정 충돌과 모델 충돌이 더 많이 생긴다.

이 선택 덕분에 capstone은 예제 모음이 아니라, "작은 패턴을 제품 구조로 재해석한 결과물"로 읽히게 된다.
