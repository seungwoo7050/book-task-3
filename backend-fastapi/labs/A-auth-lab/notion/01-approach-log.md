# 접근 로그

## 고려한 선택지

- access token만 쓰는 단순 세션 모델
- access token + refresh token을 분리하는 모델
- 실제 SMTP를 붙이는 메일 흐름
- Mailpit 중심의 로컬 메일 흐름

## 채택한 방향

- access token과 rotating refresh token을 분리했다.
- cookie 인증을 유지하되, 상태 변경 요청에는 CSRF 검증을 붙였다.
- 메일 흐름은 실 SMTP 대신 Mailpit 중심 로컬 검증으로 제한했다.

## 버린 대안

- OAuth와 2FA를 한 랩 안에 같이 넣는 방향은 범위가 과해져서 버렸다.
- 문서에 마이그레이션 선행 단계를 강하게 요구하는 방식은 학습 시작 장벽이 높아져서 피했다.

## 트레이드오프

- 장점: 로컬 인증의 핵심 경계를 한 랩 안에서 끝까지 설명할 수 있다.
- 단점: 실서비스형 메일 인프라, 공급자 장애, 다중 인증 경로 같은 현실성은 의도적으로 줄였다.

실제로 구현할 때는 토큰 수명, 쿠키 설정, CSRF 헤더 요구사항을 표로 먼저 정리해 두는 편이 가장 덜 흔들린다.
