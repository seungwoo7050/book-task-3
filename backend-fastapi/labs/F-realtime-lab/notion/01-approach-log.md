# 접근 로그

## 고려한 선택지

- 실시간 기능을 capstone 안에서만 다루는 방식
- 별도 랩으로 WebSocket과 presence만 분리하는 방식
- 처음부터 Redis 의존 fan-out을 구현하는 방식
- 인메모리 연결 관리에서 시작하는 방식

## 채택한 방향

- WebSocket, presence, fan-out을 별도 랩으로 분리했다.
- 연결 관리와 presence 저장은 인메모리 구조로 시작했다.
- Redis는 "다음 확장 경계"로 열어 두되, 핵심 로직은 단일 프로세스 설명에 집중했다.

## 버린 대안

- 처음부터 broker 기반 수평 확장을 구현하는 방향은 학습 비용이 너무 커서 제외했다.
- JWT와 제품 도메인을 같이 붙이는 방향은 연결 모델 설명을 흐리게 만들어 버렸다.

## 트레이드오프

- 장점: 실시간 연결의 핵심 개념을 작은 코드베이스에서 설명하기 쉽다.
- 단점: 프로세스 수평 확장과 실제 보안성은 이후 단계로 남는다.

처음부터 broker를 붙이지 않은 이유는, 단일 프로세스 모델만으로도 연결 수명주기와 fan-out의 핵심을 충분히 설명할 수 있기 때문이다.
