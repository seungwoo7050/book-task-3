# 문제 프레이밍

## 학습 목표

HTTP 요청과 다른 생명주기를 가진 WebSocket 연결, presence 상태, fan-out 전달을 별도 주제로 분리해 이해하는 것이 목표다.

## 왜 중요한가

- 실시간 기능은 API 하나 더 만드는 문제로 축소되지 않는다.
- 연결 상태와 사용자 상태, 인메모리와 외부 broker의 경계를 따로 설명할 수 있어야 한다.

## 선수 지식

- HTTP와 WebSocket의 차이
- 비동기 I/O 기본
- Redis pub/sub이 어떤 문제를 해결하는지에 대한 감각

## 성공 기준

- WebSocket 인증과 연결 유지 모델을 설명할 수 있어야 한다.
- presence heartbeat와 TTL 만료가 동작해야 한다.
- 같은 사용자에 대한 다중 연결 fan-out이 가능해야 한다.

## 제외 범위

- cross-process 수평 확장 구현
- replay 보장과 대규모 채팅 모델
- 실서비스형 JWT 검증 완성

이 랩의 목적은 실시간 기능 전체를 완성하는 것이 아니라, 연결 수명주기와 상태 관리라는 핵심 문제를 분리해 이해하게 만드는 데 있다.
