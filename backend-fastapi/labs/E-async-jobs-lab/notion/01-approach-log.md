# 접근 로그

## 고려한 선택지

- 요청 처리 중 바로 작업을 실행하는 단순 방식
- outbox row를 먼저 저장하고 worker가 가져가는 방식
- 실제 Redis broker를 테스트에 그대로 쓰는 방식
- eager mode로 테스트를 단순화하는 방식

## 채택한 방향

- enqueue와 실제 실행을 분리하고, outbox를 handoff 경계로 삼았다.
- 테스트는 eager mode로 유지하되, 로컬 Compose에서는 Redis + Celery worker를 띄운다.
- 상태 모델에 retry와 attempt count를 포함해 실패를 문서화할 수 있게 했다.

## 버린 대안

- 요청 안에서 직접 외부 작업을 끝내는 방식은 실패 복구 설명이 약해져서 버렸다.
- 처음부터 분산 락이나 고급 스케줄링까지 넣는 방향은 학습 범위를 넘어가 제외했다.

## 트레이드오프

- 장점: 비동기 경계와 멱등성 문제를 작은 예제로 명확히 설명할 수 있다.
- 단점: 실제 분산 메시징 환경의 복잡성은 문서 수준으로만 남는다.

설계를 정리할 때는 요청 경로와 worker 경로를 같은 시퀀스로 그려 두면 outbox의 필요성이 훨씬 잘 드러난다.
