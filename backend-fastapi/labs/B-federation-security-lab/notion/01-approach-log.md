# 접근 로그

## 고려한 선택지

- 외부 로그인과 2FA를 분리된 랩으로 나누는 방식
- 외부 로그인 이후 곧바로 세션을 발급하는 단순 방식
- 외부 로그인 후 2FA가 필요한 경우 pending 상태를 두는 방식
- 실제 Google provider에 의존하는 테스트 방식

## 채택한 방향

- OIDC 로그인, 2FA, recovery code, throttling을 하나의 보안 랩으로 묶었다.
- 2FA가 필요한 사용자는 곧바로 access token을 주지 않고 중간 상태를 둔다.
- 테스트는 mock provider로 유지해 Google Cloud 설정에 의존하지 않게 했다.

## 버린 대안

- live provider 테스트는 학습 목표보다 인프라 설정 비용이 커서 버렸다.
- 2FA를 capstone에서만 다루는 방향은 보안 설명이 너무 늦게 등장해서 채택하지 않았다.

## 트레이드오프

- 장점: multi-step auth flow를 명시적으로 설명할 수 있다.
- 단점: 실제 공급자와의 예외 상황이나 운영 이슈는 문서 수준으로만 남는다.

보안 흐름을 설계할 때는 "callback 직후 세션 발급"과 "추가 검증 후 최종 발급"을 그림으로 구분해 두면 구현과 테스트가 훨씬 쉬워진다.
