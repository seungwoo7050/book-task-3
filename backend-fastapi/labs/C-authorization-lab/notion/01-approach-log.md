# 접근 로그

## 고려한 선택지

- 실제 JWT 인증을 포함한 권한 랩
- `X-User-Id` 헤더만으로 actor를 주입하는 단순 랩
- 엔드포인트마다 권한 if문을 두는 방식
- 서비스 계층에 역할 체크를 모으는 방식

## 채택한 방향

- 인증은 헤더 기반 actor 모델로 단순화했다.
- 권한 판단은 서비스 계층에서 역할 순서를 비교하는 방식으로 정리했다.
- workspace, invite, membership, document를 최소 도메인으로 잡아 권한 경계를 드러냈다.

## 버린 대안

- JWT 검증을 함께 넣는 방향은 권한 테스트의 원인을 흐리게 만들어서 버렸다.
- 너무 많은 리소스를 넣는 방향은 역할표 설명을 오히려 어렵게 만들어 배제했다.

## 트레이드오프

- 장점: 인가 규칙만 따로 테스트하고 설명하기 쉽다.
- 단점: 실제 서비스 인증과 결합한 경로는 capstone으로 넘겨야 한다.

실무형 포트폴리오로 확장할 때는 여기서 만든 역할표를 실제 인증 스택과 어떻게 연결할지 별도 메모를 남겨 두는 편이 좋다.
