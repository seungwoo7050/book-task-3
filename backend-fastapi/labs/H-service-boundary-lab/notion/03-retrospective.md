# 회고

## 이번 랩에서 얻은 것

- “서비스 분리”의 첫 기준은 네트워크 호출 수가 아니라 데이터 ownership이라는 점이 분명해졌다.
- 사용자 정보를 claim으로 넘기면 `workspace-service`가 자기 도메인에만 집중할 수 있다.
- 테스트가 경계 설명의 일부가 된다는 점도 드러났다. UUID 형식, claim shape, cross-DB 금지가 모두 테스트에 박혀야 한다.

## 이번 랩의 약점

- 아직 외부 클라이언트는 internal API를 직접 호출한다.
- gateway가 없어서 브라우저 상태와 내부 bearer 계약의 차이를 보여 주지는 못한다.
- 사용자 상세 조회가 필요한 시나리오는 일부러 빼 두었기 때문에 현실적인 불편도 남아 있다.

## 다음 랩으로 넘기는 질문

- claim만으로 부족한 정보는 언제 이벤트나 동기 API로 보완해야 하는가
- DB ownership을 지키면서 알림 같은 후속 처리는 어디에 둘 것인가
- public API shape를 유지하려면 gateway는 어떤 책임을 가져야 하는가

이 랩은 분산 구조의 “시작점”은 만들었지만, 아직 분산 통합은 아니다. 그래서 다음 단계에서 outbox와 gateway가 왜 필요한지 자연스럽게 이어진다.
