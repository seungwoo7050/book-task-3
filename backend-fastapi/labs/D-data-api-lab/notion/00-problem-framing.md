# 문제 프레이밍

## 학습 목표

프로젝트, 태스크, 댓글 같은 전형적인 협업형 데이터를 다루면서, CRUD 이상의 데이터 API 설계 포인트를 익히는 것이 목표다.

## 왜 중요한가

- 실무 데이터 API는 항상 필터링, 정렬, 페이지네이션, 삭제 정책, 동시 수정 충돌을 같이 가진다.
- ORM을 쓰더라도 어떤 규칙은 명시적으로 코드에서 관리해야 한다.

## 선수 지식

- SQLAlchemy 기본 모델링
- REST API와 상태 코드 기본
- 트랜잭션과 버전 충돌 개념

## 성공 기준

- CRUD와 목록 조회 규칙이 일관되게 동작해야 한다.
- page-based pagination과 sort/filter 규칙을 설명할 수 있어야 한다.
- soft delete와 optimistic locking이 실제 충돌 경로를 보여 줘야 한다.

## 제외 범위

- 인증과 인가
- full-text search
- 이벤트 소싱, CQRS 같은 더 무거운 아키텍처 실험

이 랩의 핵심은 CRUD 자체보다 데이터 규칙을 명시적으로 드러내는 데 있다. 나중에 도메인을 바꾸더라도 이 규칙 묶음은 그대로 재사용할 수 있다.
