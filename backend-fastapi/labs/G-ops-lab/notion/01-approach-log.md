# 접근 로그

## 고려한 선택지

- 운영성 항목을 각 랩에 조금씩 흩어 놓는 방식
- health, metrics, CI, 배포 문서를 별도 랩으로 묶는 방식
- Prometheus client나 IaC까지 초기에 다 넣는 방식
- 최소 surface만 구현하고 문서로 한계를 남기는 방식

## 채택한 방향

- 운영성은 별도 랩으로 분리했다.
- health / readiness, metrics, JSON 로그, Compose probe, AWS target shape 문서에 집중했다.
- 과한 인프라 자동화 대신 "어디까지 사실이고 어디부터 가정인지"를 문서로 분리했다.

## 버린 대안

- full observability stack을 처음부터 붙이는 방향은 학습 범위를 지나치게 넓혀서 제외했다.
- live AWS 배포를 완료 조건으로 두는 방향은 계정과 비용에 과도하게 의존하므로 채택하지 않았다.

## 트레이드오프

- 장점: 운영성 문서를 포트폴리오 자산으로 만드는 연습이 된다.
- 단점: 실제 운영 경험 전체를 이 랩만으로 주장할 수는 없다.

운영성 랩에서는 구현 범위를 줄이는 대신, 무엇이 사실이고 무엇이 가정인지 문서에서 더 엄격하게 구분하는 편이 중요하다.
