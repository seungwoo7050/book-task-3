# 접근 로그

## 처음 고려한 선택지

1. Prometheus, Grafana, tracing backend까지 한 번에 붙인다.
2. health endpoint만 두고 운영 문서는 최소화한다.
3. health, request id, JSON 로그, 최소 metrics, Compose health matrix, AWS target shape까지만 다룬다.

## 선택한 방향

세 번째 방식을 택했다. 관측 도구를 많이 붙이기보다 “운영 질문을 시작할 최소 기준”을 설명하는 데 집중했다.

## 그렇게 고른 이유

- 첫 번째 방식은 학습 가치가 크지만 K 랩 하나로는 범위가 너무 넓다.
- 두 번째 방식은 구조는 단순하지만 왜 운영성이 필요한지 설명하기 어렵다.
- 세 번째 방식은 아직 얇지만, 어떤 질문을 어디서 확인해야 하는지 기준을 세우기에 적절하다.

## 의도적으로 단순화한 점

- metrics는 최소 surface만 노출한다.
- request id는 로그 correlation의 최소 단위로만 쓴다.
- AWS 문서는 target shape 설명에만 머물고, 실제 infra 코드는 제공하지 않는다.

## 이번 선택이 만든 제약

- 대시보드나 장기 시계열 관측까지는 보여 주지 못한다.
- 장애 주입과 복구 실험도 최소 수준에 머문다.
- 운영성이 “도구 사용법”보다 “질문 정리”에 더 가깝게 보일 수 있다.
