# Fullstack 자소서

저는 사용자에게 보이는 화면을 만드는 일에서 개발을 시작했지만, 좋은 제품은 결국 화면과 API, 데이터, 검증 체계가 함께 맞물릴 때 완성된다고 생각합니다. 42서울 정규과정을 통해 시스템과 문제 해결의 기초를 다졌고, 이후에는 `mini-vrew`를 직접 배포하며 제품형 UI를 끝까지 완성해 봤습니다. 그 다음 단계로는 현재 워크스페이스의 프론트엔드와 백엔드 프로젝트들을 학습 순서에 맞게 하나씩 구현하며, 사용자 흐름을 앞단과 뒷단을 함께 보는 시야를 키워 왔습니다.

저는 문제를 해결할 때 기능을 화면과 서버로 따로 떼어 보기보다, 하나의 사용자 흐름으로 묶어서 생각하는 편입니다. `Client Onboarding Portal`에서는 sign-in, validation, draft restore, submit retry를 전제로 UI를 구성했고, `Workspace SaaS API`에서는 JWT auth, RBAC, invitation, async notification을 제품형 API로 정리했습니다. 이 과정에서 단순히 기능을 구현하는 것이 아니라 어떤 경계에서 실패가 발생하고 어떻게 복구할지를 함께 설계하는 습관을 길렀습니다.

제가 풀스택 역할에 적합하다고 생각하는 이유는 두 영역을 모두 조금씩 한다는 의미가 아니라, 화면과 서비스 구조를 하나의 문제로 연결해 설명할 수 있기 때문입니다. 프론트엔드에서는 제품형 UI와 브라우저 런타임을, 백엔드에서는 인증, 데이터 모델, 비동기 처리, 재현 가능한 검증 경로를 함께 다뤄 왔습니다. 그래서 새로운 기능을 맡더라도 어느 한쪽만 보는 것이 아니라 전체 흐름에서 어떤 책임이 필요한지 먼저 생각하는 편입니다.

협업에서도 저는 구현만큼 기준을 남기는 일이 중요하다고 생각합니다. 현재 워크스페이스의 프로젝트들을 정리하면서 README, docs, notion, blog를 분리해 둔 이유도 다른 사람이 맥락 없이 코드를 읽지 않도록 하기 위해서였습니다. 팀 안에서도 단순히 일을 처리하는 사람보다, 기능 구현과 함께 검증 기준과 문서화 지점을 같이 만드는 사람으로 기여하고 싶습니다.

입사 후에는 먼저 제품의 핵심 사용자 흐름과 현재 서비스 구조를 빠르게 이해하고, 맡은 기능을 안정적으로 완성하는 데 집중하겠습니다. 그 위에서 제가 쌓아 온 프론트엔드 구현력과 서비스 경계 이해를 바탕으로, 단기적으로는 화면과 API를 함께 믿고 맡길 수 있는 개발자가 되고 싶고, 장기적으로는 제품과 기술을 함께 설계하는 풀스택 엔지니어로 성장하고 싶습니다.
