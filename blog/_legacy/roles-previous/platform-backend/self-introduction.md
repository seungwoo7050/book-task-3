# Platform / Backend 자소서

저는 제품 기능을 구현할 때 화면보다 먼저 서비스 구조를 말하고 싶은 사람은 아니지만, 좋은 서비스는 결국 인증·인가, 데이터 흐름, 실패 처리, 운영 기준이 함께 정리되어야 한다고 믿습니다. 42서울 정규과정을 통해 시스템과 문제 해결의 기초를 다졌고, 이후 현재 워크스페이스에서 Go, FastAPI, Node/NestJS, Spring Boot 프로젝트를 학습 순서에 맞게 구현하며 서비스 구조를 다양한 방식으로 경험해 왔습니다. 지금은 특히 API 경계와 운영성, 비동기 처리까지 설명 가능한 백엔드 결과물을 만드는 데 관심을 두고 있습니다.

저는 문제를 해결할 때 기능 자체보다 "어디서 경계를 끊을 것인가"를 먼저 생각하는 편입니다. `Workspace SaaS API`에서는 JWT auth, RBAC, invitation, worker 분리를 갖춘 제품형 API를 만들었고, `workspace-backend-v2-msa`에서는 gateway와 서비스별 DB ownership, outbox, Redis Streams, websocket fan-out으로 이어지는 분산 경계를 다시 설계했습니다. `collab-saas-security-review`에서는 운영 관점에서 무엇을 먼저 고칠지 정하는 remediation artifact를 CLI와 보고서로 재현할 수 있게 만들었습니다.

제가 플랫폼/백엔드 역할에 적합하다고 생각하는 이유는 프레임워크를 여러 개 써 봤다는 점보다, 서비스 구조를 설명 가능한 단위로 반복해서 만들어 왔기 때문입니다. 단일 API, modular monolith, MSA 학습 버전, CLI artifact 파이프라인처럼 형태는 달라도 공통적으로 인증, 데이터 모델, 비동기 처리, 재현 가능한 검증 경로를 함께 다뤘습니다. 그래서 새로운 서비스 구조를 맡더라도 기능 구현만이 아니라 운영과 검증까지 포함한 기준을 함께 고민하는 편입니다.

협업에서도 저는 구현 결과를 다른 사람이 다시 따라갈 수 있게 남기는 태도가 중요하다고 생각합니다. 지금까지의 프로젝트들을 README, docs, notion, blog로 분리해 둔 이유도 같은 맥락입니다. 팀 안에서도 코드만 작성하는 사람보다, 실행 방법과 검증 기준까지 함께 정리해 팀 전체의 맥락 비용을 줄이는 사람으로 기여하고 싶습니다.

입사 후에는 먼저 현재 서비스의 핵심 도메인과 운영 기준을 빠르게 이해하고, 맡은 API와 서비스 구조를 안정적으로 구현하는 데 집중하겠습니다. 그 위에서 제가 쌓아 온 경계 설계와 재현 가능한 검증 습관을 바탕으로, 단기적으로는 믿고 맡길 수 있는 백엔드 엔지니어가 되고 싶고, 장기적으로는 팀 안에서 플랫폼 관점의 구조와 운영 기준까지 함께 제안할 수 있는 개발자로 성장하고 싶습니다.
