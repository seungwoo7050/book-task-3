# Frontend 자소서

프론트엔드 개발을 좋아하는 이유는 사용자가 바로 체감하는 변화를 가장 직접적으로 만들 수 있기 때문입니다. 하지만 저는 좋은 화면은 단순히 보기 좋게 만드는 것만으로 완성되지 않는다고 생각합니다. 42서울 정규과정을 통해 시스템과 문제 해결의 기초를 다졌고, 이후 `mini-vrew`를 직접 설계하고 배포하면서 브라우저 안에서 무거운 처리와 사용자 경험을 함께 설계하는 개발을 경험했습니다. 그 뒤로는 현재 워크스페이스의 프론트엔드 프로젝트들을 학습 순서에 맞게 하나씩 완성하며 제품형 UI를 설명 가능한 결과물로 정리해 왔습니다.

저는 문제를 해결할 때 기능을 한 번에 크게 쌓기보다, 사용자 흐름을 기준으로 단위를 나누고 각각을 검증 가능한 상태로 만드는 편입니다. `mini-vrew`에서는 플레이어, waveform, trimming, captions, filters, export처럼 서로 얽힌 기능을 책임 단위로 나눠 점진적으로 완성했습니다. `Ops Triage Console`과 `Client Onboarding Portal`에서는 optimistic update, retry, validation, draft restore 같은 상태 전이를 분리해 다루면서 실제 제품 흐름처럼 보이도록 설계했습니다.

제가 프론트엔드 역할에 특히 잘 맞는 이유는 화면을 "예쁘게" 만드는 것에만 관심이 있지 않기 때문입니다. 저는 브라우저 API, 데이터 흐름, 실패 처리, 검증 루프를 함께 봅니다. React, Next.js, TypeScript를 기반으로 제품형 UI를 만드는 것을 좋아하고, Playwright와 테스트 루프를 함께 묶어 품질을 유지하는 방식에 익숙합니다. 동시에 React internals와 시스템 구조까지 꾸준히 학습해, 화면 뒤에 있는 동작 원리도 함께 이해하려고 노력해 왔습니다.

협업에서도 저는 설명 가능한 결과물을 남기는 태도가 중요하다고 생각합니다. 혼자 만든 프로젝트에서도 README, 검증 명령, 문서를 꾸준히 정리해 온 이유는 결국 개발이란 나만 아는 코드를 만드는 일이 아니라 다른 사람이 이어받을 수 있는 자산을 남기는 일이라고 생각하기 때문입니다. 팀 안에서도 구현뿐 아니라 검증 기준과 문서화 지점을 같이 고민하는 사람으로 기여하고 싶습니다.

입사 후에는 먼저 제품과 사용자 흐름을 빠르게 이해하고, 맡은 화면을 안정적으로 구현하는 데 집중하겠습니다. 그 위에서 제가 강점을 가진 제품형 UI 설계와 검증 습관을 바탕으로, 단기적으로는 기능을 믿고 맡길 수 있는 프론트엔드 개발자가 되고 싶고, 장기적으로는 팀 안에서 프론트엔드 품질 기준과 구조를 함께 만드는 개발자로 성장하고 싶습니다.
