# blog-writing-guide

이 문서는 `/Users/woopinbell/work/book-task-3/` 워크스페이스에서 각 독립 프로젝트의 `blog/` 시리즈를 만들 때 따를 공용 기준이다.

이 가이드는 단순히 "글을 더 자연스럽게 써라"를 요구하지 않는다.  
목표는 더 명확하다.

- 어떤 AI를 쓰더라도 품질 편차를 줄인다.
- 블로그가 실제 개발 순서와 판단 변화를 복원하게 만든다.
- 독자가 `무엇을 만들었는지`뿐 아니라 `어떻게 만들었고 왜 그렇게 바꿨는지`를 따라갈 수 있게 만든다.
- 핵심 코드, CLI, 검증 신호, 새로 배운 개념이 한 문서 안에서 연결되게 만든다.

한 문장으로 줄이면 이렇다.

`blog/`는 "코드를 바탕으로 실제 개발 과정을 끝까지 복원하는 탐색형 개발 기록"이어야 한다.

## 이 가이드가 원하는 글

좋은 `blog/` 글은 결과를 정리한 회고문이 아니다.  
좋은 글은 독자가 아래 질문에 답할 수 있게 만든다.

- 처음 무엇부터 만들기 시작했는가
- 구현 순서는 파일 단위로 어떻게 흘렀는가
- 함수 단위로 어디서 설계가 갈렸는가
- 어떤 CLI를 돌리며 범위를 줄였는가
- 어떤 실패나 오해를 거쳐 수정했는가
- 어떤 코드 조각이 판단의 전환점이었는가
- 작업 중 새로 이해한 개념은 무엇이었는가
- 마지막에는 무엇이 해결됐고 무엇이 남았는가

즉, 글의 중심은 `정답 요약`이 아니라 아래 흐름이다.

`시작점 -> 구현 순서 -> 막힌 지점 -> 가설 -> 수정 -> 검증 -> 새로 배운 것 -> 다음 단계`

## 이 가이드가 원하지 않는 글

아래 유형은 기본적으로 실패로 본다.

- 처음부터 답을 알고 있었던 것처럼 쓰는 글
- README를 길게 늘인 설명문
- 커밋 요약을 문장으로 풀어 쓴 글
- 버그 하나만 고친 장면에 과도하게 매달려 전체 구현 흐름을 놓친 글
- 코드 없이 추상 설명만 이어지는 글
- CLI 없이 "테스트했다", "검증했다"라고만 적는 글
- 새로 이해한 개념이 있었는데 설명을 생략한 글

## 적용 단위

기본 적용 단위는 `독립 프로젝트 단위`다.

여기서 `독립 프로젝트`란 아래 조건을 만족하는 가장 작은 작업 단위를 뜻한다.

- 자기 문제 범위를 설명할 수 있다
- 자기 검증 명령 또는 실행 진입점이 있다
- 자기 구현 흐름을 독립적으로 복원할 수 있다

기본값은 아래가 아니다.

- `track 전체`
- `repo 전체`
- 여러 프로젝트를 한 번에 묶은 회고문

즉 `blog/`의 기본값은 언제나 `하나의 프로젝트를 처음부터 끝까지 따라가는 시리즈`다.

## 소스 오브 트루스

글의 근거는 문장 감각이 아니라 작업 흔적이다.  
아래 순서로 근거를 우선한다.

1. 실제 소스코드와 변경 이력
2. CLI 명령과 테스트 실행 흔적
3. 에러 메시지와 검증 출력
4. 프로젝트 README, problem 문서, `docs/`
5. `notion/`, `notion-archive/` 같은 보조 노트
6. 기존 `blog/` 초안 또는 예전 글

중요한 원칙은 간단하다.

- 소스와 실행 흔적으로 설명할 수 있는 것은 거기서 끝낸다
- 보조 노트는 부족한 chronology를 메우는 보조 수단일 뿐이다
- 기존 블로그 초안은 기본 입력 자료가 아니다

## 기존 `blog/` 잔재물 처리 규칙

기존 `blog/` 초안이나 낮은 품질의 예전 글이 `있어도 되는가`에 대한 기본 답은 이렇다.

`디스크에 남아 있는 것 자체는 괜찮지만, 새 초안을 만들 때 입력 자료로 읽히면 안 된다.`

이유는 단순하다.

- 예전 글이 약하면 새로운 글도 그 문체와 구조를 베끼기 쉽다
- 모델은 소스코드보다 가까운 서술 예시를 더 강하게 모방하는 경향이 있다
- 낮은 품질의 초안은 chronology보다 말투와 요약 습관을 오염시킨다

따라서 기본 규칙은 아래와 같다.

- 새로 작성할 때는 기존 `blog/` 글을 기본 입력에서 제외한다
- 기존 글은 초안 완성 후 비교 검토 단계에서만 본다
- 자동 수집 도구가 `blog/`를 함께 읽는다면 예전 글은 `blog/_legacy/`나 별도 보관 위치로 옮긴다
- 정말 다시 시작하려는 경우, 삭제보다 `격리`가 더 안전하다

한 줄 규칙:

`기존 blog는 보관해도 되지만, 새 초안의 근거로 쓰지 않는다.`

## 문서 정체성

`blog/`는 탐색형 개발 기록이다.  
하지만 여기서 말하는 탐색은 막연한 감상이 아니라 `증거에 기반한 복원`이다.

좋은 글은 당시의 무지를 드러내되, 사실을 지어내지 않는다.

- "처음엔 이 함수가 문제라고 생각했다"
- "이 시점에서는 race의 중심이 mask 순서라는 걸 몰랐다"
- "나중에 보니 버그는 계산식이 아니라 상태 전이 정리 순서였다"

반대로 아래 문장은 피한다.

- "문제의 원인은 X였고 이를 수정했다"
- "성능을 높이기 위해 최적화했다"
- "적절히 디버깅한 뒤 해결했다"

정보가 맞더라도, 이런 문장은 개발 과정이 아니라 결과 보고처럼 읽힌다.

## 품질을 안정시키는 핵심 원칙

모델별 품질 차이를 줄이려면, 한 번에 "블로그를 써라"라고 하지 않는다.  
반드시 `근거 추출`과 `서술`을 분리한다.

이 가이드의 기본 작업 순서는 아래 네 단계다.

1. 증거 수집
2. chronology 복원
3. 초안 작성
4. 리라이트와 검토

이 순서를 건너뛰면, 모델은 빈칸을 자기 스타일로 메운다.  
그 순간 문체, 길이, 밀도, 설명 방식이 모델마다 크게 흔들린다.

## 작성 절차

### 1. 증거 수집

먼저 아래 자료를 모은다.

- 주요 변경 파일 목록
- 각 파일에서 핵심적으로 바뀐 함수 또는 블록
- 커밋 흐름 또는 저장 시점 순서
- 실제 실행한 CLI
- 실패한 테스트와 통과한 테스트
- 판단을 바꾼 에러 메시지나 출력
- 작업 중 새로 이해한 개념 메모

이 단계에서는 아직 블로그 문장을 쓰지 않는다.

### 2. chronology ledger 작성

모은 자료를 시간 순서 표로 다시 세운다.  
정확한 날짜와 시각이 있으면 쓰고, 없으면 `Day 1`, `Session 2`, `Phase 3`처럼 적는다.

ledger에는 최소한 아래 항목이 있어야 한다.

- `순서`: 실제 작업 순서
- `시간 표지`: 날짜/시각 또는 Day/Session
- `당시 목표`: 무엇을 만들거나 확인하려 했는가
- `변경 단위`: 어떤 파일/함수/블록을 건드렸는가
- `처음 가설`: 당시 무엇을 의심했는가
- `실제 조치`: 무엇을 수정했는가
- `CLI`: 실제 실행 명령
- `검증 신호`: 통과/실패/수치/에러
- `핵심 코드 앵커`: 꼭 보여줘야 하는 짧은 코드 조각
- `새로 배운 것`: 그 시점에서 이해가 바뀐 개념
- `다음`: 다음 세션으로 이어지는 질문이나 작업

이 단계가 허술하면, 최종 글도 반드시 허술해진다.

### 3. 블로그 초안 작성

초안은 반드시 chronology ledger를 바탕으로 쓴다.  
소스코드를 보지 않고 예전 글을 베끼거나, 기억만으로 분위기를 채우지 않는다.

초안을 쓸 때의 핵심 원칙은 아래와 같다.

- 구현 순서를 파일/함수 단위로 보이게 한다
- 핵심 코드와 CLI를 근거로 문장을 쓴다
- 새 개념이 나오면 짧게라도 설명한다
- 이미 아는 사람이 뒤에서 해설하는 톤이 아니라, 당시 판단이 이동하는 흐름을 보이게 한다

### 4. 리라이트와 검토

마지막 단계에서만 문체를 다듬는다.

이때 확인할 것은 아래다.

- chronology가 실제보다 더 매끈하게 재배열되지 않았는가
- 구현 순서가 빠지지 않았는가
- 코드 스니펫이 판단 전환점과 연결되어 있는가
- CLI가 추상 요약으로 사라지지 않았는가
- 개념 설명이 뜬금없는 강의처럼 분리되지 않았는가
- 글이 과하게 짧거나 과하게 건조하지 않은가

## Batch Mode

이 가이드는 사용자가 각 독립 프로젝트마다 같은 지시를 반복 입력하는 방식을 기본으로 삼지 않는다.  
상위 단위 하나를 주면, AI가 내부적으로 독립 프로젝트를 찾아 같은 절차를 반복 수행하는 `Batch Mode`를 지원해야 한다.

예를 들면 아래 같은 범위다.

- `cs-core/`
- `network-atda/`
- `systems-programming/`

이 경우 사용자는 `lab`마다 다시 지시하지 않는다.  
AI가 해야 할 일은 아래와 같다.

1. 상위 디렉터리 아래에서 `독립 프로젝트 단위`를 자동 탐지한다.
2. 각 프로젝트마다 `evidence ledger -> 글 구조 -> 최종 초안` 순서를 내부적으로 반복한다.
3. 한 프로젝트가 끝나면 다음 프로젝트로 넘어간다.
4. 마지막에 처리 결과만 요약 보고한다.

중요한 원칙은 이것이다.

`1, 2, 3 단계는 사용자가 반복 입력하는 프롬프트가 아니라, AI가 프로젝트마다 반복 수행해야 하는 내부 프로토콜이다.`

### Batch Mode에서 프로젝트를 고르는 기준

상위 디렉터리 아래의 모든 하위 폴더를 무조건 프로젝트로 간주하지 않는다.  
아래 조건을 우선 확인한다.

- 문제 범위를 독립적으로 설명할 수 있는가
- 자기 `README.md` 또는 문제 설명 문서가 있는가
- 자기 검증 명령 또는 실행 진입점이 있는가
- 구현 흐름을 다른 폴더와 분리해서 복원할 수 있는가

이 기준에 맞지 않으면 아래처럼 처리한다.

- 공용 라이브러리 폴더면 건너뛴다
- assets, notes, snippets, temp 폴더면 건너뛴다
- 프로젝트인지 애매하면 `근거 부족`으로 보고하고 기본적으로 건너뛴다

### Batch Mode에서 기존 `blog/`를 다루는 방식

상위 단위에서 일괄 작업할 때는 기존 `blog/` 처리 방식을 먼저 정해야 한다.  
기본 옵션은 아래 셋이다.

1. `skip`
   이미 `blog/`가 있는 프로젝트는 건너뛴다.
2. `rewrite`
   기존 `blog/`가 있어도 무시하고 새로 다시 쓴다.
3. `isolate-and-rewrite`
   기존 `blog/`를 `blog/_legacy/` 같은 위치로 격리한 뒤 새로 쓴다.

품질 기준의 기본 추천은 `isolate-and-rewrite`다.

- 기존 기록은 보존된다
- 새 초안이 예전 문체에 오염되지 않는다
- 자동 수집 과정에서 기존 저품질 글이 근거로 읽힐 위험이 줄어든다

명시가 없으면 아래 기본값을 쓴다.

- 기존 `blog/`가 없으면 새로 작성
- 기존 `blog/`가 있으면 기본적으로 `skip`
- 다만 사용자가 "전부 다시 써라"라고 하면 `rewrite` 또는 `isolate-and-rewrite`로 해석

### Batch Mode에서 기대하는 최종 보고

일괄 작업이 끝나면 긴 회고문을 다시 쓰지 말고 아래만 짧게 보고한다.

- 처리한 프로젝트 목록
- 건너뛴 프로젝트 목록
- 건너뛴 이유
- 기존 `blog/` 처리 방식
- 수동 확인이 필요한 프로젝트

### Batch Mode 지시 예시

아래처럼 한 번만 지시하면 된다.

````md
/Users/woopinbell/work/book-task-3/blog/blog-writing-guide.md 기준으로 `/path/to/cs-core` 아래의 독립 프로젝트를 자동 탐지하라.

각 프로젝트마다 내부적으로 아래 순서를 반복하라.
1. evidence ledger 작성
2. 글 구조 설계
3. 최종 blog 작성

중요:
- 내가 각 lab마다 다시 지시하지 않아도 된다.
- 기존 blog 초안은 입력 근거로 쓰지 마라.
- 소스코드, README, docs, 테스트, CLI를 근거로 사용하라.
- 독립 프로젝트 기준에 맞지 않는 디렉터리는 건너뛰어라.
- 마지막에 처리한 프로젝트와 건너뛴 프로젝트만 요약 보고하라.

기존 blog 처리 방식:
- isolate-and-rewrite
````

없는 프로젝트만 채우고 싶다면 마지막 줄만 아래처럼 바꾼다.

````md
기존 blog 처리 방식:
- skip
````

## 파일 구조 규칙

번호는 장식이 아니라 순서와 삽입 가능성을 위한 장치다.

- `00`은 시리즈 지도 또는 개요
- `10`, `20`, `30`은 연속된 개발 구간
- 중간 삽입이 필요하면 `15`, `25`, `35`
- 상한은 없다
- 가능하면 기존 번호는 자주 재배열하지 않는다

권장 예시는 아래와 같다.

- `00-series-map.md`
- `10-getting-the-first-version-working.md`
- `20-fixing-the-first-real-breakage.md`
- `30-hardening-and-verifying.md`

중요한 점은 `주제 분류`가 아니라 `실제 개발 흐름 분할`이어야 한다는 것이다.

## 각 글에 반드시 들어가야 하는 것

각 글은 아래 요소를 빠뜨리지 않는다.

- 지금 시점이 프로젝트 전체에서 어디인지 알려 주는 짧은 도입
- 이 글에서 다루는 구현 순서 요약
- 세션별 chronology
- 핵심 코드 스니펫
- 실제 CLI와 검증 신호
- 새로 이해한 개념 설명
- 다음 글 또는 다음 세션으로 이어지는 질문

여기서 특히 중요하게 보는 것은 `구현 순서 요약`이다.  
당신이 원하는 글은 단순히 "버그를 어떻게 고쳤는가"만 보여 주는 글이 아니라, `무엇을 어떤 순서로 만들어 갔는가`를 따라가게 만드는 글이기 때문이다.

## 세션 엔트리 기본 형식

모든 필드를 기계적으로 다 채울 필요는 없다.  
하지만 아래 뼈대는 거의 항상 유지한다.

````md
## Day 2
### Session 1

- 당시 목표:
- 변경 단위: `src/parser.py`의 `parse_expr`, `parse_term`
- 처음 가설:
- 실제 진행:

CLI:

```bash
$ pytest tests/test_parser.py -q
$ python -m parser_interpreter --demo precedence
```

검증 신호:

- 어떤 테스트가 실패/통과했는가
- 어떤 출력이 판단을 바꿨는가

핵심 코드:

```python
while self.peek().type in ADDITIVE_TOKENS:
    ...
```

왜 이 코드가 중요했는가:

새로 배운 것:

- 여기서 처음 이해한 개념이나 규칙

다음:
````

이 형식에서 특히 빠지면 안 되는 것은 아래 네 가지다.

- `변경 단위`
- `CLI`
- `왜 이 코드가 중요했는가`
- `새로 배운 것`

## 코드 스니펫 규칙

코드는 길게 복붙하지 않는다.  
짧고 결정적인 조각만 넣는다.

좋은 코드 스니펫은 보통 아래 중 하나다.

- 분기 판단이 바뀐 부분
- 상태 전이가 드러나는 부분
- 불변식을 강제하는 부분
- 버그의 실제 원인이 드러난 부분
- 성능이나 설계의 핵심 trade-off가 보이는 부분

항상 함께 적어야 하는 문장은 이것이다.

`왜 이 코드가 그 시점의 판단 전환점이었는가`

한 줄 규칙:

`코드는 짧게, 의미는 길게 설명한다.`

## CLI 기록 규칙

CLI는 장식이 아니라 근거다.  
실행한 명령은 실제 chronology를 따라 남긴다.

원칙은 아래와 같다.

- 실제 실행한 명령은 빠짐없이 남긴다
- 순서는 실제 작업 순서를 따른다
- 최종 pass/fail과 핵심 숫자, 상태 문자열은 그대로 남긴다
- 반복 출력이나 장황한 로그는 요약할 수 있다
- 판단을 바꾼 에러 메시지는 verbatim으로 남긴다
- raw transcript 전체 dump는 금지한다

한 줄 규칙:

`코드는 핵심만, CLI는 전체 흐름을.`

## 개념 설명 규칙

당신이 원하는 글의 중요한 특징은 `작업 중 새로 이해한 개념`이 함께 설명된다는 점이다.  
이 요소는 선택 사항이 아니라 품질의 핵심이다.

아래 상황에서는 짧게라도 개념 설명을 넣는다.

- 새로운 알고리즘 전략을 처음 적용했을 때
- 자료구조 invariant를 처음 제대로 이해했을 때
- concurrency, memory, parser, cache, signal 같은 비직관적 개념을 만났을 때
- 성능 병목의 원리가 코드 수준 판단과 직접 연결될 때

좋은 개념 설명은 아래 성격을 가진다.

- 현재 세션의 코드와 직접 연결된다
- 위키 문서처럼 일반론만 말하지 않는다
- "왜 이걸 지금 알아야 했는가"를 함께 적는다

나쁜 개념 설명은 아래와 같다.

- 문서 본문과 분리된 교과서 요약
- 지금 문제와 연결되지 않는 일반론
- 용어 정의만 있고 판단 변화가 없는 설명

## 원하는 문체

이 가이드가 목표로 삼는 문체는 특정 모델의 브랜드 문체가 아니다.  
하지만 아래 성질을 강하게 원한다.

- 관찰이 구체적이다
- 당시 판단이 바뀌는 장면이 보인다
- 설명이 친절하지만 과하게 강의체는 아니다
- 추상 요약보다 코드와 실행 흔적에 기대어 말한다
- 문장이 건조한 보고서처럼 끊기지 않는다
- 그렇다고 감상문처럼 흐르지도 않는다

좋은 문체는 보통 아래 느낌에 가깝다.

- "처음엔 X라고 봤는데, 로그를 다시 보니 문제의 중심은 Y였다"
- "이 시점에서는 구현을 더 늘리기보다 invariant부터 고정해야 했다"
- "여기서 처음 이해한 건 단순히 버그 위치가 아니라, 왜 이 순서가 항상 먼저여야 하는가였다"

피해야 할 문체는 아래와 같다.

- "이번 글에서는 ..."
- "적절히 구현했다"
- "효율적으로 최적화했다"
- "문제를 해결할 수 있었다"

이런 문장은 정보량이 거의 없고, 모델마다 제일 쉽게 복제하는 저품질 패턴이다.

## 길이와 밀도 규칙

품질은 단순 토큰 수가 아니라 `근거 밀도`에서 나온다.  
하지만 모델 편차를 줄이려면 최소 밀도는 강제해야 한다.

각 글은 아래 기준을 만족하도록 쓴다.

- 핵심 개발 구간이 2개 이상 보인다
- 주요 변경 파일/함수가 최소 한 번 이상 명시된다
- 코드 스니펫이 최소 2개 이상 들어간다
- CLI 블록이 최소 2개 이상 들어간다
- 검증 신호가 추상 요약으로만 끝나지 않는다
- 새로 배운 개념 설명이 최소 1개 이상 들어간다

짧게 끝내야 하는 특별한 근거가 없다면, 압축 요약보다 `충분히 설명된 초안`을 우선한다.

한 줄 규칙:

`짧아도 되지만, 빈약하면 안 된다.`

## 날짜와 시각 규칙

정확한 날짜와 시각은 근거가 있을 때만 쓴다.

- 실제 작업 로그
- 커밋 시각
- 노트 timestamp
- 저장된 terminal history
- 다른 문서의 시계열 기록

근거가 없으면 지어내지 않는다.

- 날짜가 없으면 `Day 1`, `Day 2`
- 시각이 없으면 `Session 1`, `Session 2`
- 더 거친 구분만 가능하면 `Phase 1`, `Phase 2`

중요한 것은 정확한 시각 흉내가 아니라, 실제 순서를 보존하는 것이다.

## 언어 선택 규칙

영어로 쓰면 무조건 품질이 좋아진다고 가정하지 않는다.

기본 원칙은 아래와 같다.

- 최종 독자가 한국어 독자라면 최종 글도 한국어를 기본으로 한다
- 기술 용어, 함수명, 에러 문자열은 원문 영어를 그대로 둬도 된다
- 모델이 한국어에서 너무 압축되면 `증거 추출`이나 `개요 작성`만 영어로 시도할 수 있다
- 하지만 최종 글은 번역투가 아니라 자연스러운 한국어로 다시 써야 한다

즉 영어는 `보조 전략`일 수는 있어도, 품질의 핵심 해법은 아니다.

## AI 사용 규칙

AI에게 바로 "좋은 블로그를 써라"라고 하지 않는다.  
그 방식은 모델마다 결과가 가장 크게 흔들린다.

AI 사용 시 기본 규칙은 아래와 같다.

1. 먼저 evidence ledger를 만들게 한다
2. 그다음 글 구조를 짜게 한다
3. 마지막에만 실제 문장을 쓰게 한다
4. 예전 `blog/` 초안은 첫 단계 입력에 넣지 않는다
5. 특정 모델 이름을 문체 목표로 주지 말고, 원하는 성질을 직접 적는다

예:

````md
너의 임무는 예쁜 요약문을 쓰는 것이 아니라, 코드와 실행 흔적을 바탕으로 실제 개발 순서를 복원하는 것이다.

작업은 세 단계로 나눈다.

1. 먼저 chronology ledger를 작성하라.
   - 각 항목에 시간 표지, 당시 목표, 변경 파일/함수, 처음 가설, 실제 조치, CLI, 검증 신호, 핵심 코드 앵커, 새로 배운 것, 다음을 포함하라.
2. 그 다음 ledger만 바탕으로 글 구조를 짜라.
3. 마지막에만 최종 블로그 초안을 한국어로 작성하라.

출력 규칙:
- 결과 요약보다 구현 순서가 먼저 보여야 한다.
- 주요 변경 파일/함수를 명시하라.
- 핵심 코드 스니펫을 본문 중간에 넣어라.
- CLI는 실제 실행 순서를 보이게 하라.
- 새로 이해한 개념은 현재 세션의 문제와 연결해서 설명하라.
- 나중에 안 사실을 앞부분에 미리 흘리지 마라.
- 기존 blog 초안의 문체를 모방하지 마라.
````

## 반례

### 반례 1: 결과만 맞는 해설문

```md
문제의 원인은 free list unlink 순서였다. 이를 수정하여 allocator를 해결했다.
```

이 문장은 사실이 맞더라도 개발 과정이 보이지 않는다.

### 반례 2: 구현 순서가 없는 글

```md
테스트를 수행했고 코드를 수정했다. 이후 구조를 개선하고 문서를 정리했다.
```

이 문장만으로는 무엇을 어떤 순서로 만들었는지 복원할 수 없다.

### 반례 3: 개념이 빠진 글

```md
cache miss를 줄이기 위해 접근 순서를 바꿨다.
```

왜 접근 순서가 중요했는지, 무엇을 새로 이해했는지가 빠져 있다.

### 반례 4: 기존 저품질 초안을 베낀 글

```md
이번 글에서는 parser 구현 과정을 소개한다. 다양한 시행착오 끝에 원하는 결과를 얻을 수 있었다.
```

이 문장은 가장 흔한 저품질 AI 서술을 다시 복제한 것이다.

## 기본 원칙 요약

- `blog/`는 탐색형 개발 기록이다
- 기본 적용 단위는 `독립 프로젝트 단위`다
- 글의 근거는 소스코드와 실행 흔적이다
- 기존 `blog/` 초안은 기본 입력 자료가 아니다
- 먼저 chronology ledger를 만들고, 그 다음 초안을 쓴다
- 구현 순서, 코드, CLI, 개념 설명을 모두 남긴다
- 특정 모델 흉내보다 원하는 서술 성질을 직접 강제한다
- 짧게 쓸 수는 있어도 빈약하게 쓰면 실패다
