# Spring Backend 자소서

저는 Spring 백엔드를 배울 때도 기능을 얼마나 많이 붙였는지보다, 같은 도메인을 어디까지 설계와 검증의 언어로 설명할 수 있는지가 더 중요하다고 생각했습니다. 42서울과 공통 코어에서 쌓은 기반 위에, `backend-spring`은 그 기준을 도메인 중심 결과물로 정리한 트랙이었습니다.

대표 결과물인 `commerce-backend-v2`에서는 persisted auth, JPA + Flyway, Redis cart/throttling, outbox + Kafka notification을 modular monolith에 연결했습니다. 같은 커머스 도메인을 baseline보다 더 깊게 구현해, 왜 v2가 필요한지를 문서와 검증 경로로 설명할 수 있게 만들었습니다.

`backend-common` 경험은 Spring 제출본에도 중요한 공통 기반이었습니다. 특정 프레임워크의 문법보다 인증과 경계, 비동기 처리와 verification report를 어떤 기준으로 설명할지 먼저 생각하게 해 줬기 때문입니다.

협업에서도 저는 코드와 문서, 테스트가 서로 다른 이야기를 하지 않도록 정리하는 사람이고 싶습니다. 다른 사람이 프로젝트를 빠르게 따라올 수 있게 만들면 팀의 속도와 안정성이 함께 올라간다고 생각합니다.

입사 후에는 현재 도메인과 아키텍처를 빠르게 이해하고, 맡은 서비스의 안정성과 설명 가능성을 함께 높이는 백엔드 엔지니어로 기여하고 싶습니다.
