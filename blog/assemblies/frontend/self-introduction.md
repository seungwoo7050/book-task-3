# Frontend 자소서

프론트엔드 개발을 좋아하는 이유는 사용자가 바로 체감하는 변화를 가장 직접적으로 만들 수 있기 때문입니다. 다만 저는 좋은 화면이 단순히 보기 좋은 화면으로 끝나지 않고, 상태 전이와 실패 처리, 검증 루프까지 함께 설계되어야 한다고 생각합니다. 42서울과 현재 워크스페이스의 공통 코어 학습은 그 기준을 만드는 데 큰 기반이 됐습니다.

대표 사례인 `mini-vrew`에서는 브라우저 안에서 무거운 처리와 사용자 경험을 함께 설계했습니다. FFmpeg WASM, WebGL, Web Audio API, IndexedDB를 기능 목록이 아니라 하나의 편집 흐름으로 연결하면서, 프론트엔드가 런타임과 얼마나 가까운 문제인지 더 분명하게 느꼈습니다.

`Ops Triage Console`과 `Client Onboarding Portal`에서는 운영 화면과 고객-facing 플로우를 각각 다른 방식으로 설계했습니다. optimistic update, retry, validation, draft restore, route guard 같은 상태 전이를 분리해 다루며 제품형 UI를 설명 가능한 결과물로 정리했습니다.

협업에서도 저는 구현만큼 설명 가능한 결과물을 남기는 태도가 중요하다고 생각합니다. README와 검증 명령, 문서를 정리해 다른 사람이 프로젝트를 다시 따라올 수 있게 만드는 편이고, 팀 안에서도 이런 기준을 함께 만드는 사람으로 기여하고 싶습니다.

입사 후에는 먼저 제품과 사용자 흐름을 빠르게 이해하고, 맡은 화면을 안정적으로 구현하는 데 집중하겠습니다. 그 위에서 제품형 UI 설계와 검증 습관을 바탕으로, 팀 안에서 믿고 맡길 수 있는 프론트엔드 개발자로 성장하고 싶습니다.
