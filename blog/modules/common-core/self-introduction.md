# Common Core 자소서 Module

저는 어떤 직무를 지원하더라도 개발의 출발점은 같다고 생각합니다. 문제를 잘게 분해하고, 실패 원인을 추적하고, 다시 검증 가능한 형태로 남기는 습관입니다. 42서울 정규과정과 현재 워크스페이스의 CS·네트워크·데이터 프로젝트들은 그 습관을 만든 기반이었습니다.

특히 42서울에서는 기술 키워드보다 문제의 경계를 먼저 잡는 태도를 배웠습니다. `ft_transcendence`에서는 Django 백엔드를 전담하며 42 OAuth, JWT, TOTP 기반 2FA 같은 인증/인가 흐름을 다뤘고, `minishell`, `irc`, `raytracing` 같은 팀과제에서는 시스템과 협업의 기본기를 쌓았습니다.

이후에는 `cs-core`, `network-atda`, `database-systems`를 순차적으로 구현하며 런타임, 프로토콜, 데이터 계층을 더 구조적으로 이해하려고 했습니다. 덕분에 특정 프레임워크만 보는 대신, 기능 뒤에 있는 실행 원리와 실패 조건을 함께 보게 됐습니다.

협업에서도 저는 구현 결과를 설명 가능한 자산으로 남기는 태도가 중요하다고 생각합니다. README와 검증 명령, 문서와 노트를 함께 정리하는 이유도 다른 사람이 코드를 다시 읽고 이어갈 수 있게 만들기 위해서입니다.

입사 후에도 먼저 현재 서비스의 문제 정의와 검증 기준을 빠르게 이해하고, 그 위에서 맡은 영역의 결과물을 분명하게 남기는 개발자로 기여하고 싶습니다.
