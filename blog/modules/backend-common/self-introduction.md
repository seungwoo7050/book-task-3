# Backend Common 자소서 Module

저는 백엔드 기능을 구현할 때 단순히 API가 동작하는지만 보기보다, 인증과 인가, 서비스 경계, 실패 후 복구까지 함께 설명할 수 있어야 한다고 생각합니다. FastAPI 학습 트랙은 그 기준을 가장 반복적으로 다듬은 구간이었습니다.

`workspace-backend`에서는 협업형 도메인을 단일 서비스로 통합했고, `workspace-backend-v2-msa`에서는 같은 도메인을 gateway와 내부 서비스로 다시 분해했습니다. 이 과정에서 public route shape를 유지하면서도 내부 경계와 비동기 알림 구조를 분리하는 감각을 익혔습니다.

제가 특히 중요하게 본 것은 이벤트를 어디서 끊고, 어떤 검증 경로를 남길 것인가였습니다. outbox, Redis Streams, websocket fan-out 같은 흐름도 키워드로 끝내지 않고 `make test`, `make smoke`, verification report로 이어지게 정리했습니다.

협업에서도 이런 태도는 그대로 이어집니다. 코드만 작성하는 것이 아니라, 다른 사람이 프로젝트를 다시 띄우고 구조를 이해할 수 있는 문서와 검증 기준까지 남기는 사람이 되고 싶습니다.

입사 후에도 먼저 현재 서비스의 경계와 검증 기준을 파악하고, 맡은 API를 구조와 운영성까지 포함해 안정적으로 정리하는 데 기여하고 싶습니다.
