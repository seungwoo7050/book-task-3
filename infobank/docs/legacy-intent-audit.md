# 기존 의도 감사

이 문서는 과제 설명과 초기 메모가 원래 무엇을 배우게 하려 했는지 복원하고, 현재 레포가 그 의도를 어떻게 다시 정리했는지 설명한다. 이름에 `legacy`가 들어가지만, 현재 레포 안에 별도 `legacy/` 디렉터리가 있다는 뜻은 아니다.

## 원래 의도 요약

설명회 메모와 초기 과제 해석의 핵심 의도는 두 갈래로 정리할 수 있다.

1. MCP 추천을 실제 운영 시나리오로 설명할 수 있어야 한다.
2. 상담 챗봇의 답변 품질을 정의하고 자동 평가할 수 있어야 한다.

현재 레포는 이 의도를 `mcp-recommendation-demo/`와 `chat-qa-ops/` 두 활성 트랙으로 구체화했다.

## 지금 문서에서 바로잡은 사실

- 예전 번호형 폴더 구조(`study1`, `study2`, `study3`)를 현재 레포의 사실로 가정하지 않는다.
- 현재 레포에는 별도 `legacy/` 디렉터리가 없다.
- `notion/`은 개인 숨김 폴더가 아니라 공개 백업 문서로 레포에 포함한다.
- 오래된 절대 경로, 영어 중심 안내, 이미 사라진 구조 전제를 제거한다.

## 왜 2번 과제가 가장 먼저 성숙했는가

`chat-qa-ops/`는 다음 이유로 가장 빠르게 완성도를 높이기 쉬웠다.

1. 품질 정의, 자동 평가, 모니터링이라는 문제 구분이 분명했다.
2. 작은 stage 프로젝트를 `00~08`로 쪼개기 쉬웠다.
3. `v0 -> v3` 버전 스냅샷 전략으로 학습용 산출물과 공개용 산출물을 함께 설명하기 좋았다.

## 왜 1번 과제도 반드시 남겨 두는가

MCP 추천 최적화는 상담 품질 관리와는 다른 설계 근육을 요구한다.

- metadata/schema 설계
- recommendation explanation
- release gate와 artifact export
- self-hosted 운영 콘솔

즉, 두 트랙을 함께 봐야 "평가 시스템"과 "추천 시스템"을 모두 다룰 수 있고, 학생 입장에서도 포트폴리오의 폭이 넓어진다.

## 현재 레포의 운영 기준

- 레거시 자료가 나중에 추가되더라도 읽기 전용 참조로만 사용한다.
- 새 구조와 새 설명은 현재 활성 트랙 내부에 작성한다.
- 구현보다 문서가 먼저 필요한 단계에서는 없는 구현을 가장하지 않고 `planned` 상태를 명시한다.
- `notion/`을 다시 쓸 때는 기존 문서를 `notion-archive/`로 남겨 두고 새 문서를 추가한다.
