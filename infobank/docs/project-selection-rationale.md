# Project Selection Rationale

## 세 과제 재구성

설명회 STT를 기준으로 세 과제는 다음처럼 해석한다.

## 1번 과제

- 주제: MCP 추천 최적화
- 핵심 질문
  - MCP 메타데이터와 manifest를 어떻게 구조화할 것인가
  - MCP를 어떻게 차별화하고 노출할 것인가
  - 실제 사용 로그와 release gate로 개선을 어떻게 증명할 것인가

## 2번 과제

- 주제: 챗봇 상담 품질 관리
- 핵심 질문
  - 상담 품질을 어떤 기준으로 정의할 것인가
  - LLM 평가, 규칙/가드레일, 근거 검증을 어떻게 합성할 것인가
  - golden set, 대시보드, 개선 실험으로 품질을 어떻게 지속 모니터링할 것인가

## 3번 과제

- 주제: 음성 회의 어시스턴트
- 핵심 질문
  - 안건/토픽 전환과 쟁점 구조화를 어떻게 할 것인가
  - 어떤 시점에 어떤 개입 액션을 제시할 것인가
  - STT, 자료 수집, replayable demo를 어떻게 검증할 것인가

## 왜 study2를 먼저 구현하는가

`study2`가 먼저인 이유는 다음과 같다.

1. 기존 `legacy/` 자산의 재사용성이 가장 높다.
2. 요구가 상대적으로 명확하고, 제출 가능한 초기 데모를 가장 빨리 만들 수 있다.
3. QA/평가/대시보드 설계 경험은 `study1`, `study3`에도 직접 재사용된다.

## 왜 study1과 study3도 미리 설계하는가

- 설명회 정보와 과제 해석을 다시 수집하지 않기 위해서다.
- 현재 설계 판단을 문서와 폴더 구조로 남겨두면 이후 전환 비용이 줄어든다.
- 세 트랙이 같은 규칙을 공유하면 제출 준비와 확장 구현 방식도 통일된다.

## 공통 capstone 규칙

모든 `studyN/08-capstone-submission/`은 다음을 따른다.

1. `v0-*`: 최초 제출 가능한 동작 데모
2. `v1-*`: `v0` 폴더를 그대로 복제해 확장
3. `v2-*`: `v1` 폴더를 그대로 복제해 제출용 마감

버전 폴더는 덮어쓰지 않는다.
각 시점의 제출물과 확장 판단을 폴더 단위로 보존한다.

## 한국 시장 기준

- 한국어 UI와 한국어 테스트 시나리오를 기본으로 둔다.
- 서울 리전 또는 한국 기업 도입 설명이 가능한 운영 구성을 기본값으로 둔다.
- 국내 벤더가 강한 영역은 국내 벤더를 우선 고려하고, 프로토콜/생태계 우위가 큰 영역은 글로벌 표준을 사용한다.
