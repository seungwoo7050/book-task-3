# Retrospective — Realtime Chat

## 가장 크게 배운 것

이 프로젝트를 시작하기 전에는 "오프라인 메시지 처리"를 하나의 큰 문제로 봤다.
끝나고 나서는 그것이 실제로는 네 개의 독립적인 문제(pending 생성, ack 매칭, replay 필터, 중복 제거)라는 걸 체감했다.

각 문제를 별도 함수로 분리하고 독립적으로 테스트하니까, 전체 시스템이 복잡해 보이지만 각 조각은 놀라울 정도로 단순했다. "분해해서 따로 테스트하면 복잡한 시스템도 다룰 수 있다"는 원칙을 채팅 도메인에서 직접 경험한 셈이다.

## 예상과 달랐던 것

**Typing/Presence의 위치 결정이 생각보다 중요했다.** 처음에는 "그냥 상태 하나 추가하면 되지"라고 생각했는데, 이 상태를 DB에 넣느냐 메모리에 두느냐에 따라 replay 로직, 스키마 마이그레이션, 테스트 범위가 전부 달라졌다. 결국 "ephemeral state와 durable record를 섞지 않는다"는 원칙이 설계 전체를 단순하게 만들었다.

**dedupe와 eventId 필터가 둘 다 필요한 이유도 처음엔 와닿지 않았다.** eventId 필터만 있으면 될 것 같았는데, 서버가 같은 메시지를 다른 eventId로 두 번 보내는 경우(서버 측 retry)를 생각하면 serverId 기준 dedupe도 반드시 필요했다.

## 약했던 점

- WatermelonDB의 실제 CRUD를 다루지 않았다. 스키마만 정의하고 순수 함수로 로직을 증명했기 때문에, 실제 DB 연동에서 발생하는 비동기 문제, 트랜잭션 충돌 같은 경험은 빠져 있다.
- 실제 WebSocket 연결 테스트가 없다. replay 로직은 순수 함수로 검증했지만, 실제 연결 끊김/재연결 시나리오는 `incident-ops-mobile`에서 다룰 예정이다.
- UI 테스트가 앱 레벨에서는 없다. 컴포넌트 렌더링 확인만 하고, 사용자 상호작용 시뮬레이션은 하지 않았다.
