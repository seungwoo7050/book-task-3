# Retrospective — App Distribution

## 가장 크게 배운 것

배포는 코드를 빌드해서 올리는 한 번의 동작이 아니라, **"배포할 준비가 되었는지를 지속적으로 검증하는 규율"**이었다.

이 프로젝트에서 실제 스토어 업로드는 단 한 번도 하지 않았다. 하지만 "env 파일이 일관적인가", "Fastlane lane이 정의되어 있는가", "CI가 모든 검증을 통과하는가"를 코드와 스크립트로 증명하는 경험이 배포의 핵심이라는 걸 깨달았다.

실제 팀에서 배포 사고가 나는 건 빌드 명령이 잘못되어서가 아니라, 환경이 어긋나있거나 검증 단계가 빠져있기 때문인 경우가 많다. 이 프로젝트는 그 "빠져있을 수 있는 검증 단계"를 미리 채워보는 훈련이었다.

## 예상과 달랐던 것

**env 파일 관리가 생각보다 중요했다.** 세 개의 env 파일이 같은 키를 가지는지 확인하는 것은 사소해 보이지만, 이것이 무너지면 staging에서는 되던 게 production에서 안 되는 상황이 생긴다. 키 일관성을 스크립트로 강제한 덕분에 "어떤 환경에서도 같은 설정 구조"라는 보장을 얻을 수 있었다.

**Fastlane을 "실제 빌드 없이" 쓰는 것도 가치가 있었다.** lane 구조를 정의하고, validate → rehearsal 순서를 명시하는 것만으로도 "배포 파이프라인의 골격"을 이해할 수 있었다.

## 약했던 점

- 실제 서명(signing)을 다루지 않았다. keystore나 provisioning profile 관리는 이 프로젝트 범위 밖이다.
- CodePush/OTA 관련 설정은 `problem/README.md`에 예시 코드로만 남아 있고, 실제 구현은 하지 않았다.
- CI 워크플로우가 빌드 단계를 포함하지 않으므로, 실제 APK/IPA 생성을 검증하지는 않는다.

## 다음 단계에 넘긴 것

- 실제 서명과 빌드 → 팀 환경이나 실제 프로덕트에서
- OTA 업데이트 인프라 → 별도 학습 과제로 분리
- 실제 CI에서의 release artifact 생성 → 이 프로젝트의 구조를 기반으로 확장 가능
