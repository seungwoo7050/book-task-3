# Problem Framing — Realtime Chat

## 이 프로젝트는 어떤 질문에서 시작했나

채팅 앱에서 메시지를 보내는 건 쉬워 보인다. 텍스트를 입력하고, 전송 버튼을 누르면 끝이다.
그런데 "보내는 중" 상태의 메시지가 화면에 먼저 보이고, 서버가 확인해야 비로소 "보냈음"이 되는 그 사이의 시간은 어떻게 다뤄야 할까?

더 까다로운 상황도 있다. 네트워크가 끊겼다가 다시 연결되면, 그 사이에 서버에 쌓인 메시지를 빠짐없이 가져와야 한다. 같은 메시지가 두 번 들어올 수도 있고, 순서가 뒤섞일 수도 있다. 이 모든 상황을 조용히 처리해서 사용자에게는 "그냥 잘 되는 채팅"으로 보여야 한다.

이 프로젝트는 그 "그냥 잘 되는" 뒤에 숨은 네 가지 문제를 local-first 모델 하나로 풀어보는 과제다.

## 풀어야 했던 네 가지 문제

1. **Pending message 생성** — 서버 응답 전에 UI에 즉시 반영되는 메시지를 어떻게 만들 것인가
2. **Server ack reconciliation** — 서버가 확인 응답을 보내면 pending 메시지를 어떻게 확정 상태로 바꿀 것인가
3. **Replay dedupe** — 재연결 후 서버에서 밀려오는 이벤트 중 이미 처리한 것을 어떻게 걸러낼 것인가
4. **Typing/Presence 업데이트** — 휘발성 상태(누가 타이핑 중인지)를 영속 메시지와 어떻게 분리할 것인가

## 명시적으로 하지 않은 것

- 실제 WebSocket 서버 구축 (이건 `incident-ops-mobile`에서 한다)
- 메시지 암호화나 E2E encryption
- 파일 첨부, 이미지 전송
- 읽음 확인 (read receipt)

## 이 과제가 학습 경로에서 차지하는 위치

`offline-sync-foundations`에서 queue, retry, DLQ 같은 오프라인 동기화의 기초를 먼저 익혔다.
이 프로젝트는 그 기초 위에 채팅이라는 구체적인 제품 도메인을 얹어서,
"일반적인 sync 엔진이 실제 제품에서는 어떻게 달라지는가"를 체감하는 단계다.

이후 `app-distribution`에서는 이 앱을 기반으로 배포 프로세스를 리허설한다.
